# MVP 开发指南

> 来源：基于 `docs/功能需求定义.md`、`docs/数据模型设计.md`、`docs/API风格规范.md`、`docs/API设计.md` 和 `docs/页面与交互文档.md` 整理。
>
> 定位：本文用于归档 MVP 阶段的后端优先开发、测试验收和前后端联调节奏。MVP 已冻结，当前开发不得仅凭本文判断能力状态；请以 `AGENTS.md`、实际代码、`docs/API设计.md` 和 `docs/前后端联调文档.md` 为准。
>
> 开发准则：后端代码按模块组织，开发批次按功能链路推进；每完成一个功能链路，即完整实现、完整测试、完整更新联调说明。
>
> 执行说明：本文保留为 MVP 历史总览。`docs/dev/batches/` 下的单 batch 文件也是历史归档；AI 开发时应先读取根目录 `AGENTS.md` 和当前任务相关文档。

## 1. 技术栈

### 1.1 后端技术栈

| 类型 | 技术 |
|---|---|
| 语言 | Java 17 |
| 框架 | Spring Boot |
| 构建 | Maven |
| Web | Spring Web |
| 参数校验 | Validation |
| ORM | MyBatis-Plus |
| 数据库 | MySQL |
| 数据库迁移 | Flyway |
| 单元测试 | JUnit 5 |
| 接口测试 | MockMvc |

后端 API 必须遵循 `docs/API风格规范.md`：路径以 `/api/v1` 开头，请求和响应使用 JSON，字段使用 `camelCase`，金额使用整数分并以 `Amount` 结尾，成功和错误响应统一包裹。

### 1.2 前端技术栈

| 类型 | 技术 |
|---|---|
| 形态 | H5 移动端网页 |
| 框架 | Vue 3 |
| 构建 | Vite |
| 语言 | TypeScript |
| 路由 | Vue Router |
| 状态管理 | Pinia |
| UI 组件 | Vant |
| HTTP | Axios |
| 后续单元测试 | Vitest |
| 后续端到端测试 | Playwright |

前端第一阶段以移动端 H5 为主，体验接近小程序，不建设桌面端管理后台。

## 2. 开发策略

### 2.1 总体策略

| 策略 | 要求 |
|---|---|
| 后端先行 | 先完成 API、数据迁移、业务事务、MockMvc 测试，再进入对应页面联调 |
| 链路递进 | 不采用“全部后端做完再联调”，每完成一个功能链路就同步更新 `docs/前后端联调文档.md` |
| 代码组织 | 后端按模块组织，如 auth、user、store、product、groupbuy、address、order、subscription、member |
| 开发批次 | 按功能链路推进，不按模块或 Controller 文件拆批次；模块组织服务于代码边界，批次组织服务于可验收链路 |
| 完整交付 | 每个 batch 必须完整实现、完整测试、完整更新联调说明，不把已知问题留到下一批 |
| 范围控制 | 只做 MVP 必需能力，P1/P2 只保留扩展口径，不进入当前开发 |

### 2.2 每个 Batch 的完成定义

每个 batch 完成时必须同时满足：

| 项目 | 完成要求 |
|---|---|
| 接口实现 | 本批涉及接口全部可调用，响应结构符合 API 规范 |
| 数据迁移 | 涉及表结构通过 Flyway 管理，能从空库初始化 |
| 业务规则 | 权限、状态、事务、快照、库存等规则按文档实现 |
| 自动化测试 | 每个接口至少有 MockMvc 测试，关键 Service 有单元测试 |
| 异常路径 | 本批已知异常路径全部有测试覆盖 |
| 联调文档 | 更新对应功能链路的页面入口、前置条件、调用顺序、样例、错误码和测试数据 |

## 3. H5 风格口径

MVP 前端视觉以移动端优先，整体接近微信小程序体验。

| 元素 | 风格要求 |
|---|---|
| 页面基调 | 白底内容区，浅灰页面背景 |
| 信息容器 | 圆角卡片承载团购、订单、地址、会员卡等信息 |
| 商品呈现 | 商品图突出，首屏优先展示封面、商品名、价格和销量 |
| 价格呈现 | 价格使用高对比样式，金额比描述和标签更突出 |
| 信任信息 | 团长头像、团长名称、店铺名称、基础信任标签在商品详情和团长主页中明显可见 |
| 导航 | 使用底部 Tab 承载首页、订单、个人中心等主入口 |
| 详情页操作 | 团购详情页底部固定操作栏，承载订阅、跟团购买等核心操作 |
| 表单体验 | 创建店铺、发布团购、地址填写、发货填写均按移动端表单交互设计 |

MVP 不建设平台后台，也不为桌面端单独设计复杂布局。

## 4. 严格测试要求

### 4.1 接口测试

| 范围 | 要求 |
|---|---|
| 所有 API | 每个接口至少有 MockMvc 测试 |
| 成功响应 | 校验 HTTP 状态、`success`、`data`、`traceId` 和关键字段 |
| 错误响应 | 校验错误码、错误消息、HTTP 状态和响应结构 |
| 登录接口 | 覆盖模拟登录、获取当前用户、无效 token |
| 公开接口 | 覆盖未登录可访问 |
| 私有接口 | 覆盖未登录、非资源所有者、非团长身份 |

### 4.2 Service 单元测试

关键 Service 必须有单元测试：

| Service | 必测规则 |
|---|---|
| 登录 / 当前用户 | 用户创建或查询、token 解析、团长和店铺摘要 |
| 店铺 | 一个用户一个团长身份、一个团长一个店铺、重复创建 |
| 团购 | 普通团购发布、内联创建商品、商品归属、库存非负 |
| 订单 | 订单预览、创建订单、地址快照、商品快照、状态初始化 |
| 支付 | 状态校验、库存扣减、销量增加、会员关系创建 / 更新 |
| 发货 | 团长权限、订单状态校验、发货记录和状态更新 |
| 订阅 / 会员 | 重复订阅处理、会员卡展示数据 |

### 4.3 事务和异常路径测试

订单、支付、库存、发货必须有事务和异常路径测试。

| 场景 | 必测内容 |
|---|---|
| 权限校验 | 普通用户不能发布团购，团长不能处理他人店铺订单，用户不能访问他人订单和地址 |
| 状态流转 | `pendingPay -> paid -> shipped -> completed`，以及 `pendingPay -> canceled` |
| 库存不足 | 预览、创建订单、模拟支付均能返回 `INSUFFICIENT_STOCK`，支付失败不改订单状态 |
| 重复支付 | 已支付订单再次支付不能重复扣库存、不能重复增加销量、不能重复累计会员金额 |
| 重复发货 | 已发货订单再次发货不能重复创建发货记录 |
| 订单快照 | 商品或团购后续修改不影响历史订单明细 |
| 地址快照 | 地址后续修改或删除不影响历史订单地址 |
| 事务回滚 | 支付中任一步失败时，订单状态、库存、销量、会员关系保持一致 |

## 5. 后端 MVP 开发批次

### Batch 0：项目骨架、通用响应、错误码、数据库迁移、测试基础

| 项目 | 内容 |
|---|---|
| 目标 | 搭建可运行、可测试、可迁移的后端基础工程，并一次性创建 MVP 核心表结构 |
| 涉及接口 | 暂无业务接口；可提供健康检查接口用于启动验证 |
| 涉及数据表 | `users`、`leaders`、`stores`、`products`、`group_buys`、`group_buy_items`、`addresses`、`orders`、`order_items`、`shipments`、`subscriptions`、`member_relations` |
| 实现任务 | 初始化 Spring Boot + Maven；配置 Java 17、MySQL、MyBatis-Plus、Flyway；定义统一响应、错误响应、错误码枚举、全局异常处理、traceId；配置 Validation；建立 MockMvc 和 Service 测试基类；按 `docs/数据模型设计.md` 创建全部 MVP 核心表、主键、必要唯一约束和基础索引；后续 batch 只在确有必要时追加迁移 |
| 测试要求 | 骨架启动测试；统一错误响应测试；参数校验错误测试；Flyway 空库迁移测试；核心表存在性、关键字段、唯一约束和基础索引校验 |
| 验收标准 | 应用可启动；测试可运行；错误响应符合 `API风格规范.md`；空库执行 Flyway 后一次性具备全部 MVP 核心表；后续 batch 原则上不再为已知 MVP 表反复补基础结构 |
| 联调文档更新项 | 在 `docs/前后端联调文档.md` 补充通用响应格式、认证请求头、错误响应格式、本地联调基础约定和空库初始化说明 |

### Batch 1：模拟登录与当前用户

| 项目 | 内容 |
|---|---|
| 目标 | 完成 MVP 登录态、当前用户识别和用户基础数据 |
| 涉及接口 | `POST /api/v1/auth/mock-login`；`GET /api/v1/me` |
| 涉及数据表 | `users`、`leaders`、`stores` |
| 实现任务 | 实现模拟登录、用户查找或创建、accessToken 生成和解析、当前用户上下文；`GET /me` 返回 user、leader、store 摘要 |
| 测试要求 | MockMvc 覆盖模拟登录、获取当前用户、未登录访问、无效 token；Service 覆盖用户创建 / 查询和当前身份组装 |
| 验收标准 | 前端可保存 `accessToken` 并通过 `GET /api/v1/me` 恢复登录态；未登录接口返回 `UNAUTHORIZED` |
| 联调文档更新项 | 完成“模拟登录与获取当前用户”链路的请求样例、响应样例、错误码和测试数据准备方式 |

### Batch 2：创建店铺与激活团长身份

| 项目 | 内容 |
|---|---|
| 目标 | 用户创建店铺后自动生成团长身份，形成经营入口 |
| 涉及接口 | `POST /api/v1/stores`；`GET /api/v1/my/store`；`PATCH /api/v1/my/store` |
| 涉及数据表 | `users`、`leaders`、`stores` |
| 实现任务 | 实现一个用户最多一个团长身份、一个团长最多一个店铺；创建店铺时创建 leader 和 store；实现我的店铺查询和更新；校验 `/my/store` 均从 token 获取当前店铺 |
| 测试要求 | MockMvc 覆盖创建、查询、更新、未登录、重复创建、参数错误；Service 覆盖唯一约束和重复创建异常 |
| 验收标准 | 创建成功后 `GET /api/v1/me` 中 `hasLeader=true`，并返回 `leaderId`、`storeId`；重复创建返回 `STORE_ALREADY_EXISTS` |
| 联调文档更新项 | 完成“创建店铺”链路，补充创建成功后跳转团长主页或发布入口的判断方式 |

### Batch 3A：商品管理

| 项目 | 内容 |
|---|---|
| 目标 | 团长可管理自己店铺下的商品，为后续发布团购提供可复用商品资产 |
| 涉及接口 | `GET /api/v1/my/store/products`；`POST /api/v1/my/store/products`；`GET /api/v1/my/store/products/{productId}`；`PATCH /api/v1/my/store/products/{productId}`；`DELETE /api/v1/my/store/products/{productId}` |
| 涉及数据表 | `stores`、`products` |
| 实现任务 | 实现当前店铺商品列表、创建、详情、更新、软删除；所有 `/my/store/products/**` 接口从 token 获取当前团长自己的店铺；校验商品归属、金额字段、库存非负和商品状态；删除商品不得影响后续历史订单快照口径 |
| 测试要求 | MockMvc 覆盖商品列表、创建、详情、更新、删除、未登录、非团长、跨店铺商品拒绝、参数错误；Service 覆盖商品归属、库存非负、金额字段和软删除状态 |
| 验收标准 | 团长只能管理自己店铺商品；普通用户不能访问商品管理接口；商品基础信息可被 Batch 3B 发布团购时复用 |
| 联调文档更新项 | 在“发布普通团购”链路中补充复用已有商品前的商品创建和选择说明；如前端需要独立商品管理入口，补充商品管理联调小节 |

### Batch 3B：发布普通团购

| 项目 | 内容 |
|---|---|
| 目标 | 团长可发布普通团购，支持复用已有商品或在发布时内联创建商品 |
| 涉及接口 | `POST /api/v1/my/store/group-buys` |
| 涉及数据表 | `stores`、`products`、`group_buys`、`group_buy_items` |
| 实现任务 | 实现普通团购创建并直接发布；MVP 仅支持 `groupType=normal`；一个团购至少包含一个团购商品；支持 item 传入 `productId` 复用当前店铺商品，或传入 `product` 内联创建商品；同一事务创建 `group_buys`、必要的 `products` 和 `group_buy_items`；校验商品归属、`groupStock >= 0`、团购价为整数分、时间字段合法 |
| 测试要求 | MockMvc 覆盖发布成功、复用商品、内联商品、未登录、非团长、跨店铺商品拒绝、无商品、库存为负、参数错误；Service 覆盖发布事务、内联商品事务回滚、商品归属和团购商品库存校验 |
| 验收标准 | 团长可发布 `groupType=normal` 且 `status=published` 的团购；发布失败不能留下半成品团购、商品或团购商品关系；`groupStock` 不允许为负数或 `-1` |
| 联调文档更新项 | 完成“发布普通团购”链路，补充内联创建商品和复用已有商品两种请求样例，以及失败回滚的联调验证点 |

### Batch 3C：团购管理

| 项目 | 内容 |
|---|---|
| 目标 | 团长可查看、更新和结束自己店铺发布的团购 |
| 涉及接口 | `GET /api/v1/my/store/group-buys`；`GET /api/v1/my/store/group-buys/{groupBuyId}`；`PATCH /api/v1/my/store/group-buys/{groupBuyId}`；`POST /api/v1/my/store/group-buys/{groupBuyId}/end` |
| 涉及数据表 | `stores`、`group_buys`、`group_buy_items`、`products`、`orders`、`order_items` |
| 实现任务 | 实现我的团购列表和详情；实现团购展示字段、时间、发货时间等可更新内容；校验团购属于当前团长自己的店铺；实现结束团购；有订单后不得修改已下单团购商品价格，不得删除已下单团购商品；历史订单始终以订单快照为准 |
| 测试要求 | MockMvc 覆盖列表、详情、更新、结束、未登录、非团长、跨店铺团购拒绝、不可结束状态；Service 覆盖团购归属、状态流转、已下单团购商品价格保护和结束团购规则 |
| 验收标准 | 团长只能管理自己店铺团购；`published` 团购可以结束为 `ended`；产生订单后的团购管理不会破坏订单快照和价格口径 |
| 联调文档更新项 | 在“发布普通团购”链路后补充我的团购列表、我的团购详情、更新团购和结束团购的联调说明 |

### Batch 4：公共浏览，首页团购列表、团购详情、团长主页

| 项目 | 内容 |
|---|---|
| 目标 | 未登录用户可浏览首页团购、团购详情和团长主页 |
| 涉及接口 | `GET /api/v1/group-buys`；`GET /api/v1/group-buys/{groupBuyId}`；`GET /api/v1/leaders/{leaderId}/homepage` |
| 涉及数据表 | `leaders`、`stores`、`group_buys`、`group_buy_items`、`products`、`subscriptions` |
| 实现任务 | 实现公开团购分页列表、详情聚合、团长主页聚合；如果订阅功能尚未实现，`viewer.subscribed` 暂时统一返回 `false`；Batch 9 完成订阅功能后，再更新公共浏览接口，使已登录用户返回真实订阅状态 |

| 联调文档更新项 | 完成“公共浏览团购”链路，补充首页、详情页、团长主页调用顺序；注明 Batch 9 前 `viewer.subscribed=false`，Batch 9 后再改为真实订阅状态 |

### Batch 5：地址管理与订单预览

| 项目 | 内容 |
|---|---|
| 目标 | 用户可维护地址，并在下单确认页完成订单预览 |

| 涉及数据表 | `addresses`、`group_buys`、`group_buy_items`、`products` |

### Batch 6：创建订单，地址快照、商品快照、订单明细

| 项目 | 内容 |
|---|---|
| 目标 | 用户提交订单并生成订单主表、地址快照和商品快照 |

## 6. MVP 不实现范围

以下能力不进入 MVP 开发，不在后端接口、前端页面和联调计划中扩展：

| 能力 | 处理方式 |
|---|---|
| 真实微信支付 | MVP 使用模拟支付 |
| 优惠券 / 红包 | 不参与订单金额 |
| 售后退款完整流程 | 只预留订单状态，不实现申请、审核、退款 |
| 购物车 | MVP 直接购买 |
| 帮卖分销 | 不实现佣金、帮卖关系和结算 |
| 积分商城 | 不建立积分账户、积分流水和兑换 |
| 公众号推送 | 不实现真实触达 |
| 平台后台 | 不建设后台页面和后台 API |
