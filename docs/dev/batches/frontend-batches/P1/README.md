# P1 前端 batch 总览

> 范围来源：`docs/功能需求定义.md` 的 P1 能力。P1 前端批次应与后端 P1 能力按链路联调推进，不改变 MVP 已冻结的核心交易闭环。

## 1. 拆批原则

P1 前端可以按后端 P1 batch 对齐，也可以在后端接口稳定后按页面链路合并实现。涉及页面级、链路级或核心组件级改动时，必须遵守 `docs/dev/frontend-ai-workflow.md`，先完成需求澄清和前端技术方案，再进入代码实现。

本目录从当前新增需求开始记录 P1 前端批次。此前 MVP 前端批次仍保留原阶段价值，不需要回写改名。

MVP 批次中关于“消息页空态”“不请求消息 API”“不新增通知接口”的描述是 MVP 阶段冻结口径；进入 P1 Batch 08 后，以本目录的站内通知批次为准。

| 顺序 | Batch | 主题 | 核心能力 | 后端依赖 |
|---:|---|---|---|---|
| 8 | `frontend-batch-08-notification-upload-asset-governance.md` | 站内通知和上传资产前端联调 | 消息页真实通知、未读轮询、Tabbar 角标、上传响应兼容 | 后端 `batch-08-notification-upload-asset-governance.md` |

## 2. 执行方式

站内通知和上传资产治理建议前后端同批执行：

- 后端先提供通知和上传资产接口契约、测试数据和事件生成。
- 前端同步替换消息页占位数据，接入真实通知列表和未读数。
- 联调时按支付、发货、确认收货、订阅、发布团购事件逐个验证通知生成和跳转。

如果后端接口尚未完成，前端可以先完成类型、API 封装、页面结构和 mock 单测，但不得用前端假数据替代最终联调。

## 3. 本轮 P1 前端明确不做

- 真实公众号推送、微信服务通知。
- WebSocket / SSE 实时消息。
- 商家和用户聊天会话。
- 客服在线状态、撤回、敏感词、客服评价。
- 图片素材库、图片管理后台、手动删除图片页面。
- 为了展示通知绕过登录、权限、真实接口或状态流转。

## 4. 通用验收要求

- 每个前端 batch 必须更新 `docs/前后端联调文档.md`。
- 每个前端 batch 必须运行 `npm run typecheck`、`npm run lint`、`npm run test:unit`、`npm run build` 和 `npm run test:e2e`。
- 新增或修改页面必须覆盖 loading、empty、error、success、disabled 和登录拦截状态。
- 非 MVP / 非本批能力只能灰态或 toast，不得接入不存在的 API。
