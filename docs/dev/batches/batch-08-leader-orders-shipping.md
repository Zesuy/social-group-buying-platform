# Batch 8：团长订单管理与发货

## 1. 目标

团长可查看自己店铺订单并对已支付订单发货。

## 2. 前置条件

Batch 7 已完成；买家可完成模拟支付，订单可进入 `paid` 状态。

## 3. 涉及接口

`GET /api/v1/my/store/orders`；`GET /api/v1/my/store/orders/{orderId}`；`POST /api/v1/my/store/orders/{orderId}/ship`。

## 4. 涉及数据表

`orders`、`order_items`、`shipments`、`stores`、`leaders`。

## 5. 实现任务

实现团长订单列表和详情；校验订单属于当前团长自己的店铺；只有 `paid` 状态可发货；发货记录创建和订单状态更新在同一事务中完成。

## 6. 测试要求

MockMvc 覆盖列表、详情、发货成功、普通用户访问、跨店铺访问、重复发货、订单不可发货；Service 覆盖发货事务和状态校验。

## 7. 验收标准

发货成功后订单变为 `shipped`，创建一条 `shipments`；重复发货不重复创建记录。

## 8. 联调文档更新项

完成“团长发货”链路，补充物流公司、单号和发货状态展示。

## 9. 禁止事项

不实现非本批接口；不实现物流轨迹查询、批量发货导入、售后退款；不扩大 MVP 范围；不跳过测试；不把已知问题留到下一批；不私自修改 API 字段、状态枚举、错误码、金额口径；需要修改上游文档时，必须明确说明原因。
