# Batch 3A：商品管理

## 1. 目标

团长可管理自己店铺下的商品，为后续发布团购提供可复用商品资产。

## 2. 前置条件

Batch 2 已完成；用户可创建店铺并激活团长身份。

## 3. 涉及接口

`GET /api/v1/my/store/products`；`POST /api/v1/my/store/products`；`GET /api/v1/my/store/products/{productId}`；`PATCH /api/v1/my/store/products/{productId}`；`DELETE /api/v1/my/store/products/{productId}`。

## 4. 涉及数据表

`stores`、`products`。

## 5. 实现任务

实现当前店铺商品列表、创建、详情、更新、软删除；所有 `/my/store/products/**` 接口从 token 获取当前团长自己的店铺；校验商品归属、金额字段、库存非负和商品状态；删除商品不得影响后续历史订单快照口径。

## 6. 测试要求

MockMvc 覆盖商品列表、创建、详情、更新、删除、未登录、非团长、跨店铺商品拒绝、参数错误；Service 覆盖商品归属、库存非负、金额字段和软删除状态。

## 7. 验收标准

团长只能管理自己店铺商品；普通用户不能访问商品管理接口；商品基础信息可被 Batch 3B 发布团购时复用。

## 8. 联调文档更新项

在“发布普通团购”链路中补充复用已有商品前的商品创建和选择说明；如前端需要独立商品管理入口，补充商品管理联调小节。

## 9. 禁止事项

不实现非本批接口；不实现复杂 SKU、商品审核、商品库分类；不扩大 MVP 范围；不跳过测试；不把已知问题留到下一批；不私自修改 API 字段、状态枚举、错误码、金额口径；需要修改上游文档时，必须明确说明原因。
