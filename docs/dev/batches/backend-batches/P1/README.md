# P1 后端 batch 总览

> 范围来源：`docs/功能需求定义.md` 的 P1 能力。P1 目标是补齐体验和运营效率，不改变 MVP 已冻结的核心交易闭环。

## 1. 拆批原则

P1 后端按依赖顺序拆成 5 个 batch：

| 顺序 | Batch | 主题 | 核心能力 |
|---:|---|---|---|
| 1 | `batch-01-product-library-search-favorites.md` | 商品库、分类搜索、收藏浏览 | 商品库效率和买家发现基础 |
| 2 | `batch-02-groupbuy-draft-presale-permission-share.md` | 团购草稿、预售、权限、复制、分享 | 团长发布效率和私域传播控制 |
| 3 | `batch-03-cart-order-idempotency.md` | 购物车和订单幂等 | 买家批量购买与重复提交保护 |
| 4 | `batch-04-coupon-member-growth.md` | 优惠券 / 红包和会员成长 | 轻量营销和会员运营 |
| 5 | `batch-05-after-sale-refund-regression.md` | 售后退款和 P1 回归冻结 | 交易后链路补齐和阶段收口 |

## 2. Grill-me 质询结论

本轮按 `grill-me` 思路先追问再拆分，结论如下：

| 质询问题 | 结论 |
|---|---|
| P1 是否要纳入真实微信支付？ | 不纳入本轮 5 个后端 batch。`功能需求定义.md` 的 P1 重点没有要求真实支付，且真实微信支付依赖外部平台、回调、退款和对账，建议单独立支付专项。 |
| 商品库、搜索和分类能否分开？ | 不建议分开。分类字段会同时影响商品库筛选和公共商品流搜索，先统一完成可以减少后续接口返工。 |
| 预售团购是否意味着定金尾款？ | P1 仅做最小预售语义：`groupType=presale`、预售时间、发货承诺和页面标识。暂不做定金、尾款、分阶段支付。 |
| 购物车为什么放在优惠券前？ | 优惠券需要接入订单预览和创建订单。购物车会改变预览入参和多商品结算形态，应先稳定购物车到订单的契约。 |
| 优惠券和会员成长为什么同批？ | 两者都挂在支付成功和订单金额口径上。放在同批可以统一处理 `discountAmount`、`payAmount`、会员累计消费和成长值。 |
| 分享海报后端要不要生成图片？ | P1 后端先提供分享 token、落地页校验和海报数据，不强制生成 bitmap 图片。海报渲染优先由前端完成，后端图片生成可后续专项。 |
| 售后退款为什么最后做？ | 售后会回写订单、支付、库存、优惠券和会员统计，必须等购物车、优惠券、会员成长等交易口径稳定后再接。 |

## 3. 本轮 P1 明确不做

- 真实微信支付、真实退款回调、对账和资金结算。
- 预售定金尾款、卡券团购、报名团购。
- 帮卖分销、佣金结算、积分商城。
- 公众号推送、服务通知、平台后台、风控审核。
- 后端强制生成复杂图片海报。

## 4. 通用要求

- 每个 batch 必须同步维护 `docs/前后端联调文档.md`。
- 如果新增或改变 P1 API 契约，必须同步更新 `docs/API设计.md` 和必要的数据模型说明。
- 所有新增接口必须有 MockMvc 测试，关键 Service 必须有单元测试。
- 涉及订单、支付、库存、优惠券、售后、会员统计的变更必须覆盖事务和异常路径。
- 不得为了 P1 新能力破坏 MVP 已完成接口的兼容性。
