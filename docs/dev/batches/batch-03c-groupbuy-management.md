# Batch 3C：团购管理

## 1. 目标

团长可查看、更新和结束自己店铺发布的团购。

## 2. 前置条件

Batch 3B 已完成；团长可发布普通团购。

## 3. 涉及接口

`GET /api/v1/my/store/group-buys`；`GET /api/v1/my/store/group-buys/{groupBuyId}`；`PATCH /api/v1/my/store/group-buys/{groupBuyId}`；`POST /api/v1/my/store/group-buys/{groupBuyId}/end`。

## 4. 涉及数据表

`stores`、`group_buys`、`group_buy_items`、`products`、`orders`、`order_items`。

## 5. 实现任务

实现我的团购列表和详情；实现团购展示字段、时间、发货时间等可更新内容；校验团购属于当前团长自己的店铺；实现结束团购；有订单后不得修改已下单团购商品价格，不得删除已下单团购商品；历史订单始终以订单快照为准。

## 6. 测试要求

MockMvc 覆盖列表、详情、更新、结束、未登录、非团长、跨店铺团购拒绝、不可结束状态；Service 覆盖团购归属、状态流转、已下单团购商品价格保护和结束团购规则。

## 7. 验收标准

团长只能管理自己店铺团购；`published` 团购可以结束为 `ended`；产生订单后的团购管理不会破坏订单快照和价格口径。

## 8. 联调文档更新项

在“发布普通团购”链路后补充我的团购列表、我的团购详情、更新团购和结束团购的联调说明。

## 9. 禁止事项

不实现非本批接口；不实现复杂内容模块、复制团购权限、隐私权限细分、帮卖分销；不扩大 MVP 范围；不跳过测试；不把已知问题留到下一批；不私自修改 API 字段、状态枚举、错误码、金额口径；需要修改上游文档时，必须明确说明原因。
