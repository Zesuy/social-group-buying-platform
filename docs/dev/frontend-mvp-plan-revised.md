# 前端 MVP 总计划（页面形态修正版，不拆 batch）

> 目的：在后端 Batch 10 接口冻结后，为前端 MVP 提供完整、不拆 batch 的总规划。后续如需执行，再基于本文拆分 `frontend-batch-00`、`frontend-batch-01` 等执行文档。
>
> 本版重点补充快团团小程序参考截图中的页面形态：5 个底部 Tab、一键开团入口、消息空态、发布团购三段式、首页/详情/团长主页的卡片化和信任信息展示。

## 1. 前端总体目标

实现一个微信私域社群团购 H5 MVP，体验接近小程序，覆盖以下闭环：

- 未登录用户可以浏览首页、团购详情、团长主页。
- 用户模拟登录后可以下单、管理地址、查看订单、模拟支付、取消订单、确认收货。
- 普通用户可以创建店铺，创建成功后成为团长。
- 团长可以发布普通团购、管理轻量商品、管理团购、查看店铺订单并发货。
- 用户可以订阅团长、取消订阅团长、查看基础会员卡。
- 全链路与后端冻结接口 `/api/v1/**` 对齐。
- 不实现非 MVP 能力：真实微信支付、购物车、优惠券/红包、完整售后退款、帮卖分销、积分商城、公众号真实推送、平台后台。

视觉目标：

- 整体接近快团团小程序：浅灰背景、白底圆角卡片、绿色主操作、橙红价格、底部固定 Tab、强商品图片、强团长信任信息。
- 不追求像素级复刻，但页面信息层级、入口位置、操作路径要贴近参考截图。
- 非 MVP 入口可以作为灰态/占位展示，但不能接入真实功能和新接口。

## 2. 技术栈与工程骨架

前端工程放在 `frontend/`：

- Vue 3 + Vite + TypeScript
- Vue Router
- Pinia
- Vant
- Axios
- Vitest
- Playwright
- ESLint / Prettier
- 可选：dayjs 用于时间格式化

建议目录结构：

```text
frontend/
├── src/
│   ├── api/
│   ├── assets/
│   ├── components/
│   ├── composables/
│   ├── constants/
│   ├── router/
│   ├── stores/
│   ├── styles/
│   ├── types/
│   ├── utils/
│   ├── views/
│   ├── App.vue
│   └── main.ts
├── tests/
│   ├── unit/
│   └── e2e/
├── index.html
├── vite.config.ts
├── package.json
└── tsconfig.json
```

核心原则：

- API 类型全部用 TypeScript 显式建模。
- 金额统一按整数分处理，展示时通过 `formatAmount(2990) => ¥29.90`。
- 状态枚举统一映射中文展示，不在页面里散落字符串。
- 登录态由 Pinia + localStorage 维护。
- 所有需要登录的页面和操作统一走登录拦截。
- 所有异步操作有 loading、disabled、防重复点击、错误提示。
- 页面优先按 375px 移动端设计，兼容 768px 宽屏，不做复杂桌面后台。
- 底部固定栏、购买栏、发布栏必须处理 safe-area。

## 3. 全局基础能力

先建立这些底层能力：

- Axios 实例：统一 `baseURL`、`Authorization` 注入、响应解包、错误标准化。
- API 响应类型：`ApiResponse<T>`、`PageResponse<T>`、`ApiError`。
- 错误码映射：`UNAUTHORIZED`、`VALIDATION_ERROR`、`LEADER_REQUIRED`、`STORE_FORBIDDEN`、`INSUFFICIENT_STOCK` 等统一转中文。
- Auth store：保存 `accessToken`、当前 `user`、`leader`、`store`。
- App 启动恢复登录态：有 token 时调用 `GET /api/v1/me`。
- 登录拦截器：操作型拦截支持登录后恢复原动作。
- 路由守卫：页面级 `meta.requiresAuth`、`meta.requiresLeader`。
- Toast / Dialog / Loading 封装：统一体验。
- 分页加载 composable：用于首页、订单、商品、团购、订阅、会员卡列表。
- 状态格式化工具：订单状态、支付状态、团购状态、配送方式。
- 空状态组件：列表为空、未登录、未开店、无地址、无会员卡、暂无消息。
- 页面壳组件：顶部导航、底部 Tab、固定底部操作栏、安全区 padding。
- 协议勾选组件：发布团购和下单确认均可复用。

## 4. 页面形态修正规则

本项目不是做传统商城后台，也不是做普通 H5 商城，而是做“用户可成为团长”的社群团购 H5。因此页面形态需要满足以下规则：

1. 首页要像信息流，不像商品后台列表。商品卡片要突出团长信息、商品图、价格、热度、订阅/分享/去跟团操作。
2. 一键开团必须作为底部一级入口出现，但 MVP 只开放普通团购。
3. 发布团购必须是“介绍 / 商品 / 设置”三段式，不要做成单页长表单。
4. 消息页保留为底部一级入口，但 MVP 只做空态和提醒引导，不做真实通知。
5. 个人中心同时承载买家能力和团长经营入口，不单独做复杂后台首页。
6. 商品管理仅作为发布团购的轻量复用能力，不做完整商品库、分类管理、批量导入。
7. 所有非 MVP 功能若出现入口，必须灰态/占位，不得接入真实逻辑。

## 5. 路由规划

### 5.1 一级 Tab 路由

```text
/                         首页
/orders                   订单
/open-group               一键开团
/messages                 消息
/profile                  个人中心
```

底部 Tab 固定 5 个：

| Tab | 路由 | MVP 处理 |
|---|---|---|
| 首页 | `/` | 展示公开已发布团购信息流 |
| 订单 | `/orders` | 登录后展示当前用户订单 |
| 一键开团 | `/open-group` | 普通团购可进入，预售/卡券/报名置灰 |
| 消息 | `/messages` | 只做空态和提醒横幅，不做真实通知 |
| 个人中心 | `/profile` | 用户资产 + 团长经营入口 |

### 5.2 公共浏览路由

```text
/group-buys/:id           团购详情
/leaders/:id              团长主页
/login                    模拟登录
/member-cards             会员卡
/subscriptions            我的订阅
```

### 5.3 买家链路路由

```text
/checkout                 下单确认
/addresses                地址列表
/addresses/new            新增地址
/addresses/:id/edit       编辑地址
/orders/:id               订单详情
```

### 5.4 团长链路路由

```text
/store/create             创建店铺
/leader/store             我的店铺资料
/leader/products          轻量商品管理
/leader/products/new      新建商品
/leader/products/:id/edit 编辑商品
/leader/group-buys        我的团购
/leader/group-buys/new    发布团购
/leader/group-buys/:id    我的团购详情
/leader/orders            团长订单列表
/leader/orders/:id        团长订单详情 / 发货
```

说明：原“团长工作台”不作为底部 Tab。团长入口主要从“一键开团”和“个人中心”进入，避免出现传统后台感。

## 6. 页面计划

### 6.1 首页

目标：公开展示已发布团购，形成类似快团团的信息流首页。

接口：

- `GET /api/v1/group-buys?page=1&pageSize=20`
- `POST /api/v1/leaders/{leaderId}/subscription`
- `DELETE /api/v1/leaders/{leaderId}/subscription`

页面内容：

- 顶部品牌区：Logo / 平台名 / 搜索占位。
- 公众号或提醒横幅：仅视觉提示，可关闭，不接公众号能力。
- 频道 Tab：全部、视频、最新。
  - MVP 只请求默认团购列表。
  - 视频、最新可以作为静态筛选壳或灰态，不新增后端依赖。
- 分类横滑：综合、生鲜、食品、服饰、美妆、百货等。
  - MVP 可不做真实筛选，点击后 toast “后续开放”或仅本地高亮。
- 团购卡片流：
  - 团长头像、团长名、店铺名。
  - 信任标签：回复超快、回头客多、金牌团长等可用静态标签或后端已有字段为空时隐藏。
  - 商品标题、封面图、最低价、销量、已团/浏览数。
  - 操作按钮：订阅、分享、去跟团。
- 下拉刷新、触底加载。
- 空状态：暂无团购。
- 点击卡片进入团购详情。
- 点击团长信息进入团长主页。
- 点击订阅时触发登录拦截。

MVP 边界：

- 不做真实搜索。
- 不做真实分类筛选。
- 不做真实领券。
- 不做购物车浮窗。
- 分享按钮可以调用系统复制链接或显示“后续开放”，不生成海报。

### 6.2 团购详情页

目标：承接首页/团长主页点击，完成商品转化和购买入口。

接口：

- `GET /api/v1/group-buys/{groupBuyId}`
- `POST /api/v1/leaders/{leaderId}/subscription`
- `DELETE /api/v1/leaders/{leaderId}/subscription`

页面内容：

- 顶部商品封面 / 图文区域。
- 返回、收藏占位、分享占位等小程序感操作按钮。
- 团长 / 店铺信任区域：头像、名称、店铺名、订阅按钮。
- 团购标题、介绍、价格、配送方式、发货时间、结束时间。
- 商品列表：名称、图片、价格、库存、销量、规格文本。
- 购买数量选择。
- 底部固定操作栏：主页、订单、跟团购买。
- 未登录点击订阅或购买时触发登录拦截。
- 已结束、库存不足时禁用购买并展示原因。

MVP 边界：

- 不做红包/优惠券弹窗。
- 不做购物车。
- 不做真实收藏。
- 不做视频轮播。
- 底部“跟团购买”进入 `/checkout`，不拉起真实微信支付。

### 6.3 团长主页

目标：展示团长信任信息、店铺信息、团购列表和订阅状态。

接口：

- `GET /api/v1/leaders/{leaderId}/homepage?page=1&pageSize=20`
- `POST /api/v1/leaders/{leaderId}/subscription`
- `DELETE /api/v1/leaders/{leaderId}/subscription`

页面内容：

- 顶部团长背景区：头像、昵称、简介、身份标签。
- 店铺摘要：店铺名、默认配送方式、简介。
- 信任信息：关注数、会员数、累计销量等有字段则展示，无字段则隐藏或显示 0。
- 订阅按钮。
- 客服按钮占位：不接真实客服，可 toast “暂未开放”。
- 会员权益 / 积分商城入口作为灰态/占位，不接真实积分。
- 团购列表：默认、上新、销量三个 Tab 可做视觉壳。
  - MVP 默认请求团长主页接口返回的团购列表。
  - 上新/销量可仅前端排序或后续开放。
- 空状态：团长暂未发布团购。

MVP 边界：

- 不做编辑主页。
- 不做订阅邀请卡。
- 不做新人红包。
- 不做积分商城。
- 不做客服会话。

### 6.4 模拟登录页 / 登录弹窗

接口：

- `POST /api/v1/auth/mock-login`
- `GET /api/v1/me`

页面内容：

- 昵称、手机号、头像 URL。
- 可提供“买家测试用户”“团长测试用户”快捷填充。
- 登录成功保存 token，刷新当前用户，恢复之前操作或返回个人中心。
- 表单校验：昵称、手机号必填。

### 6.5 一键开团页

目标：作为底部一级入口，把普通用户引导到创建店铺和发布普通团购流程。

接口：

- `GET /api/v1/me`
- `GET /api/v1/my/store`

页面内容：

- 顶部提醒横幅：打开提醒，官方活动早知道。仅视觉提示，可关闭。
- 插画/标题：请选择开团类型。
- 开团教程、新人攻略入口：可展示静态说明或 toast，不接复杂教程页面。
- 开团类型卡片：
  - 普通团购：可点击。
  - 预售团购：置灰，显示“后续开放”。
  - 卡券团购：置灰，显示“后续开放”。
  - 报名团购：置灰，显示“后续开放”。
- 点击普通团购后的逻辑：
  1. 未登录：触发登录拦截。
  2. 已登录但未创建店铺：进入 `/store/create`。
  3. 已创建店铺：弹出“我是新手，体验开团 / 我有经验，自己开团”动作面板。
  4. 两个选项 MVP 都进入 `/leader/group-buys/new`。

MVP 边界：

- 不实现预售、卡券、报名。
- 不实现真实开团教程内容库。
- 不实现禁发商品规范详情页，可显示静态提示或外链占位。

### 6.6 消息页

目标：保留底部消息入口，呈现空态，不接真实通知。

接口：无。

页面内容：

- 顶部标题：消息。
- 提醒横幅：关注公众号，收到活动和订单、物流通知。
- 关注按钮：toast “MVP 暂不接公众号”。
- 空状态插画：暂无消息。
- 底部 Tab 高亮消息。

MVP 边界：

- 不做真实活动通知。
- 不做订单/物流消息推送。
- 不做客服消息。
- 不新增消息接口。

### 6.7 下单确认页

接口：

- `GET /api/v1/my/addresses`
- `POST /api/v1/orders/preview`
- `POST /api/v1/orders`

页面内容：

- 地址卡片：无地址时引导新增地址。
- 商品明细：图片、名称、单价、数量。
- 数量加减。
- 备注输入。
- 协议勾选。
- 金额区：商品总额、优惠金额 0、应付金额。
- 底部固定操作栏：立即支付 / 提交订单。
- 进入页面先预览，不创建订单。
- 切换地址或数量后重新预览。
- 提交订单按钮 loading + disabled 防重复点击。
- 成功后进入订单详情页。

页面形态建议：

- 可以做成完整页面，也可以在视觉上模拟“购买确认浮层”。
- 实现上仍按 `/checkout` 页面处理，保证路由、刷新、登录拦截和预览逻辑稳定。

MVP 边界：

- 按钮文案可以是“立即支付”，但实际只创建订单；订单详情页再调用模拟支付。
- 不做加入购物车。
- 不做真实微信支付。
- 不做优惠抵扣。
- 不做“多买一份送给他人”。

### 6.8 地址管理

接口：

- `GET /api/v1/my/addresses`
- `POST /api/v1/my/addresses`
- `PATCH /api/v1/my/addresses/{addressId}`
- `DELETE /api/v1/my/addresses/{addressId}`

页面内容：

- 地址列表，默认地址标识。
- 新增 / 编辑地址表单。
- 删除确认弹窗。
- 从下单页进入时，选择地址后返回下单确认页。
- 首个地址自动默认由后端处理，前端刷新列表即可。

### 6.9 我的订单列表

接口：

- `GET /api/v1/my/orders?status=xxx&page=1&pageSize=20`

页面内容：

- 顶部标题：我购买的订单。
- 搜索框占位：搜索商品名，不做真实搜索。
- 状态 Tab：全部、待支付、待发货、已发货、已完成、已取消。
- 订单卡片：订单号、状态、商品摘要、金额、操作按钮。
- 待支付订单显示“去支付”“取消订单”。
- 已发货显示“确认收货”。
- 点击进入订单详情。
- 空状态按 tab 展示。
- 空状态推荐区可展示“团长热卖好货”静态壳，但不接推荐算法。

MVP 边界：

- 不做售后 tab 的真实售后功能。
- 不做订单搜索。
- 不做猜你喜欢真实推荐。

### 6.10 订单详情页

接口：

- `GET /api/v1/my/orders/{orderId}`
- `POST /api/v1/orders/{orderId}/simulate-pay`
- `POST /api/v1/orders/{orderId}/cancel`
- `POST /api/v1/orders/{orderId}/complete`

页面内容：

- 订单状态步骤条：待支付、待发货、已发货、已完成。
- 地址快照。
- 商品快照。
- 金额明细。
- 订单号、支付时间、发货时间、完成时间。
- 待支付：模拟支付、取消订单。
- 已发货：确认收货。
- 已完成 / 已取消：只展示结果。
- 重复点击通过 loading + disabled 防住，后端错误也正常提示。

### 6.11 个人中心

接口：

- `GET /api/v1/me`

页面内容：

- 用户头像、昵称、身份标签。
- 顶部资产 / 入口：我的订单、会员卡、地址管理。
- 买家功能区：收货地址、会员卡、我的订单、官方客服占位、设置占位。
- 团长经营入口：
  - 未开店：显示“创建店铺 / 成为团长”。
  - 已开店：显示“我的店铺”“发布团购”“我的团购”“团长订单”。
- 猜你喜欢区域可用首页团购卡片样式复用，也可先不展示。
- 退出登录。

MVP 边界：

- 不做钱包。
- 不做优惠券。
- 不做收藏与浏览。
- 不做官方客服真实会话。
- 不做社群。

### 6.12 创建店铺页

接口：

- `POST /api/v1/stores`
- `GET /api/v1/me`

页面内容：

- 店铺名称。
- 店铺 logo URL / 上传占位。
- 店铺简介。
- 默认物流方式。
- 一键开店按钮。
- 创建成功后刷新 `me`，进入发布团购页或我的店铺资料页。
- `STORE_ALREADY_EXISTS` 时提示并进入已有店铺。

MVP 边界：

- 不做帮卖分销设置。
- 不做真实图片上传，可先使用 URL 输入或默认图。
- 不做店铺认证。

### 6.13 我的店铺资料

接口：

- `GET /api/v1/my/store`
- `PATCH /api/v1/my/store`

页面内容：

- 店铺名称、logo、简介、默认物流方式。
- 保存后刷新 `me` 和店铺资料。
- 未开店返回创建店铺页。

### 6.14 轻量商品管理

接口：

- `GET /api/v1/my/store/products`
- `POST /api/v1/my/store/products`
- `GET /api/v1/my/store/products/{productId}`
- `PATCH /api/v1/my/store/products/{productId}`
- `DELETE /api/v1/my/store/products/{productId}`

页面内容：

- 商品列表：封面、名称、基础价、库存、状态。
- 新建 / 编辑商品表单。
- 删除确认。
- 发布团购时可以从此列表选择已有商品。

MVP 定位：

- 商品管理只服务“发布团购时复用已有商品”。
- 不做完整商品库。
- 不做分类管理。
- 不做商品素材库。
- 不做批量选择。
- 不做从商品库导入。

### 6.15 发布团购页（三段式）

目标：按参考截图改为“介绍 / 商品 / 设置”三段式表单。

接口：

- `POST /api/v1/my/store/group-buys`
- 可选：`GET /api/v1/my/store/products` 用于复用商品

页面结构：

```text
发布团购
├── 介绍
├── 商品
└── 设置
```

#### 介绍 Tab

字段：

- 团购标题。
- 团购介绍。
- 封面图 URL。

展示：

- 可放“复制已有团购”占位按钮，但 MVP 不实现复制。
- 可放“添加大图 / 小图 / 视频 / 标签”等占位图标，但 MVP 不实现内容模块编辑。

#### 商品 Tab

字段：

- 商品来源：内联创建商品 / 选择已有商品。
- 内联商品字段：名称、描述、封面图、基础价、库存。
- 团购商品字段：展示名、团购价、团购库存、排序。

规则：

- 至少一个团购商品。
- `groupStock >= 0`。
- 不允许 `-1` 表示不限库存。
- 金额用整数分。

#### 设置 Tab

字段：

- 配送方式。
- 发货时间。
- 开始时间。
- 结束时间。
- 协议勾选。

操作：

- 发布团购。
- 发布成功进入我的团购详情或团长主页。

MVP 边界：

- 只支持普通团购。
- 不做保存并预览。
- 不做优惠设置。
- 不做帮卖设置。
- 不做隐私权限细分。
- 不做复制团购。
- 不做开团通知推送。
- 不做内容模块排序。

### 6.16 我的团购管理

接口：

- `GET /api/v1/my/store/group-buys?status=published|ended`
- `GET /api/v1/my/store/group-buys/{groupBuyId}`
- `PATCH /api/v1/my/store/group-buys/{groupBuyId}`
- `POST /api/v1/my/store/group-buys/{groupBuyId}/end`

页面内容：

- Tabs：进行中、已结束。
- 团购卡片：标题、状态、商品数、时间。
- 详情页可编辑基础信息。
- 结束团购需要确认弹窗。
- 已产生订单后修改价格的错误 `BUSINESS_RULE_VIOLATION` 要清晰提示。

### 6.17 团长订单列表与发货

接口：

- `GET /api/v1/my/store/orders?status=xxx&page=1&pageSize=20`
- `GET /api/v1/my/store/orders/{orderId}`
- `POST /api/v1/my/store/orders/{orderId}/ship`

页面内容：

- 订单列表按状态筛选。
- 详情展示买家地址快照、商品快照、支付状态。
- 仅 `orderStatus=paid` 显示发货表单。
- 发货字段：配送方式、物流公司、物流单号。
- 发货成功刷新订单状态为 `shipped`。
- 重复发货、非自己店铺订单、非可发货状态都要正常提示。

### 6.18 订阅与会员卡

接口：

- `GET /api/v1/my/subscriptions`
- `GET /api/v1/my/member-cards`

页面内容：

- 我的订阅列表：团长 / 店铺卡片，进入团长主页。
- 会员卡列表：店铺、团长、会员关系基础信息。
- 空状态：暂无订阅 / 暂无会员卡。
- 不做等级升级、折扣、积分。

## 7. Pinia Store 规划

## 10. 样式系统

## 11. 非 MVP 入口处理规则

为了贴近截图，但不扩展范围，以下入口统一使用灰态、占位或 toast：

| 入口 | MVP 处理 |
|---|---|
| 预售团购 | 置灰，提示“后续开放” |
| 卡券团购 | 置灰，提示“后续开放” |
| 报名团购 | 置灰，提示“后续开放” |
| 优惠券 / 红包 | 不展示或灰态，不影响金额 |

| 权限设置 | 不进入页面 |

| 公众号关注 | 展示提醒横幅，不接真实公众号 |
| 消息通知 | 只做空态，不接真实消息 |
| 分享海报 | 可复制链接，不生成海报 |

## 12. 联调策略

前端开发期间建议保持三类环境变量：

```text
VITE_API_BASE_URL=http://localhost:8080
VITE_APP_ENV=development
VITE_ENABLE_MOCK=false
```

联调顺序按完整业务闭环走：

1. 买家模拟登录。
2. 团长模拟登录。
3. 团长创建店铺。
4. 团长创建商品或发布团购时内联创建商品。
5. 团长发布普通团购。
6. 未登录访问首页、详情、团长主页。
7. 买家登录、创建地址。
8. 买家下单预览、创建订单。
9. 买家模拟支付。
10. 团长查看订单并发货。
11. 买家确认收货。
12. 买家查看会员卡。
13. 买家订阅 / 取消订阅团长。
14. 消息页空态、一键开团非 MVP 入口灰态检查。

每条链路都要对照 `docs/前后端联调文档.md` 更新前端入口、调用顺序和实际结果。

## 17. 页面验收清单

前端 MVP 总体验收时，至少检查：

- 底部是否为 5 Tab：首页、订单、一键开团、消息、个人中心。
- 首页是否具备小程序式信息流：提醒横幅、频道、分类、团购卡片、价格、团长信息、去跟团。
- 团购详情是否有团长信任区、商品信息、底部购买栏。
- 团长主页是否有团长资料、店铺资料、订阅按钮、团购列表、空状态。
- 一键开团是否只允许普通团购，其余类型置灰。
- 发布团购是否为介绍/商品/设置三段式。
- 下单是否先预览，再创建订单，提交按钮防重复点击。
- 订单列表/详情是否覆盖待支付、已支付、已发货、已完成、已取消。
- 团长订单是否只能发货 paid 状态订单。
- 消息页是否为空态，不调用不存在的消息接口。
- 会员卡是否只做基础展示，不做折扣/积分计算。
- 所有非 MVP 入口是否灰态或提示后续开放。

本计划的核心是：前端不发明新产品，不扩大 MVP 范围；在后端已冻结接口之上，用更接近快团团小程序的信息架构和页面形态，把交易闭环、开店闭环和基础私域关系完整呈现出来。
