# 测试规则

## 1. 总体要求

| 范围 | 要求 |
|---|---|
| 所有 API | 每个接口至少有 MockMvc 测试 |
| 关键 Service | 必须有单元测试 |
| 订单 / 支付 / 库存 / 发货 | 必须有事务和异常路径测试 |
| 每个 batch | 测试必须随 batch 完成，不得留到下一批 |

## 2. MockMvc 测试

MockMvc 必须覆盖：

| 类型 | 要求 |
|---|---|
| 成功响应 | HTTP 状态、`success`、`data`、`traceId` 和关键字段 |
| 错误响应 | HTTP 状态、错误码、错误消息和响应结构 |
| 登录接口 | 模拟登录、获取当前用户、未登录、无效 token |
| 公开接口 | 未登录可访问 |
| 私有接口 | 未登录、非资源所有者、非团长身份 |

## 2.1 API 契约校验

`docs/openapi/groupshop-api.yaml` 是机器可校验的 API 契约源，必须与 `docs/API设计.md`、实现和联调文档保持一致。

| 场景 | 要求 |
|---|---|
| 每个 batch | 只追加或修改本批已实现接口，不提前加入后续 batch 接口 |
| Controller 测试 | 合法业务请求必须追加 OpenAPI 契约校验 |
| 故意非法请求 | 继续断言统一错误响应；非法请求本身不作为符合请求 schema 的交互 |
| 契约文件 | 必须能被 OpenAPI 校验器加载 |
| 完成输出 | 增加 `API 契约校验结果`，说明契约是否通过、是否修改契约 |
| 区分合法请求 vs 非法请求 | `contractResult()` 只加在结构合法的请求上（即请求体、认证头格式符合 OpenAPI schema）；故意传入缺失字段、错误认证 scheme、不兼容类型的请求，不加 `contractResult()`（因为请求本身不匹配 request schema，校验器没有比对依据） |

**合法请求 + 业务错误**（例如 valid token 但无权限）→ 加 `contractResult()`，因为请求结构合法，只是业务校验不通过。**非法请求**（例如缺少必填字段、空 phone、没有 Authorization 头、Basic 而非 Bearer）→ 不加 `contractResult()`。

## 3. Service 单元测试

| Service | 必测规则 |
|---|---|
| 登录 / 当前用户 | 用户创建或查询、token 解析、团长和店铺摘要 |
| 店铺 | 一个用户一个团长身份、一个团长一个店铺、重复创建 |
| 商品 | 商品归属、金额字段、库存非负、软删除状态 |
| 团购 | 普通团购发布、内联创建商品、商品归属、库存非负、状态流转 |
| 订单 | 订单预览、创建订单、地址快照、商品快照、状态初始化 |
| 支付 | 状态校验、库存扣减、销量增加、会员关系创建 / 更新 |
| 发货 | 团长权限、订单状态校验、发货记录和状态更新 |
| 订阅 / 会员 | 重复订阅处理、会员卡展示数据 |

## 4. 事务测试

订单创建、模拟支付、发货必须验证事务一致性。支付中任一步失败时，订单状态、库存、销量、会员关系必须保持一致；发货失败时不得创建半成品发货记录。

## 5. 异常路径测试

必须覆盖参数错误、资源不存在、无权限、状态不允许、库存不足、重复动作和跨店铺访问。

## 6. 权限测试

| 场景 | 要求 |
|---|---|
| 未登录 | 需要登录的接口返回 `UNAUTHORIZED` |
| 普通用户 | 不能发布团购、管理商品、管理团长订单、发货 |
| 团长 | 只能管理自己的店铺、商品、团购和订单 |
| 买家 | 只能查看自己的订单和地址 |

## 7. 快照测试

订单创建必须保存商品快照和地址快照。商品、团购商品或地址后续修改 / 删除，不得影响历史订单展示和发货。

## 8. 库存测试

库存以 `group_buy_items.group_stock` 为支付扣减口径，`products.stock` 不参与支付扣减。必须覆盖库存不足、支付成功扣减库存、增加 `sold_count`。

## 9. 重复支付 / 重复发货测试

重复支付不能重复扣库存、不能重复增加销量、不能重复累计会员金额。重复发货不能重复创建发货记录。
