# 团购活动内容块设计

> 定位：本文用于预留和约束团购活动详情内容的表达方式。
>
> 当前阶段处于 P1 页面业务还债期，本文先作为产品、前端和后端的设计约束，不要求立即实现完整富文本能力。

## 1. 背景

本项目不是传统低价货架商城，而是面向私域经营者的社群团购活动平台。

团购详情页承载的不是单个商品参数，而是一次销售活动的说服过程：

- 团长为什么推荐；
- 商品为什么值得买；
- 产地、规格、口感、适用场景；
- 发货时间、配送范围、履约承诺；
- 生鲜、特产、季节性商品的图文说明；
- 微信群和朋友圈传播时需要的活动介绍。

如果只依赖 `title`、`coverImageUrl`、`description`、`price`、`stock` 等字段，首页卡片可以勉强成立，但团购详情页会显得薄，发布团购也会退化成后台表单。

## 2. 结论

当前不引入完整 HTML 富文本编辑器。

本项目应优先采用：

```text
结构化内容块，而不是自由 HTML 富文本
```

原因：

| 方案 | 判断 |
|---|---|
| HTML 富文本 | 编辑器重、移动端体验复杂、XSS 和图片处理成本高，不适合当前 P1 |
| Markdown | 对普通团长不友好，移动端编辑体验弱 |
| 纯 description | 实现简单，但表达能力不足，难以支撑详情页转化 |
| 结构化内容块 | 可控、易渲染、易测试、适合移动端和 AI 生成 |

## 3. 阶段策略

### 3.1 当前 P1 页面还债期

继续优先还 MVP 页面业务表达债。

本阶段不新增后端字段、不改接口、不实现内容块编辑器。

但页面设计必须预留内容块思想：

- 发布团购页的“活动介绍”应作为重要区块，不放在普通字段末尾；
- 团购详情页应按“活动说明、商品说明、发货说明、团长承诺”组织内容；
- 当前只有 `description` 时，前端应按段落渲染，而不是一行灰字带过；
- 首页卡片不展示长介绍，只展示活动标题、团长、价格、销量、时间等摘要。

### 3.2 P1.5 内容块能力

在 P1 关键页面校准完成后，再考虑新增内容块字段。

建议字段：

```text
group_buys.content_blocks
```

或在 API 层使用：

```text
contentBlocks
```

推荐 JSON 结构：

```json
[
  {
    "type": "paragraph",
    "text": "本团是云南产地直发牛油果，成熟度控制在可即食前后。"
  },
  {
    "type": "image",
    "url": "https://example.com/avocado.jpg",
    "caption": "实拍果型参考"
  },
  {
    "type": "section",
    "title": "发货说明",
    "text": "每周三统一截单，周四发货。"
  },
  {
    "type": "list",
    "items": ["适合做沙拉", "可搭配吐司", "收到后常温催熟"]
  }
]
```

### 3.3 P2 增强

P2 再考虑：

- 多图上传；
- 内容块排序；
- 草稿保存；
- 发布前预览；
- 分享卡片预览；
- 活动介绍模板；
- AI 辅助生成活动介绍。

## 4. 内容块类型

P1.5 建议只支持以下最小集合：

| 类型 | 用途 | 字段 |
|---|---|---|
| `paragraph` | 普通说明文字 | `text` |
| `section` | 小标题 + 说明 | `title`, `text` |
| `image` | 商品实拍、产地、细节 | `url`, `caption` |
| `list` | 卖点、注意事项、食用方式 | `items` |
| `deliveryNote` | 发货和履约说明 | `text` |

暂不支持：

- 任意 HTML；
- 表格；
- 视频；
- iframe；
- 自定义 CSS；
- 外链脚本；
- 复杂字号、颜色、字体编辑。

## 5. 前端发布体验

发布团购页不做传统富文本编辑器。

推荐交互：

```text
活动介绍
  添加文字说明
  添加图片
  添加小标题
  添加要点列表
  添加发货说明

商品
  选择或创建商品

设置
  活动价、库存、开始/结束时间、发货时间

发布
  预览后发布
```

每个内容块是一个可编辑卡片：

- 可编辑；
- 可删除；
- 可上移 / 下移；
- 有空内容校验；
- 图片块必须有图片 URL；
- 列表块至少 1 项。

## 6. 前端详情页渲染

团购详情页应按内容块顺序渲染。

渲染要求：

| 类型 | 渲染方式 |
|---|---|
| `paragraph` | 正文段落，支持换行 |
| `section` | 小标题 + 正文 |
| `image` | 图片 + 可选说明 |
| `list` | 要点列表 |
| `deliveryNote` | 发货说明卡片或强调区块 |

详情页内容顺序建议：

1. 商品封面和活动核心信息；
2. 团长 / 店铺信任区；
3. 发货说明；
4. 内容块；
5. 底部购买栏。

## 7. 后端与安全约束

若后续实现内容块，后端必须校验：

- `type` 只能来自白名单；
- 文本长度限制；
- 图片 URL 长度限制；
- 内容块数量限制；
- 列表项数量限制；
- 不接受任意 HTML；
- 不接受脚本、样式、iframe 等危险内容；
- 返回 API 时保持原始块顺序。

建议限制：

| 项目 | 建议 |
|---|---|
| 内容块数量 | 最多 20 个 |
| 段落文本 | 最多 1000 字 |
| 小标题 | 最多 40 字 |
| 图片说明 | 最多 80 字 |
| 列表项 | 最多 10 项 |
| 单个列表项 | 最多 80 字 |

## 8. 当前页面还债要求

在内容块正式实现前，当前页面还债仍应继续。

但后续调整首页、团购详情页、发布团购页时，必须避免把内容表达做死：

- 不要把团购详情页做成只有商品参数的页面；
- 不要把发布团购页做成普通后台字段表单；
- 不要在首页展示长富文本；
- 不要为了页面丰富添加无业务意义占位；
- 不要提前接入完整富文本编辑器。

优先级判断：

```text
先把 MVP 页面业务表达还清
同时在详情页和发布页预留内容块位置
再在 P1.5 / P2 实现结构化内容块
```
