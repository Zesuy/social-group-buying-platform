# 前端产品与页面设计准则

> 本文用于约束前端 MVP 和后续前端 batch 的页面设计、组件使用和交互取舍。
>
> 本项目不是传统低价商城，也不是普通货架电商。本项目定位为：**面向私域经营者的社群团购活动平台**。
>
> 前端实现必须同时遵循 `docs/功能需求定义.md`、`docs/API设计.md`、`docs/API风格规范.md`、`docs/页面与交互文档.md`、`docs/dev/h5-style-guide.md` 和本文。若页面表现与业务定位冲突，以本文对前端产品表达的约束为准。

---

## 1. 产品定位

本项目真正解决的问题是：

> 帮助有私域关系的普通用户、社区团长和小商家，通过一次次短周期团购活动，把微信社群、朋友圈、熟人关系和本地信任转化为订单、履约、订阅和复购。

因此，前端页面不能按传统商城理解为：

```text
商家上传商品 -> 用户搜索商品 -> 加购物车 -> 比价下单
```

而应按私域团购活动理解为：

```text
团长/小商家组织一次团购活动
-> 通过主页、微信群、分享链接传播
-> 用户基于信任和场景购买
-> 团长集中履约
-> 用户订阅、成为会员、后续复购
```

核心对象关系：

| 对象 | 前端表达 |
|---|---|
| 团购 | 一次可传播、可下单、可履约的销售活动 |
| 团长 | 用户信任和服务承诺的主体 |
| 店铺 | 团长长期经营资产和可信主页 |
| 商品 | 团购活动中的售卖资产，不是唯一中心 |
| 订单 | 团购活动的履约结果 |
| 订阅 / 会员 | 私域关系和复购入口 |

## 2. 前端设计总原则

### 2.1 不做传统商城感

前端不得把页面重点放在搜索、比价、货架陈列、SKU 丰富度和低价标签堆叠上。

MVP 页面应优先表达：

- 谁在组织这次团购；
- 这次团购活动卖什么；
- 为什么现在买；
- 活动何时开始、结束和发货；
- 已有多少人参与或购买；
- 用户如何快速下单；
- 用户如何订阅团长并复购；
- 团长如何快速发布和履约。

### 2.2 团购是活动，不只是商品

团购详情页和团购卡片不能做成普通商品卡。

团购必须体现：

| 信息 | 页面意义 |
|---|---|
| 活动标题 | 说明这次销售活动的主题 |
| 团长 / 店铺 | 建立信任主体 |
| 商品图和商品信息 | 让用户确认买什么 |
| 活动价 | 说明本次活动的成交价格 |
| 活动时间 | 形成明确购买窗口 |
| 团购库存 | 形成数量边界 |
| 销量 / 参与感 | 强化社群购买氛围 |
| 发货说明 | 解决生鲜、预售、集中配送的履约疑虑 |
| 订阅入口 | 沉淀复购关系 |

### 2.3 团长可信比平台感更重要

用户购买的原因不只是便宜，还包括信任团长、信任本地关系、信任社群口碑。

所以前端必须让团长信息在关键页面明显可见：

- 团长头像；
- 团长名称；
- 店铺名称；
- 店铺简介；
- 订阅状态；
- 当前开团数量或可见活动；
- 发货承诺或履约说明；
- 可复用的基础信任标签。

不得让团长信息只出现在角落、底部或二级页面。

### 2.4 轻商家效率优先

团长端页面不是复杂后台。发布团购、订单履约、店铺管理都应服务低门槛经营。

前端应避免：

- 后台管理系统式侧边栏；
- 大量表格；
- 复杂筛选器；
- 多层级配置；
- 类 ERP / WMS 的库存和履约表达。

团长端应优先使用移动端表单、步骤分组、卡片列表和底部主操作。

### 2.5 MVP 可以少，但不能虚

允许 MVP 功能少、页面简洁。

不允许为了“看起来丰富”保留无意义占位组件。组件必须服务当前页面主任务，否则应删除、隐藏或明确灰态。

## 3. 页面级设计准则

### 3.1 首页

首页不是低价商品商城首页，而是公开团购活动信息流。

首页首屏优先展示：

1. 平台身份和当前浏览场景；
2. 正在进行的团购活动；
3. 团长 / 店铺可信信息；
4. 一键开团入口；
5. 订单和个人中心入口。

团购卡片必须优先包含：

| 信息 | 要求 |
|---|---|
| 团长 / 店铺 | 在卡片上方或核心信息区可见 |
| 商品图 | 清晰、占比足够，不用装饰图替代 |
| 团购标题 | 表达活动主题，而非只有商品名 |
| 活动价 | 价格突出，但不得压倒所有信息 |
| 销量 / 库存 / 时间 | 至少展示一个活动状态信息 |
| 主操作 | 使用“去跟团”“查看团购”等活动语义 |

首页不得优先做成：

- 大搜索框 + 商品瀑布流；
- 低价标签集合；
- 平台补贴频道；
- 复杂分类商城；
- 购物车导向页面。

### 3.2 团购详情页

团购详情页是 MVP 转化核心页，不是普通商品详情页。

首屏必须回答：

- 这是谁发起的团购；
- 这次团购卖什么；
- 活动价是多少；
- 什么时候结束；
- 什么时候发货；
- 现在还能不能买；
- 如何下单。

推荐页面顺序：

1. 商品封面 / 活动主图；
2. 团购标题、活动价、销量、库存、倒计时或结束时间；
3. 团长 / 店铺信任区，包含订阅入口；
4. 发货说明、配送方式或履约承诺；
5. 商品和活动介绍；
6. 底部固定操作栏。

底部固定操作栏必须突出主操作：

- 可购买时：`跟团购买`；
- 需要登录时：点击后触发登录拦截并恢复购买；
- 不可购买时：展示明确原因，如已结束、库存不足、未发布。

### 3.3 团长主页 / 店铺页

团长主页不是传统店铺货架，而是团长的可信经营主页。

页面必须突出：

- 团长头像和名称；
- 店铺名称；
- 店铺简介；
- 订阅按钮；
- 当前正在进行的团购；
- 用户可复购、可关注的关系入口。

团长主页上的团购列表应表现为活动列表，而非普通商品列表。

### 3.4 发布团购页

发布团购页不是后台录入表单，而是创建一次私域销售活动。

页面必须按以下结构组织：

```text
介绍 -> 商品 -> 设置 -> 发布
```

各段含义：

| 段落 | 目标 |
|---|---|
| 介绍 | 说明这次活动为什么值得买 |
| 商品 | 选择或内联创建本次团购售卖商品 |
| 设置 | 配置活动价、库存、时间和发货说明 |
| 发布 | 校验后发布普通团购 |

发布页不得做成所有字段平铺的长表单。必须通过分组、步骤、折叠或区块降低认知负担。

MVP 只发布普通团购，不得提前接入预售、卡券、报名、帮卖、优惠券等非 MVP 能力。

### 3.5 个人中心页

个人中心同时服务买家和潜在团长，不能只做买家账户页。

必须包含：

- 当前用户信息；
- 我的订单入口；
- 我的订阅 / 会员卡入口；
- 创建店铺或我的店铺入口；
- 发布团购或团长经营入口；
- MVP 非支持能力的明确灰态或隐藏。

### 3.6 订单页

订单页要围绕履约状态组织，不应只展示普通电商订单卡。

买家订单重点：

- 待支付；
- 待发货；
- 已发货；
- 已完成；
- 已取消。

团长订单重点：

- 哪些订单需要发货；
- 对应哪个团购活动；
- 收货地址和发货信息；
- 防重复发货；
- 发货后的状态变化。

## 4. 组件使用准则

### 4.1 组件必须有业务角色

每个组件必须能回答：

```text
它服务哪个页面主任务？
它表达哪个业务对象？
它是否影响浏览、下单、发布、履约或复购？
```

答不上来的组件，不应进入页面。

### 4.2 Page 不是简单拼组件

页面文件可以组合组件，但页面必须负责信息层级和业务路径。

不得出现：

- 组件各自正确，但页面主任务不清；
- 组件堆叠后缺少首屏重点；
- 多个卡片争抢主操作；
- 页面看似完整但无法解释用户下一步做什么。

### 4.3 卡片要表达活动，不只是容器

团购卡片、订单卡片、店铺卡片、会员卡片应有不同信息层级，不得共享一个过于泛化的万能卡片样式。

卡片内间距和层级应遵循：

- 主体信息紧凑成组；
- 价格、状态、主操作突出；
- 辅助信息弱化但可读；
- 图文比例稳定；
- 点击区域足够大；
- 不使用多层嵌套卡片。

### 4.4 Vant 是基础，不是设计结果

可以优先使用 Vant 组件，但不能把使用 Vant 等同于完成 UI。

使用 Vant 时必须二次处理：

- 页面间距；
- 信息层级；
- 主次按钮；
- 底部固定栏 safe-area；
- 空状态；
- loading 和 disabled；
- 表单错误提示；
- 移动端触控尺寸。

## 5. 视觉与交互准则

### 5.1 视觉基调

MVP H5 体验接近微信内移动网页或小程序。

推荐基调：

| 元素 | 要求 |
|---|---|
| 页面背景 | 浅灰，减少大面积纯白刺眼感 |
| 内容区 | 白底为主，区块清晰 |
| 主色 | 稳定用于主操作，不滥用 |
| 价格色 | 可突出，但只服务成交信息 |
| 圆角 | 克制统一，避免过度装饰 |
| 阴影 | 轻量，避免漂浮感 |
| 字号 | 移动端正文可读，避免过小灰字 |
| 图像 | 商品图必须真实表达商品，不用抽象装饰替代 |

### 5.2 间距

前端应建立统一 spacing 习惯：

```text
4 / 8 / 12 / 16 / 20 / 24
```

页面常用建议：

| 场景 | 建议 |
|---|---|
| 页面左右边距 | 12px 或 16px |
| 卡片内边距 | 12px 或 16px |
| 同组信息间距 | 4px 或 8px |
| 不同区块间距 | 12px 或 16px |
| 底部固定栏上方内容 | 必须预留安全距离 |

不得每个组件随意写不同间距。

### 5.3 主操作

每个页面只应有一个最重要的主操作。

示例：

| 页面 | 主操作 |
|---|---|
| 首页 | 去跟团 / 一键开团 |
| 团购详情 | 跟团购买 |
| 创建店铺 | 一键开店 |
| 发布团购 | 发布团购 |
| 订单详情 | 模拟支付 / 确认收货 / 发货 |

次要操作必须视觉降级。

### 5.4 状态反馈

所有异步操作必须有：

- loading；
- disabled 防重复点击；
- 成功反馈；
- 失败原因；
- 可恢复路径。

关键操作包括：

- 登录；
- 订阅 / 取消订阅；
- 创建店铺；
- 发布团购；
- 下单；
- 模拟支付；
- 取消订单；
- 确认收货；
- 发货。

## 6. 占位与灰态规则

### 6.1 禁止无意义占位

不得出现只有视觉存在、无业务意义、无说明、无交互结果的组件。

以下属于不合格占位：

- 无功能的 banner；
- 永远为空的快捷入口；
- 与 MVP 无关的频道；
- 点击无反馈的按钮；
- 只有标题没有内容的区块；
- 为了撑页面高度添加的装饰模块。

### 6.2 非 MVP 能力处理

非 MVP 能力可以隐藏，也可以灰态展示，但必须满足：

- 不调用不存在的接口；
- 不误导用户认为当前可用；
- 点击后有明确提示；
- 不抢占主流程位置。

非 MVP 能力包括但不限于：

- 真实微信支付；
- 优惠券 / 红包；
- 完整售后退款；
- 购物车；
- 帮卖分销；
- 积分商城；
- 公众号真实推送；
- 平台后台。

## 7. 前端 batch 验收清单

每个前端 batch 完成时，除既有测试要求外，必须补充检查以下项目：

| 检查项 | 要求 |
|---|---|
| 业务定位 | 页面是否表达“私域团购活动”，而不是普通商城 |
| 页面主任务 | 用户进入页面后是否知道下一步做什么 |
| 团长信任 | 关键浏览和转化页面是否明显展示团长 / 店铺 |
| 活动属性 | 团购是否展示时间、价格、库存、发货或销量等活动信息 |
| 组件必要性 | 页面中是否存在无意义占位组件 |
| 主操作 | 每页是否只有一个最突出的主操作 |
| 移动端体验 | 触控区域、底部栏、safe-area 和滚动是否正常 |
| 状态反馈 | loading、disabled、empty、error 是否完整 |
| MVP 边界 | 是否误接入或强化非 MVP 功能 |
| 文档同步 | 如页面流程变化，是否同步更新 `docs/前后端联调文档.md` |

若任一项不通过，本 batch 不应视为完成。

## 8. 当前前端还债建议

如果当前前端已经出现组件占位、页面拼合不自然、业务表达偏商城的问题，应优先增加一个前端校准 batch。

建议顺序：

1. 审计所有页面，标记占位组件、传统商城倾向、业务表达错误和间距层级问题；
2. 建立最小 UI 规则，包括颜色、字号、间距、卡片、按钮、价格、状态标签和底部操作栏；
3. 优先重做首页、团购详情页、发布团购页；
4. 再整理团长主页、个人中心、订单页；
5. 最后再推进更细的组件复用。

校准目标：

```text
少一点虚假的丰富，多一点真实的业务表达。
页面可以简单，但必须让用户理解：
这是一次团购活动；
这是可信团长；
现在可以下单；
团长可以发布和履约；
用户可以订阅并复购。
```
