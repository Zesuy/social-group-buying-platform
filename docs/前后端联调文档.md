# 前后端联调文档

> 来源：基于 `docs/功能需求定义.md`、`docs/API风格规范.md`、`docs/API设计.md`、`docs/页面与交互文档.md` 和 `docs/MVP开发指南.md` 整理。
>
> 定位：本文用于记录 MVP 阶段前后端按功能链路联调的入口、前置条件、调用顺序、样例、成功判定、错误码和测试数据准备方式。
>
> 维护规则：本文按功能链路组织，不按 Controller 文件组织。每个后端 batch 完成后，必须同步更新对应链路，不等待所有后端接口完成后再统一联调。

## 1. 联调通用约定

### 1.1 基础信息

| 项目 | 约定 |
|---|---|
| API 前缀 | `/api/v1` |
| 数据格式 | JSON |
| 字段命名 | `camelCase` |
| 时间格式 | ISO 8601 字符串，如 `2026-06-24T12:00:00+08:00` |
| 金额格式 | 整数分，字段名以 `Amount` 结尾 |
| 登录头 | `Authorization: Bearer <accessToken>` |

### 1.2 成功响应

```json
{
  "success": true,
  "data": {},
  "traceId": "req_001"
}
```

### 1.3 错误响应

```json
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "参数校验失败",
    "details": {}
  },
  "traceId": "req_001"
}
```

### 1.4 常用错误码

| 错误码 | 场景 |
|---|---|
| `UNAUTHORIZED` | 未登录或令牌无效 |
| `VALIDATION_ERROR` | 参数校验失败 |
| `LEADER_REQUIRED` | 当前操作需要团长身份 |
| `STORE_ALREADY_EXISTS` | 当前用户已创建店铺 |
| `STORE_FORBIDDEN` | 不能操作他人店铺 |
| `GROUP_BUY_NOT_PURCHASABLE` | 团购不可购买 |
| `GROUP_BUY_ENDED` | 团购已结束 |
| `ITEM_NOT_IN_GROUP_BUY` | 下单商品不属于该团购 |
| `ADDRESS_FORBIDDEN` | 地址不属于当前用户 |
| `INSUFFICIENT_STOCK` | 库存不足 |
| `ORDER_NOT_PAYABLE` | 订单不可支付 |
| `ORDER_ALREADY_PAID` | 订单已支付 |
| `ORDER_NOT_SHIPPABLE` | 订单不可发货 |
| `ORDER_ALREADY_SHIPPED` | 订单已发货 |
| `SUBSCRIPTION_EXISTS` | 已订阅该团长 |

### 1.5 空库初始化说明

首次启动后端项目时，数据库迁移会自动执行：

| 步骤 | 说明 |
|---|---|
| 创建数据库 | 在 MySQL 中创建 `groupshop` 数据库（如 `CREATE DATABASE groupshop DEFAULT CHARSET utf8mb4`） |
| 配置连接 | 在 `application.yml` 中配置 `spring.datasource` 连接信息 |
| 启动应用 | 运行 `GroupshopApplication`，Flyway 自动执行 `classpath:db/migration/` 下的迁移脚本 |
| 迁移结果 | 空库执行 Flyway 后，一次性获得全部 12 张 MVP 核心表 |
| 验证迁移 | 启动日志中应看到 Flyway 迁移成功记录，或调用 `GET /api/v1/health` 确认应用可用 |

本地开发推荐配置：
- 数据库名称：`groupshop_dev`（通过 `application-dev.yml` profile 切换）
- Flyway 迁移位置：`src/main/resources/db/migration/`
- 迁移版本管理：后续 batch 只在确有必要时追加迁移文件，不反复修改 V1

## 2. 测试数据总口径

MVP 联调建议至少准备以下数据：

| 数据 | 准备方式 |
|---|---|
| 买家用户 | 调用 `POST /api/v1/auth/mock-login` 创建或获取 |
| 团长用户 | 使用另一个手机号调用 `POST /api/v1/auth/mock-login` |
| 店铺 | 团长用户调用 `POST /api/v1/stores` 创建 |
| 团购 | 团长用户调用 `POST /api/v1/my/store/group-buys` 发布普通团购 |
| 地址 | 买家用户调用 `POST /api/v1/my/addresses` 创建 |
| 订单 | 买家用户调用 `POST /api/v1/orders` 创建 |
| 已支付订单 | 买家用户调用 `POST /api/v1/orders/{orderId}/simulate-pay` |
| 已发货订单 | 团长用户调用 `POST /api/v1/my/store/orders/{orderId}/ship` |
| 会员关系 | 买家对某店铺订单模拟支付成功后自动创建或更新 |

说明：同一条联调链路中的 ID，如 `leaderId`、`storeId`、`groupBuyId`、`groupBuyItemId`、`addressId`、`orderId`，均以前一步接口响应为准，不写死。

## 3. 模拟登录与获取当前用户

| 项目 | 内容 |
|---|---|
| 页面入口 | 登录页 / 登录弹窗；应用启动恢复登录态 |
| 前置条件 | 无 |
| 调用顺序 | `POST /api/v1/auth/mock-login` -> 保存 `accessToken` -> `GET /api/v1/me` |

### 3.1 请求样例

```http
POST /api/v1/auth/mock-login
Content-Type: application/json
```

```json
{
  "nickname": "买家用户",
  "avatarUrl": "https://example.com/avatar.png",
  "phone": "13800000000"
}
```

```http
GET /api/v1/me
Authorization: Bearer mock_access_token
```

      "avatarUrl": "https://example.com/avatar.png",

### 3.3 成功判定

| 项目 | 判定 |
|---|---|
| 登录成功 | 返回 `accessToken` 和 `user` |
| 恢复登录态 | `GET /api/v1/me` 返回当前用户、团长摘要和店铺摘要 |
| 未创建店铺用户 | `hasLeader=false`，`leaderId=null`，`storeId=null` |

### 3.4 常见错误码

| 错误码 | 场景 |
|---|---|
| `VALIDATION_ERROR` | 昵称、手机号等参数不合法 |
| `UNAUTHORIZED` | `GET /api/v1/me` 未携带或携带无效 token |

### 3.5 测试数据准备方式

直接调用模拟登录接口。需要买家和团长两个身份时，使用不同手机号分别登录。

## 4. 创建店铺

| 项目 | 内容 |
|---|---|
| 页面入口 | 个人中心页 -> 创建店铺页 |
| 前置条件 | 用户已登录，且当前用户未创建店铺 |
| 调用顺序 | `GET /api/v1/me` 判断是否已有店铺 -> `POST /api/v1/stores` -> `GET /api/v1/me` 刷新身份 |

### 4.1 请求样例

```http
POST /api/v1/stores
Authorization: Bearer <accessToken>
Content-Type: application/json
```

```json
{
  "name": "某某的小店",
  "logoUrl": "https://example.com/logo.png",
  "description": "店铺简介",
  "defaultDeliveryType": "express"
}
```

### 4.2 响应样例

```json
{
  "success": true,
  "data": {
    "leader": {
      "id": 10,
      "displayName": "某某的小店",
      "avatarUrl": "https://example.com/logo.png"
    },
    "store": {
      "id": 20,
      "leaderId": 10,
      "name": "某某的小店",
      "logoUrl": "https://example.com/logo.png",
      "description": "店铺简介",
      "defaultDeliveryType": "express",
      "distributionEnabled": false,
      "status": "active"
    }
  },
  "traceId": "req_001"
}
```

### 4.3 成功判定

创建成功后生成 `leader` 和 `store`；再次调用 `GET /api/v1/me` 时 `hasLeader=true`，并返回 `leaderId`、`storeId`。

### 4.4 常见错误码

| 错误码 | 场景 |
|---|---|
| `UNAUTHORIZED` | 未登录 |
| `STORE_ALREADY_EXISTS` | 当前用户已创建店铺 |
| `VALIDATION_ERROR` | 店铺名称或物流方式不合法 |

### 4.5 测试数据准备方式

使用团长测试手机号完成模拟登录，然后调用创建店铺接口。重复创建场景使用同一 token 再次提交。

## 5. 发布普通团购

| 项目 | 内容 |
|---|---|
| 页面入口 | 个人中心页或团长主页管理入口 -> 发布团购页 |
| 前置条件 | 用户已登录，且已经创建店铺成为团长 |
| 调用顺序 | `GET /api/v1/me` -> `POST /api/v1/my/store/group-buys` -> `GET /api/v1/my/store/group-buys/{groupBuyId}` |

### 5.1 请求样例

```http
POST /api/v1/my/store/group-buys
Authorization: Bearer <leaderAccessToken>
Content-Type: application/json
```

```json
{
  "title": "山东蜜桃团购",
  "introduction": "产地直发，香甜多汁",
  "coverImageUrl": "https://example.com/cover.png",
  "deliveryType": "express",
  "shippingTime": "2026-06-30T18:00:00+08:00",
  "startTime": "2026-06-24T12:00:00+08:00",
  "endTime": "2026-07-01T12:00:00+08:00",
  "items": [
    {
      "product": {
        "name": "白玉蜜桃",
        "description": "山东蒙阴产地直发",
        "coverImageUrl": "https://example.com/product.png",
        "basePriceAmount": 2990,
        "stock": 100
      },
      "displayName": "白玉蜜桃 5 斤装",
      "groupPriceAmount": 2990,
      "groupStock": 100,
      "sortOrder": 1
    }
  ]
}
```

### 5.2 响应样例

```json
{
  "success": true,
  "data": {
    "groupBuy": {
      "id": 100,
      "storeId": 20,
      "leaderId": 10,
      "title": "山东蜜桃团购",
      "groupType": "normal",
      "status": "published"
    },
    "items": [
      {
        "id": 1001,
        "groupBuyId": 100,
        "productId": 501,
        "displayName": "白玉蜜桃 5 斤装",
        "groupPriceAmount": 2990,
        "groupStock": 100,
        "soldCount": 0,
        "sortOrder": 1
      }
    ]
  },
  "traceId": "req_001"
}
```

### 5.3 成功判定

返回 `groupBuy.status=published`，`groupBuy.groupType=normal`；每个 item 都有 `groupBuyItemId`，`groupStock >= 0`，`soldCount=0`。

### 5.4 常见错误码

| 错误码 | 场景 |
|---|---|
| `UNAUTHORIZED` | 未登录 |
| `LEADER_REQUIRED` | 当前用户不是团长 |
| `STORE_FORBIDDEN` | 复用商品不属于当前店铺 |
| `VALIDATION_ERROR` | 团购参数、商品参数或库存不合法 |

### 5.5 测试数据准备方式

先完成“模拟登录与获取当前用户”和“创建店铺”链路。发布成功后记录 `leaderId`、`storeId`、`groupBuyId`、`groupBuyItemId`、`productId`，供浏览和下单链路使用。

## 9. 模拟支付

| 项目 | 内容 |
|---|---|
| 页面入口 | 订单详情页 -> 模拟支付按钮 |
| 前置条件 | 买家已登录，订单属于当前用户，订单为 `pendingPay + unpaid` |
| 调用顺序 | `GET /api/v1/my/orders/{orderId}` -> `POST /api/v1/orders/{orderId}/simulate-pay` -> 刷新订单详情 |

### 9.1 请求样例

```http
POST /api/v1/orders/9001/simulate-pay
Authorization: Bearer <buyerAccessToken>
```

### 9.2 响应样例

```json
{
  "success": true,
  "data": {
    "id": 9001,
    "payStatus": "paid",
    "orderStatus": "paid",
    "payAmount": 2990,
    "items": [
      {
        "groupBuyItemId": 1001,
        "quantity": 1,
        "unitPriceAmount": 2990
      }
    ]
  },
  "traceId": "req_001"
}
```

### 9.3 成功判定

订单变为 `payStatus=paid`、`orderStatus=paid`；对应 `group_buy_items.group_stock` 扣减购买数量，`sold_count` 增加购买数量；会员关系被创建或更新。

### 9.4 常见错误码

| 错误码 | 场景 |
|---|---|
| `ORDER_NOT_PAYABLE` | 订单不是可支付状态 |
| `ORDER_ALREADY_PAID` | 订单已支付，重复请求不能再次扣库存 |
| `INSUFFICIENT_STOCK` | 支付时库存不足 |
| `RESOURCE_NOT_FOUND` | 订单不存在 |

### 9.5 测试数据准备方式

先创建待支付订单。重复支付场景对同一 `orderId` 连续调用两次支付接口；库存不足场景可在多个订单创建后让库存不足，再支付后一个订单。

## 10. 团长发货

| 项目 | 内容 |
|---|---|
| 页面入口 | 团长订单列表页 -> 团长订单详情页 |
| 前置条件 | 团长已登录，订单属于当前团长自己的店铺，订单为 `paid` |
| 调用顺序 | `GET /api/v1/my/store/orders` -> `GET /api/v1/my/store/orders/{orderId}` -> `POST /api/v1/my/store/orders/{orderId}/ship` |

### 10.1 请求样例

```http
POST /api/v1/my/store/orders/9001/ship
Authorization: Bearer <leaderAccessToken>
Content-Type: application/json
```

```json
{
  "deliveryType": "express",
  "logisticsCompany": "顺丰速运",
  "trackingNo": "SF123456789"
}
```

### 10.2 响应样例

```json
{
  "success": true,
  "data": {
    "order": {
      "id": 9001,
      "orderStatus": "shipped",
      "payStatus": "paid"
    },
    "shipment": {
      "id": 6001,
      "orderId": 9001,
      "deliveryType": "express",
      "logisticsCompany": "顺丰速运",
      "trackingNo": "SF123456789"
    }
  },
  "traceId": "req_001"
}
```

### 10.3 成功判定

发货成功后订单状态变为 `shipped`，创建一条发货记录；只有当前团长自己店铺的订单可发货。

### 10.4 常见错误码

| 错误码 | 场景 |
|---|---|
| `LEADER_REQUIRED` | 当前用户不是团长 |
| `STORE_FORBIDDEN` | 订单不属于当前团长自己的店铺 |
| `ORDER_NOT_SHIPPABLE` | 订单不是可发货状态 |
| `ORDER_ALREADY_SHIPPED` | 订单已发货，重复请求不能重复创建发货记录 |
| `RESOURCE_NOT_FOUND` | 订单不存在 |

### 10.5 测试数据准备方式

买家先创建订单并模拟支付，团长使用发布该团购的账号登录后进入团长订单列表。

## 11. 订阅团长

| 项目 | 内容 |
|---|---|
| 页面入口 | 团长主页；团购详情页底部操作栏 |
| 前置条件 | 买家已登录，目标团长存在 |
| 调用顺序 | `GET /api/v1/leaders/{leaderId}/homepage` 或 `GET /api/v1/group-buys/{groupBuyId}` -> `POST /api/v1/leaders/{leaderId}/subscription` -> 需要取消时 `DELETE /api/v1/leaders/{leaderId}/subscription` |

### 11.1 请求样例

```http
POST /api/v1/leaders/10/subscription
Authorization: Bearer <buyerAccessToken>
Content-Type: application/json
```

```json
{
  "source": "homepage"
}
```

### 11.2 响应样例

```json
{
  "success": true,
  "data": {
    "id": 7001,
    "userId": 1,
    "leaderId": 10,
    "storeId": 20,
    "status": "active",
    "source": "homepage",
    "subscribedAt": "2026-06-24T12:00:00+08:00"
  },
  "traceId": "req_001"
}
```

### 11.3 成功判定

订阅成功后详情页或团长主页中的 `viewer.subscribed=true`；取消订阅后 `viewer.subscribed=false`。

### 11.4 常见错误码

| 错误码 | 场景 |
|---|---|
| `UNAUTHORIZED` | 未登录 |
| `SUBSCRIPTION_EXISTS` | 已订阅该团长，且后端选择返回错误 |
| `RESOURCE_NOT_FOUND` | 团长不存在 |

### 11.5 测试数据准备方式

先使用团长账号创建店铺并发布团购，再使用买家账号登录后订阅该团长。
