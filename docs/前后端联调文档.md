## 1. 联调通用约定

### 1.1 基础信息

| 项目 | 约定 |
|---|---|
| API 前缀 | `/api/v1` |
| 数据格式 | JSON |
| 字段命名 | `camelCase` |
| 时间格式 | ISO 8601 字符串，如 `2026-06-24T12:00:00+08:00` |
| 金额格式 | 整数分，字段名以 `Amount` 结尾 |
| 登录头 | `Authorization: Bearer <accessToken>` |

### 1.2 成功响应

```json
{
  "success": true,
  "data": {},
  "traceId": "req_001"
}
```

### 1.3 错误响应

```json
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "参数校验失败",
    "details": {}
  },
  "traceId": "req_001"
}
```

### 1.4 ID 序列化约定

> **背景**：后端主键使用 MyBatis-Plus 雪花 ID（`ASSIGN_ID`），值超过 JavaScript `Number.MAX_SAFE_INTEGER`（9,007,199,254,740,991）。

所有 API 响应中的 `id` / `*Id` 字段统一以 **JSON string** 序列化，防止前端 JS 精度丢失。前端在 URL 路径、请求体、查询参数中使用 ID 时，直接传递字符串即可。

```json
{
  "id": "2071642363297644546",
  "storeId": "2071642363297644547",
  "leaderId": "2071642363297644548"
}
```

请求体中的 ID 字段同时支持字符串和数值输入（后端反序列化兼容两种格式）。

### 1.5 常用错误码

| 错误码 | 场景 |
|---|---|
| `UNAUTHORIZED` | 未登录或令牌无效 |
| `VALIDATION_ERROR` | 参数校验失败 |
| `LEADER_REQUIRED` | 当前操作需要团长身份 |
| `STORE_ALREADY_EXISTS` | 当前用户已创建店铺 |
| `STORE_FORBIDDEN` | 不能操作他人店铺 |
| `GROUP_BUY_NOT_PURCHASABLE` | 团购不可购买 |
| `GROUP_BUY_ENDED` | 团购已结束 |
| `ITEM_NOT_IN_GROUP_BUY` | 下单商品不属于该团购 |
| `ADDRESS_FORBIDDEN` | 地址不属于当前用户 |
| `INSUFFICIENT_STOCK` | 库存不足 |
| `ORDER_NOT_PAYABLE` | 订单不可支付 |
| `ORDER_ALREADY_PAID` | 订单已支付 |
| `ORDER_NOT_SHIPPABLE` | 订单不可发货 |
| `ORDER_ALREADY_SHIPPED` | 订单已发货 |
| `SUBSCRIPTION_EXISTS` | 已订阅该团长 |
| `COUPON_NOT_AVAILABLE` | 优惠券不可用（已停用/未到有效期/未达门槛） |

### 1.5 空库初始化说明

首次启动后端项目时，数据库迁移会自动执行：

| 步骤 | 说明 |
|---|---|
| 创建数据库 | 在 MySQL 中创建 `groupshop` 数据库（如 `CREATE DATABASE groupshop DEFAULT CHARSET utf8mb4`） |
| 配置连接 | 在 `application.yml` 中配置 `spring.datasource` 连接信息 |
| 启动应用 | 运行 `GroupshopApplication`，Flyway 自动执行 `classpath:db/migration/` 下的迁移脚本 |
| 迁移结果 | 空库执行 Flyway 后，获得当前核心表和已落地增强表 |
| 验证迁移 | 启动日志中应看到 Flyway 迁移成功记录，或调用 `GET /api/v1/health` 确认应用可用 |

本地开发推荐配置：
- 数据库名称：`groupshop_dev`（通过 `application-dev.yml` profile 切换）
- Flyway 迁移位置：`src/main/resources/db/migration/`
- 迁移版本管理：后续功能只在确有必要时追加迁移文件，不反复修改已发布迁移

### 22. 订单上下文轻量聊天联调（P2 部分先行）

#### 22.1 页面入口

| 页面 | 入口 |
|---|---|
| 消息页 | `/messages`，底部 Tabbar “消息”，含“聊天 / 通知”双 Tab |
| 聊天详情 | `/chats/:conversationId` |
| 买家订单列表 | `/orders` 中已支付订单进入聊天 |
| 买家订单详情 | `/orders/:id` 中非待支付状态“带订单咨询” |
| 团长订单详情 | `/leader/orders/:id` 中“订单卡片咨询 / 备货完成” |

#### 22.2 调用顺序

1. 打开会话：`POST /api/v1/my/chat-conversations/orders/{orderId}`。
2. 聊天列表：`GET /api/v1/my/chat-conversations?page=1&pageSize=20`。
3. 未读数：`GET /api/v1/my/chat-conversations/unread-count`，前端与通知未读数合并展示。
4. 消息轮询：`GET /api/v1/my/chat-conversations/{conversationId}/messages?afterMessageId={lastId}&pageSize=20`。
5. 发送文本：`POST /api/v1/my/chat-conversations/{conversationId}/messages`，`messageType=text`。
6. 发送图片：先 `POST /api/v1/my/uploads/images`，再发送 `messageType=image`，携带 `imageAssetId` 和 `imageUrl`。
7. 发送卡片：`POST /api/v1/my/chat-conversations/{conversationId}/cards`，买家和团长打开订单咨询时发送 `order_summary`，团长可额外发送 `prepare_done`。
8. 进入聊天详情后调用 `POST /api/v1/my/chat-conversations/{conversationId}/read`。

#### 22.3 成功判定

- 买家和团长打开同一订单时得到同一个会话 ID。
- 无订单关系用户访问会话返回 `CHAT_FORBIDDEN`。
- 聊天列表和聊天详情顶部统一展示当前用户的对话方：买家视角展示团长头像 / 名称，团长或商家视角展示买家头像 / 名称；店铺名仅作为会话上下文辅助展示。
- 文本、图片和卡片消息可在对方轮询中出现；图片上传后会登记 `chat_message` 引用。
- 聊天图片支持点击打开大图预览；订单卡片展示标题、摘要、订单号、状态、金额和“查看订单”入口，并按当前用户角色跳转买家或团长订单详情。
- 订单创建、支付、发货、确认收货会自动写入系统卡片，且重复业务动作不重复写入。
- 买家或团长从订单详情进入聊天时，前端先打开订单会话，再发送 `order_summary` 卡片，确保会话中能看到当前咨询的订单上下文。
- 团长发送“备货完成”后买家看到卡片，订单状态不变化。
- 页面隐藏或未登录时轮询暂停；轮询失败不弹全局 toast。

#### 22.4 常见错误码

| 错误码 | 场景 |
|---|---|
| `UNAUTHORIZED` | 未登录 |
| `CHAT_CONVERSATION_NOT_FOUND` | 会话不存在 |
| `CHAT_FORBIDDEN` | 非会话成员或无订单关系 |
| `CHAT_MESSAGE_INVALID` | 消息内容或类型非法 |
| `CHAT_ASSET_INVALID` | 图片资产不可用于聊天 |
| `CHAT_CARD_NOT_ALLOWED` | 无权发送卡片 |

### 22A. 商家工作台联调

本节覆盖当前登录团长自己的商家工作台，不是平台后台；所有商家侧接口仍以 `/api/v1/my/store/**` 从 token 识别当前团长和店铺，前端不得传 `storeId` 切换店铺。

#### 22A.1 页面入口

| 页面 | 入口 |
|---|---|
| 我的 | `/profile`；团长展示“进入商家工作台”“电脑端管理”和“团长主页”，非团长展示“创建店铺” |
| H5 商家工作台 | `/leader/dashboard` |
| Web 商家管理端 | `/merchant/dashboard`；从 `/profile` 团长经营区“电脑端管理”进入，作为桌面入口，不替换 `/leader/**` |
| Web 订单履约 | `/merchant/orders`、`/merchant/orders/:id` |
| Web 售后管理 | `/merchant/after-sales`、`/merchant/after-sales/:id` |
| Web 客服会话 | `/merchant/chats`、`/merchant/chats/:conversationId`，管理端左右分栏处理买家会话 |
| Web 商品管理 | `/merchant/products`、`/merchant/products/new`、`/merchant/products/:id/edit` |
| Web 团购运营 | `/merchant/group-buys`、`/merchant/group-buys/new`、`/merchant/group-buys/:id` |
| Web 营销和私域 | `/merchant/coupons`、`/merchant/subscribers` |
| Web 店铺资料 | `/merchant/store` |
| H5 商品、团购、订单、售后、客服 | `/leader/products`、`/leader/group-buys`、`/leader/orders`、`/leader/after-sales`、`/leader/chats` |
| 优惠券、订阅用户、店铺资料 | `/leader/coupons`、`/leader/subscribers`、`/leader/store` |

导航与返回规则：
- 页面顶部返回、浏览器返回和 App 系统返回统一优先回到真实上一页；直达或无有效历史时回到当前页面的业务兜底页。
- Web 商家管理端侧栏和 H5 Tabbar 切换保留 history，点击当前路由不重复入栈。
- 列表页进入详情后返回恢复列表滚动位置；商家管理端同时恢复横向和纵向滚动。
- 商品、团购、店铺、地址、优惠券等编辑表单存在未保存内容时，离开页面前弹确认；保存、删除或主动重置后清空未保存状态。

#### 22A.2 工作台首页调用顺序

前置条件：团长已登录并已创建店铺。

1. `GET /api/v1/my/store/workbench-summary` 获取店铺、团长、待办和状态计数。
2. Web 桌面首页额外拉取最近列表：`GET /api/v1/my/store/orders?status=paid&page=1&pageSize=5`、`GET /api/v1/my/store/after-sales?status=pending&page=1&pageSize=5`、`GET /api/v1/my/chat-conversations?role=leader&page=1&pageSize=5`。

摘要响应样例：

```json
{
  "success": true,
  "data": {
    "store": { "id": 20, "name": "王姐社区鲜果店", "logoUrl": "/uploads/images/store.jpg", "status": "active" },
    "leader": { "id": 10, "displayName": "王姐", "avatarUrl": "/uploads/images/avatar.jpg" },
    "todos": { "paidOrders": 3, "pendingAfterSales": 1, "unreadLeaderChats": 4, "publishedGroupBuys": 5 },
    "statusCounts": {
      "orders": { "paid": 3, "shipped": 8, "completed": 21, "afterSale": 1, "canceled": 2 },
      "afterSales": { "pending": 1, "approved": 2, "rejected": 1, "completed": 6 },
      "groupBuys": { "draft": 2, "published": 5, "ended": 9 }
    }
  },
  "traceId": "req_001"
}
```

成功判定：
- 工作台顶部展示当前店铺名称、团长名、店铺状态、团长主页入口和返回 H5 工作台入口。
- Web 商家管理端右侧内容区支持横向和纵向滚动，窄屏下表格、分栏详情和长表单不可被视口裁切。
- 待办区展示待发货订单、售后待处理、未读客服消息、进行中团购。
- `/merchant/dashboard` 展示最近待发货订单、最近售后、最近客服会话；主要入口跳转订单、售后、客服和团购。

#### 22A.2B Web 订单履约调用顺序

列表页：

```http
GET /api/v1/my/store/orders?status=paid&keyword=202607&page=1&pageSize=30
Authorization: Bearer <leaderAccessToken>
```

状态筛选：全部、待发货(`paid`)、已发货(`shipped`)、已完成(`completed`)、售后中(`after_sale`)、已取消(`canceled`)。`keyword` 按订单号、收货人或收货手机号模糊搜索。

详情和发货：

```http
GET /api/v1/my/store/orders/{orderId}
POST /api/v1/my/store/orders/{orderId}/ship
```

发货请求：

```json
{
  "deliveryType": "express",
  "logisticsCompany": "顺丰速运",
  "trackingNo": "SF123456"
}
```

成功判定：桌面表格展示订单号、买家/收货人、商品摘要、金额、订单状态、支付时间、发货状态；`paid` 订单可进入详情发货。

| 错误码 | 场景 |

| `UNAUTHORIZED` | 未登录 |

| `RESOURCE_NOT_FOUND` | 售后单不存在或不属于当前店铺 |
| `AFTER_SALE_STATUS_INVALID` | 当前状态不可执行该动作 |

#### 22A.4 客服工作台调用顺序

客服工作台只组织会话入口，不建设独立客服中心，不实现客服分配、撤回、评价、敏感词审核或快捷短语库。

```http
GET /api/v1/my/chat-conversations?role=leader&page=1&pageSize=20
Authorization: Bearer <leaderAccessToken>
```

成功判定：
- `/merchant/chats` 采用左右分栏：左侧展示买家头像、买家名、店铺名、最后消息、未读数和“关联订单咨询”提示；右侧展示当前会话消息流和发送区。
- 进入 `/merchant/chats` 且存在会话时，前端自动选中第一条并跳转到 `/merchant/chats/{conversationId}`。
- 点击左侧会话在管理端内切换到 `/merchant/chats/{conversationId}`；后续消息轮询、发送、已读仍按 22 章聊天联调执行。
- 右侧聊天头部展示买家头像 / 名称，店铺名仅作为上下文，不展示商家自己的头像作为会话对象。
- 空列表显示“暂无买家咨询”。

测试数据准备：
- 优先使用 seed 团长账号 `13870010001` 登录。
- 若没有售后样本，可用买家已支付/已发货订单调用 `POST /api/v1/orders/{orderId}/after-sales` 创建，再用团长账号进入 `/leader/after-sales` 处理。

#### 22A.5 Web 商品管理调用顺序

PC 商品管理和 H5 `/leader/products` 使用同一组当前店铺接口，不传 `storeId`，不允许跨店铺查询。

列表页：

```http
GET /api/v1/my/store/products?status=active&keyword=水蜜桃&page=1&pageSize=50
Authorization: Bearer <leaderAccessToken>
```

新建、详情、编辑、删除：

```http
POST /api/v1/my/store/products
GET /api/v1/my/store/products/{productId}
PATCH /api/v1/my/store/products/{productId}
DELETE /api/v1/my/store/products/{productId}
```

创建请求样例：

```json
{
  "name": "阳山水蜜桃 5 斤装",
  "description": "当季鲜桃，社区统一发货",
  "coverImageUrl": "/uploads/images/peach-cover.jpg",
  "detailImageUrls": ["/uploads/images/peach-detail-1.jpg"],
  "basePriceAmount": 3990,
  "stock": 80
}
```

成功判定：
- `/merchant/products` 以桌面表格展示商品、基础价、库存、上下架状态和更新时间；支持状态筛选与关键字搜索。
- `/merchant/products/new` 和 `/merchant/products/:id/edit` 使用桌面双栏表单，支持封面图、最多 9 张详情图、价格、库存、上下架状态、保存和删除。
- 删除失败时按后端错误提示展示，例如商品仍被团购引用时不得前端假删除。

#### 22A.6 Web 团购管理调用顺序

## 3. 手机号验证码认证与获取当前用户

| 项目 | 内容 |
|---|---|
| 页面入口 | 登录页 `/login`；注册页 `/register`；应用启动时自动恢复登录态 |
| 前置条件 | 无 |
| 调用顺序 | 登录流程：`/login` -> `POST /api/v1/auth/codes` (`scene=login`) -> `POST /api/v1/auth/login` -> 保存 `accessToken` -> `GET /api/v1/me` 刷新完整身份 |
| 注册流程 | `/register` -> `POST /api/v1/auth/codes` (`scene=register`) -> `POST /api/v1/auth/register` -> 保存 `accessToken` -> `GET /api/v1/me` 刷新完整身份 |
| 开发测试 | 登录页底部“开发测试账号”折叠区继续调用 `POST /api/v1/auth/mock-login`，只用于本地调试和 E2E |
| 启动恢复 | 应用加载时从 localStorage 读取 `accessToken` -> 若存在则调用 `GET /api/v1/me` -> 恢复 `user/leader/store`；token 无效或过期则清除本地 token 并回到未登录状态 |
| 路由拦截 | 未登录访问受保护页面时重定向到 `/login?redirect=<fullPath>`，登录成功后回到原目标页 |
| 团长身份拦截 | 已登录但非团长访问需团长身份页面（如一键开团）时重定向到 `/store/create?redirect=<fullPath>` |
| 个人主页 | `/profile/me` 展示和编辑当前用户头像、昵称；普通用户同时展示未创建店铺的开团主页空态 |

### 3.6 前端页面与交互说明

#### 登录页 (`/login`) 与注册页 (`/register`)

- `/login` 展示手机号、验证码；`/register` 展示手机号、验证码和昵称。注册不填写头像，头像进入个人主页后再维护。
- 点击“获取验证码”调用 `POST /api/v1/auth/codes`，按钮进入倒计时；演示环境在页面中展示 `devCode`。
- 登录提交调用 `POST /api/v1/auth/login`；注册提交调用 `POST /api/v1/auth/register`。
- 提供“开发测试账号”折叠区：
  - **买家测试用户**：手机号 `13800000000`、昵称 `买家用户`
  - **团长测试用户**：手机号 `13700000000`、昵称 `团长用户`
- 表单提交前进行手机号、验证码、协议勾选和注册昵称校验。
- 加载状态：主按钮显示 `loading` 并禁用重复点击。
- 错误展示：后端返回的错误码和错误文案展示在表单下方红色提示区域，使用 `getErrorMessage()` 补充中文文案。
- 认证成功后跳转：优先回到 `redirect` 查询参数指定的路径，未指定则跳转到 `/profile`。

#### 启动恢复

- 应用初始化时（`main.ts` 中 `app.mount()` 之前）执行 `authStore.restoreSession()`：
  - 从 `localStorage` 读取 `accessToken`，若不存在则标记启动完成。
  - 若存在则调用 `GET /api/v1/me` 恢复 `user`、`leader`、`store`。
  - 若 `GET /api/v1/me` 返回 `UNAUTHORIZED`，清除本地 token 并回到未登录状态。
- 启动恢复完成后 `authStore.isBootstrapped` 置为 `true`。

#### 路由守卫

- `router.beforeEach` 中按顺序执行：
  1. 设置 `document.title`。
  2. `requiresAuth`：未登录 -> 重定向到 `/login?redirect=<fullPath>`。
  3. `requiresLeader`：已登录但非团长 -> 重定向到 `/store/create?redirect=<fullPath>`。
  4. 已登录用户访问 `/login` 或 `/register`：重定向到 `redirect` 查询参数或 `/profile`。
- 受保护页面列表：`/orders`、`/checkout`、`/addresses`、`/store/create`、`/leader`。
- 公开页面（无需登录）：`/profile`、`/open-group`、`/messages`。
- 需要团长身份的页面：`/leader`。
- 注意：`/profile`（个人中心）和 `/open-group`（一键开团）已在 Batch 04+05 改为公开页面。未登录用户可访问，页面内部根据登录状态展示不同的内容或跳转 CTA。

#### 错误展示方式

- 后端错误码通过 `src/api/errors.ts` 中的 `getErrorMessage()` 映射为中文文案。
- 登录页等表单页将错误展示在表单下方红色背景提示区域。
- 非表单页可通过 Vant Toast 或 Dialog 展示错误。

- 发布前校验：商品项必须有非负价格和库存。
- 预售团购必须指定 `startTime`、`endTime`、`shippingTime`，且 `endTime > startTime && shippingTime > endTime`。

```http
GET /api/v1/my/store/group-buys/{groupBuyId}/preview
Authorization: Bearer <leaderAccessToken>
```

```http
POST /api/v1/my/store/group-buys/{groupBuyId}/copy
Authorization: Bearer <leaderAccessToken>
```

说明：
- 复制标题、介绍、封面、配置为新草稿。
- 新商品项生成新 ID，`soldCount=0`，时间字段不复制。
- 原团购不受影响。

#### 5.6.5 修改可见性

```http
PATCH /api/v1/my/store/group-buys/{groupBuyId}/permission
Authorization: Bearer <leaderAccessToken>
Content-Type: application/json

{"visibility": "hidden"}
```

说明：
- `visibility` 只支持 `public` / `hidden`。
- 隐藏团购不出现公共列表和团长主页，只能通过分享 token 访问。
- 已结束/已移除的团购不可修改。

#### 5.6.6 生成分享卡片

```http
POST /api/v1/my/store/group-buys/{groupBuyId}/share-card
Authorization: Bearer <leaderAccessToken>
```

返回 `shareToken`、`landingPath`、团购标题、封面、价格区间、店铺、团长信息等结构数据。

前端联调口径：
- 团长在 `/leader/group-buys` 或 `/leader/group-buys/{groupBuyId}` 点击“分享”时调用该接口。
- 前端使用返回的 `shareToken` 生成 Hash 路由完整链接：`{origin}{base}#/share/group-buys/{shareToken}`。
- 分享弹层展示团购封面、标题、店铺 / 团长、价格区间、履约信息、二维码和复制链接；二维码由前端基于分享链接生成。
- 第一版只支持系统分享、二维码和复制链接，不生成可保存海报图片，不接微信 JS-SDK。

#### 5.6.7 通过分享 token 访问团购详情（公开）

```http
GET /api/v1/share/group-buys/{shareToken}
```

说明：
- 无需登录即可访问（可携带 Bearer token 获取个性化数据）。
- 支持访问隐藏团购。
- token 长期有效，同一团购复用 active token。
- 前端入口为 `#/share/group-buys/{shareToken}`，复用团购详情页展示结构。
- 从分享入口进入后，立即购买、加入购物车、购物车结算、订单预览和创建订单都必须继续透传该 `shareToken`。

#### 5.6.8 隐藏团购下单

`POST /api/v1/orders/preview` 和 `POST /api/v1/orders` 请求体增加可选字段：

```json
{
  "groupBuyId": 100,
  "addressId": 1,
  "shareToken": "abc123...",
  "items": [...]
}
```

说明：
- 隐藏团购下单必须携带匹配该团购的有效 `shareToken`。
- 公开团购不需要，`shareToken` 可省略。
- token 无效或不匹配时返回 `SHARE_TOKEN_INVALID`。

| `RESOURCE_NOT_FOUND` | 团购或商品不存在 |
| `VALIDATION_ERROR` | 团购参数、商品参数或库存不合法 |
| `BUSINESS_RULE_VIOLATION` | 结束非发布状态团购，或修改已下单商品价格，或分享草稿/已结束团购 |
| `SHARE_TOKEN_INVALID` | 分享链接无效或已过期 |
| `HIDDEN_GROUP_BUY_REQUIRES_TOKEN` | 隐藏团购下单未传 token |

### 5.8 测试数据准备方式

先完成"模拟登录与获取当前用户"和"创建店铺"链路。发布成功后记录 `leaderId`、`storeId`、`groupBuyId`、`groupBuyItemId`、`productId`，供浏览和下单链路使用。
更新和结束场景复用已发布的团购 ID。订单保护场景可先通过数据库插入模拟订单数据，或等待后续 batch 完成订单链路后再验证。

3. 如果该店铺设定了会员等级规则，等级名称会根据累计成长值自动更新

## 16B. Idempotency-Key 使用约定

P1 Batch 03 为关键交易动作引入可选的 `Idempotency-Key` 请求头。

### 16B.1 适用接口

| 接口 | 请求体 |
|---|---|
| `POST /api/v1/orders` | 有 |
| `POST /api/v1/orders/{orderId}/simulate-pay` | 无 |
| `POST /api/v1/orders/{orderId}/cancel` | 无 |
| `POST /api/v1/orders/{orderId}/complete` | 无 |
| `POST /api/v1/my/store/orders/{orderId}/ship` | 有 |

### 16B.2 使用方式

在 HTTP 请求头中传入可选 `Idempotency-Key`：

```http
POST /api/v1/orders
Authorization: Bearer <accessToken>
Idempotency-Key: unique-key-from-client-001
Content-Type: application/json

{
  "addressId": "2071642363297644501",
  "cartItemIds": ["2071642363297644550"]
}
```

### 16B.3 重复提交行为

| 场景 | 行为 |
|---|---|
| 首次请求 | 正常执行业务逻辑，返回结果 |
| 同 key 重复（请求体一致） | 返回首次响应数据或首次错误 |
| 同 key 重复（请求体不一致） | 返回 `IDEMPOTENCY_KEY_MISMATCH` |
| 同 key 并发 | 返回 `IDEMPOTENCY_IN_PROCESSING`，客户端应等 1-2 秒后重试 |
| 不同用户同 key | 独立计算，互不影响 |
| 不同路径同 key | 独立计算，互不影响 |
| 未传 key | 保持旧行为，不做幂等保护 |

### 16B.4 响应样例

首次成功后的重复请求返回相同 `data`（外层 `traceId` 重新生成）：

```json
{
  "success": true,
  "data": { /* 与首次响应相同的结构 */ },
  "traceId": "req_002"
}
```

首次失败后的重复请求返回相同错误：

```json
{
  "success": false,
  "error": {
    "code": "INSUFFICIENT_STOCK",
    "message": "库存不足"
  },
  "traceId": "req_002"
}
```

请求体冲突：

```json
{
  "success": false,
  "error": {
    "code": "IDEMPOTENCY_KEY_MISMATCH",
    "message": "同一 Idempotency-Key 请求体不一致"
  },
  "traceId": "req_002"
}
```

## 16C. 优惠券管理（团长端）

P1 Batch 04 为团长提供店铺优惠券的创建、编辑、停用和查看功能。当前前端第一版只开放“店铺新人订阅券”：用户进团长主页后看到订阅领券弹层，订阅成功后仍需手动点击领取，不自动发券。

| 项目 | 内容 |
|---|---|
| 页面入口 | 我的 -> 团长经营 -> 店铺优惠券；我的店铺资料 -> 新人订阅券 |
| 前置条件 | 团长已登录，已创建店铺 |
| 调用顺序 | `GET /api/v1/my/store/coupons` -> `POST /api/v1/my/store/coupons` -> `PATCH /api/v1/my/store/coupons/{id}` -> `POST /api/v1/my/store/coupons/{id}/disable` |

### 16C.1 创建优惠券

```http
POST /api/v1/my/store/coupons
Authorization: Bearer <leaderToken>
Content-Type: application/json
```

```json
{
  "name": "新客订阅立减券",
  "couponType": "amount",
  "claimCondition": "new_subscriber",
  "amount": 1000,
  "thresholdAmount": 0,
  "totalQuantity": 100,
  "perUserLimit": 1,
  "startTime": "2026-07-01T00:00:00",
  "endTime": "2026-08-01T00:00:00"
}
```

| 字段 | 说明 |
|---|---|
| `couponType` | 仅支持 `amount`（满减券）和 `red_packet`（红包券），两者均按固定金额抵扣 |
| `claimCondition` | 领取条件；前端第一版固定传 `new_subscriber`，表示订阅店铺团长后领取 |
| `amount` | 抵扣金额（分） |
| `thresholdAmount` | 使用门槛（分），0 表示无门槛 |
| `totalQuantity` | 总库存 |
| `perUserLimit` | 每人限领张数 |

### 16C.2 编辑优惠券

```http
PATCH /api/v1/my/store/coupons/{couponId}
Authorization: Bearer <leaderToken>
Content-Type: application/json
```

```json
{
  "name": "新名称",
  "amount": 3000
}
```

支持部分更新，只传需要修改的字段。

### 16C.3 停用优惠券

```http
POST /api/v1/my/store/coupons/{couponId}/disable
Authorization: Bearer <leaderToken>
```

停用后不可新领取，已领取但未使用的券仍可在有效期内使用。

### 16C.4 常见错误码

| 错误码 | 场景 |
|---|---|
| `LEADER_REQUIRED` | 非团长用户操作 |
| `STORE_FORBIDDEN` | 跨店铺操作 |

```http
GET /api/v1/group-buys/{groupBuyId}/coupons
```

返回该团购所属店铺当前可领取的优惠券列表（仅返回有库存、未过期、已启用的券）。

### 16D.3 领取优惠券

```http
POST /api/v1/coupons/{couponId}/claim
Authorization: Bearer <buyerAccessToken>
```

```json
{
  "success": true,
  "data": {
    "id": 5001,
    "couponId": 6001,
    "couponName": "满100减20",
    "couponType": "amount",
    "amount": 2000,
```

### 16D.4 我的优惠券

```http
GET /api/v1/my/coupons?status=unused
Authorization: Bearer <buyerAccessToken>
```

| 参数 | 说明 |
|---|---|
| `status` | 可选：`unused` / `locked` / `used` / `expired`；省略返回全部 |

返回结果懒更新过期券（查询时自动将已过期的 `unused` 券标记为 `expired`）。

### 16D.5 常见错误码

| 错误码 | 场景 |
|---|---|
| `COUPON_NOT_AVAILABLE` | 优惠券已停用、不在有效期内、未达到使用门槛或未订阅新人券所属团长 |
| `COUPON_ALREADY_CLAIMED` | 已达到每人限领数量 |
| `COUPON_OUT_OF_STOCK` | 优惠券已被领完 |
| `UNAUTHORIZED` | 未登录 |

### 16D.6 下单用券流程

1. 预览时省略 `userCouponId`，响应中带 `availableCoupons` / `unavailableCoupons` 供前端展示
2. 预览时指定 `userCouponId`，响应带 `selectedCoupon` 和抵扣后金额
3. 创建订单时传 `userCouponId`，券状态变为 `locked`
4. 模拟支付成功后券状态变为 `used`
5. 取消待支付订单后券状态恢复为 `unused`

## 17. 售后退款

P1 Batch 05 新增。买家对已支付订单申请售后，团长审核/拒绝/完成模拟退款。

| 项目 | 内容 |
|---|---|
| 页面入口 | 订单详情页 -> 申请售后；团长后台 -> 售后管理 |
| 前置条件 | 订单已支付（payStatus=paid，orderStatus 为 paid/shipped/completed 之一） |
| 调用顺序 | 买家申请 -> 团长审核通过 -> 团长完成模拟退款；或团长拒绝 |

### 17.1 申请售后

```http
POST /api/v1/orders/{orderId}/after-sales
Authorization: Bearer <buyerAccessToken>
Content-Type: application/json

{
  "type": "refund",
  "reason": "商品质量问题"
}
```

**业务规则：**
- 仅订单本人可申请。
- 仅 `payStatus=paid` 且 `orderStatus in paid/shipped/completed` 的订单可申请。
- 同一订单不能存在 `pending` 或 `approved` 的售后单。
- 创建后订单状态变为 `afterSale`（DB 为 `after_sale`）。

**响应样例：**

```json
{
  "success": true,
  "data": {
    "id": 10001,
    "orderId": 9001,
    "orderNo": "202606240001",
    "type": "refund",
    "reason": "商品质量问题",
    "status": "pending",
    "amount": 2990,
    "originalOrderStatus": "paid",
    "orderStatus": "afterSale",
    "payStatus": "paid",
    "createdAt": "2026-07-02T12:00:00+08:00"
  },
  "traceId": "req_001"
}
```

### 17.2 买家查看售后

**我的售后列表：**

```http
GET /api/v1/my/after-sales?page=1&pageSize=20
Authorization: Bearer <buyerAccessToken>
```

**我的售后详情：**

```http
GET /api/v1/my/after-sales/{afterSaleId}
Authorization: Bearer <buyerAccessToken>
```

### 17.3 团长处理售后

**店铺售后列表：**

```http
GET /api/v1/my/store/after-sales?page=1&pageSize=20
Authorization: Bearer <leaderAccessToken>
```

**店铺售后详情：**

```http
GET /api/v1/my/store/after-sales/{afterSaleId}
Authorization: Bearer <leaderAccessToken>
```

**审核通过（pending → approved）：**

```http
POST /api/v1/my/store/after-sales/{afterSaleId}/approve
Authorization: Bearer <leaderAccessToken>
```

成功后 `status=approved`，订单保持 `afterSale`。

**拒绝售后（pending → rejected）：**

```http
POST /api/v1/my/store/after-sales/{afterSaleId}/reject
Authorization: Bearer <leaderAccessToken>
Content-Type: application/json

{
  "rejectReason": "商品完好，不予退款"
}
```

成功后 `status=rejected`，订单恢复为申请时的状态。

**完成退款（approved → completed）：**

```http
POST /api/v1/my/store/after-sales/{afterSaleId}/complete-refund
Authorization: Bearer <leaderAccessToken>
```

**退款完成后：**
- 售后 `status=completed`
- 订单 `orderStatus=refunded`，`payStatus=refunded`
- 库存返还：`group_stock += quantity`，`sold_count -= quantity`（最低 0）
- 会员冲正：`total_order_amount -= payAmount`，`growth_value -= payAmount`，`total_orders -= 1`（最低 0），重算等级
- 优惠券不退回
- 重复调用幂等，不重复执行副作用

### 17.4 常见错误码

| 错误码 | HTTP | 场景 |
|---|---|---|
| `AFTER_SALE_NOT_APPLICABLE` | 422 | 订单未支付/状态不可售后 |
| `AFTER_SALE_IN_PROGRESS` | 409 | 已有进行中的售后单 |
| `AFTER_SALE_NOT_APPROVABLE` | 422 | 售后单不可审核通过 |

```ts
server: {
  proxy: {
    '/api/v1': {
      target: process.env.VITE_API_BASE_URL || 'http://localhost:8080',
      changeOrigin: true,
    },
  },
}
```

- 所有前端请求以 `/api/v1` 开头，不硬编码后端域名。

### 17. Batch 06 店铺定位与距离展示（后端 + 联调）

> 本批为后端 batch，前端 Geolocation 接入另开前端 batch。

#### 17.1 背景

- 为店铺保存 GPS 经纬度。
- 公共团购列表/详情在收到用户坐标时返回距离。
- 距离仅用于展示、可选筛选和排序，不影响下单、支付、发货、售后、优惠券或会员成长。

#### 17.2 涉及后端接口

| 接口 | 变更 |
|---|---|
| `POST /api/v1/stores` | 请求/响应中增加可选 `latitude`、`longitude` |
| `GET /api/v1/my/store` | 响应中增加 `latitude`、`longitude` |
| `PATCH /api/v1/my/store` | 请求/响应中增加可选 `latitude`、`longitude` |
| `GET /api/v1/group-buys` | 增加 `latitude`、`longitude`、`maxDistanceMeters`、`sort` 查询参数；响应中 StoreLite 增加 `latitude`、`longitude`、`distanceMeters`、`distanceText` |
| `GET /api/v1/group-buys/{groupBuyId}` | 增加 `latitude`、`longitude` 查询参数；响应中 StoreDetail 增加 `latitude`、`longitude`、`distanceMeters`、`distanceText` |

#### 17.3 联调场景

##### 场景 A：店铺创建/更新时设置坐标

1. 团长在创建店铺时请求体传入 `latitude: 31.2304, longitude: 121.4737`。
2. 验证响应中 `store.latitude` 和 `store.longitude` 返回正确坐标。
3. 团长通过 `PATCH /api/v1/my/store` 修改坐标，仅传入 `latitude` 或仅传 `longitude` 应返回 `VALIDATION_ERROR`。
4. 前端 `/leader/store` 编辑店铺资料时可通过浏览器 Geolocation 获取当前位置，并在保存时随 `PATCH /api/v1/my/store` 一起提交 `latitude`、`longitude`；展示态显示位置是否已添加，不展示原始 Logo URL。

##### 场景 B：公共列表展示距离

1. 用户（买家）请求 `GET /api/v1/group-buys?latitude=30.2741&longitude=120.1551`。
2. 验证响应中每个团购的 `store` 对象包含：
   - `store.latitude` / `store.longitude`（店铺坐标）
   - `store.distanceMeters`（整数米数，如 `164874`）
   - `store.distanceText`（人类可读文本，如 `"164.9km"`）
3. 不传坐标时，上述距离字段为 `null`。
4. 传入非法坐标（如 `latitude=100`）返回 `VALIDATION_ERROR`。

##### 场景 C：距离筛选

1. 请求 `GET /api/v1/group-buys?latitude=30.2741&longitude=120.1551&maxDistanceMeters=5000`。
2. 验证只返回距离 ≤ 5km 的团购。

##### 场景 D：距离排序

1. 请求 `GET /api/v1/group-buys?latitude=30.2741&longitude=120.1551&sort=distance`。
2. 验证结果按距离升序排列。

##### 场景 E：团购详情展示距离

1. 请求 `GET /api/v1/group-buys/{id}?latitude=30.2741&longitude=120.1551`。
2. 验证 `data.store` 返回店铺坐标和距离。
3. 不传坐标时距离字段为 `null`。

#### 17.4 距离计算规则

- 使用 Haversine 公式，基于 WGS84 坐标系。
- `distanceMeters`：四舍五入后的整数米数。
- `distanceText`：
  - 小于 1000m 时显示如 `"800m"`
  - 大于等于 1000m 时显示一位小数并去掉 `.0`，如 `"1km"`、`"1.2km"`

#### 17.5 限制说明

- 当前不接地图 SDK，不做逆地理编码，不存用户实时 GPS。
- 店铺坐标不参与配送范围、下单资格或履约承诺。
- 首页 H5 已接入浏览器 Geolocation：用户点击“开启定位”或“附近”后获取坐标，仅作为本次列表请求参数使用。

### 19. 文件上传 v1（公开图片 + 本地存储）

#### 19.1 概览

本批新增公开图片上传能力，用来生产现有图片 URL 字段。P1 Batch 08 起，上传图片同时登记为上传资产，业务对象保存成功后登记引用，后续清理任务只删除超期且无引用的本地图片。

| 使用位置 | 写入字段 |
|---|---|
| `/leader/store` 店铺资料 | `logoUrl` |
| 商品新建 / 编辑表单 | `coverImageUrl` |
| `/leader/group-buys/publish` 发布团购 | `coverImageUrl` |

#### 19.2 新增 API

```http
POST /api/v1/my/uploads/images
Authorization: Bearer <accessToken>
Content-Type: multipart/form-data
```

请求字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| file | binary | 图片文件，支持 jpg / png / webp，最大 5MB |

响应样例：

```json
{
  "success": true,
  "data": {
    "url": "/uploads/images/2026/07/uuid.png",
    "objectKey": "images/2026/07/uuid.png",
    "originalFilename": "cover.png",
    "contentType": "image/png",
    "size": 123456,
    "assetId": "2071642363297644546",
    "status": "temporary"
  },
  "traceId": "req_001"
}
```

#### 19.3 前端调用顺序

1. 用户在店铺 Logo、商品封面或团购封面控件选择本地图片。
2. 前端用 `FormData` 调用 `POST /api/v1/my/uploads/images`。
3. 上传成功后，将响应 `data.url` 回填到对应图片字段。
4. 用户点击保存 / 发布时，原业务接口继续提交 `logoUrl` 或 `coverImageUrl`。
5. 后端在业务保存成功后登记图片引用；如果用户上传后未保存，图片保持临时状态，超过保护期且无引用时可被清理。

#### 19.4 成功判定

- 上传接口返回 `success=true`，`data.url` 为可公开访问的 `http/https` URL。
- `GET /uploads/{objectKey}` 可访问刚上传的图片。
- 店铺、商品、团购保存后，页面展示使用上传返回的 URL。
- 已保存业务对象引用的图片不会被上传清理任务删除。

#### 19.5 常见错误码

| 错误码 | 场景 |
|---|---|
| `UNAUTHORIZED` | 未登录上传 |
| `VALIDATION_ERROR` | 空文件、超过 5MB、非 jpg/png/webp、文件内容与类型不匹配 |
| `INTERNAL_ERROR` | 本地目录不可写或图片保存失败 |

#### 19.6 测试数据准备方式

- 使用 `POST /api/v1/auth/mock-login` 获取登录 token。

- 上传成功后，将返回 URL 用于 `PATCH /api/v1/my/store`、`POST /api/v1/my/store/products` 或 `POST /api/v1/my/store/group-buys`。

4. 点击消息时调用 `POST /api/v1/my/notifications/{notificationId}/read`，然后按 `actionUrl` 跳转。
5. 已登录团长在应用内额外每 30 秒调用 `GET /api/v1/my/notifications?page=1&pageSize=5&unreadOnly=true&type=order_paid` 检查新支付订单；命中未展示过的未读通知后弹出“新订单来了”处理弹窗。
6. 弹窗点击“查看订单”时调用 `POST /api/v1/my/notifications/{notificationId}/read`，再跳转 `actionUrl`；点击“稍后处理”只关闭弹窗，不标记已读。
7. 页面隐藏、用户未登录或 token 失效时暂停轮询。

#### 20.5 事件验证链路

| 业务动作 | 预期通知 |
|---|---|
| 买家调用 `POST /api/v1/orders/{orderId}/simulate-pay` 成功 | 买家和团长都收到 `order_paid` |
| 团长调用 `POST /api/v1/my/store/orders/{orderId}/ship` 成功 | 买家收到 `order_shipped` |
| 买家调用 `POST /api/v1/orders/{orderId}/complete` 成功 | 团长收到 `order_completed` |
| 用户订阅团长成功 | 团长收到 `subscription_created` |
| 团长发布团购成功 | 该团长活跃订阅用户收到 `group_buy_published` |

#### 20.6 成功判定

- 未读数接口返回当前用户未读通知数量。
- 消息页只展示当前用户自己的通知。
- 标记已读后，该通知 `readStatus=read`，未读数减少。
- 重复支付、重复发货、重复确认收货不会生成重复通知。
- `/messages` 不展示公众号推送入口，也不请求真实公众号接口。
- 团长收到 `order_paid` 未读通知时看到居中的新订单弹窗；同一条通知在当前会话内关闭后不重复弹出，但仍保留在消息页未读列表。

### 21. 首页订阅、订单与确认订单前端体验修复

#### 21.1 页面入口

| 页面 | 入口 |
|---|---|
| 首页 | `/`，底部 Tabbar “首页” |
| 确认订单 | `/checkout`，由团购详情或购物车进入 |
| 地址选择 | `/addresses?from=checkout`，确认订单页点击收货地址进入 |
| 订单列表 | `/orders`，底部 Tabbar “订单” |
| 订单详情 | `/orders/:id`，订单列表点击或创建订单成功跳转 |

#### 21.2 前端调用顺序

1. 首页进入后调用 `GET /api/v1/group-buys` 拉取公开团购列表；用户点击“开启定位”或“附近”后通过浏览器 Geolocation 获取坐标，后续列表请求携带 `latitude`、`longitude` 和 `sort=distance`。
2. 首页“附近”使用后端距离筛选：已定位时调用 `GET /api/v1/group-buys?latitude={lat}&longitude={lng}&maxDistanceMeters=5000&sort=distance`；未定位或拒绝授权时展示定位提示。`生鲜水果 / 节令特产 / 适合复购` 仍基于当前已加载列表做关键词筛选。
3. 已登录用户进入首页时额外调用 `GET /api/v1/my/subscriptions`，用返回的 `leaderId` 集合初始化卡片订阅态；未登录不调用。
4. 首页点击订阅：未登录跳转 `/login?redirect=/`；已登录时根据当前态调用 `POST /api/v1/leaders/{leaderId}/subscription` 或 `DELETE /api/v1/leaders/{leaderId}/subscription`。
5. 确认订单进入后先拉地址，再调用 `POST /api/v1/orders/preview`；请求体携带 `addressId`、商品项或 `cartItemIds`，以及可选 `userCouponId`。
6. 确认订单点击地址进入 `/addresses?from=checkout`；选择地址后写入 `checkoutStore.selectedAddressId` 并返回，checkout 重新 preview。
7. 确认订单点击可用优惠券后把 `availableCoupons[].userCouponId` 写入 `checkoutStore.userCouponId`，再次调用 preview；再次点击已选优惠券取消使用。
8. 创建订单调用 `POST /api/v1/orders`，请求体同步携带 `userCouponId`；成功后清空 checkout store 并跳转 `/orders/:id`。

#### 21.3 展示与成功判定

- 首页筛选文案随筛选项变化；附近已定位时以 `maxDistanceMeters=5000` 后端筛选为准，团购卡片仅在存在 `store.distanceText` 时显示“距你 xx”，未定位用户不展示店铺是否已配置定位。
- 团长主页店铺信息区集中展示店铺位置；已定位用户请求团长主页时携带 `latitude/longitude` 并展示 `store.distanceText`，下方团购卡片不再重复展示店铺名称和距离。
- 首页订阅按钮展示 `订阅 / 已订阅 / 处理中`，接口成功后立即更新当前 leader 的按钮状态。
- 确认订单页不提供数量修改控件，只展示单价、数量、小计、商品金额、优惠和应付金额。
- preview 响应中的 `availableCoupons` 可点击，`unavailableCoupons` 只展示不可用原因；`selectedCoupon` 与金额区一致。`availableCoupons[].id` 为券模板 ID，前端下单参数必须使用 `userCouponId`。
- 订单列表不展示公众号关注条；订单卡片展示订单状态、商品快照、订单号、优惠和实付金额。
- 订单详情按状态摘要、履约进度、收货地址、商品明细、金额明细、订单信息展示；有 `couponName` 或 `discountAmount > 0` 时展示优惠券快照。

### 23. Docker Compose 单机部署联调

本节覆盖单机线上部署口径：MySQL、后端和前端 Nginx 均由根目录 `docker-compose.yml` 编排。该部署不是平台后台，也不接入真实微信支付、公众号通知或外部地图 SDK。

#### 23.1 服务入口

| 服务 | 入口 |
|---|---|
| 前端 H5 / 商家管理端 | `http://<host>/`，由 Nginx 托管 `frontend/dist` |
| 后端 API | `http://<host>/api/v1/**`，由 Nginx 反代到 `backend:8080` |
| 上传资源 | `http://<host>/uploads/**`，由 Nginx 反代到后端静态资源映射 |
| 健康检查 | `GET http://<host>/api/v1/health` |
| MySQL | 仅 Docker 内网访问，服务名 `mysql:3306` |

#### 23.2 前置条件

1. 服务器已安装 Docker 和 Docker Compose。
2. 从 `.env.deploy.example` 复制 `.env.deploy`，并替换 `MYSQL_PASSWORD`、`MYSQL_ROOT_PASSWORD`。
3. 服务器 80 端口可被访问；HTTPS 由外层负载均衡、宿主机反代或后续证书服务处理。

#### 23.3 启动与调用顺序

```bash
docker compose --env-file .env.deploy config
docker compose --env-file .env.deploy up -d --build
docker compose --env-file .env.deploy ps
```

启动顺序：

1. `mysql` 初始化数据库和持久化卷 `mysql_data`。
2. `backend` 等待 MySQL healthcheck 通过后启动，Flyway 自动执行迁移，上传目录挂载到 `backend_uploads`。
3. `frontend` 等待后端健康检查通过后启动 Nginx，托管 H5 并反代 `/api/v1`、`/uploads`。

#### 23.4 请求样例

```http
GET /api/v1/health HTTP/1.1
Host: <host>
```

响应样例：

```json
{
  "success": true,
  "data": {
    "status": "UP",
    "timestamp": "2026-07-08T10:00:00"
  },
  "traceId": "req_001"
}
```

#### 23.5 成功判定

- `docker compose ps` 中 `mysql`、`backend`、`frontend` 均为 running/healthy。
- `GET /api/v1/health` 返回 `status=UP`。
- 访问 `/` 可打开 H5 首页，刷新 Hash 路由页面不返回 404。
- 前端登录、首页列表、下单、商家工作台等请求均走同源 `/api/v1`。
- 上传图片返回 `/uploads/**`，页面可正常回显图片。

#### 23.6 常见问题

| 问题 | 处理方式 |
|---|---|
| `MYSQL_PASSWORD is required` | 检查是否传入 `--env-file .env.deploy`，以及 `.env.deploy` 是否配置密码 |
| 后端启动失败 | 查看 `docker compose --env-file .env.deploy logs -f backend`，重点检查 MySQL 连接和 Flyway 迁移 |
| 上传图片无法访问 | 检查 `backend_uploads` 卷、后端 `/app/uploads` 挂载和 Nginx `/uploads/` 反代 |
| 80 端口占用 | 停止宿主机已有 Web 服务，或临时修改 compose 的端口映射 |

```bash
docker compose --env-file .env.deploy exec -T mysql \

  < backend/scripts/dev-seed.sql
```
