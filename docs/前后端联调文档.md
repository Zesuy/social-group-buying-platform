## 1. 联调通用约定

### 1.1 基础信息

| 项目 | 约定 |
|---|---|
| API 前缀 | `/api/v1` |
| 数据格式 | JSON |
| 字段命名 | `camelCase` |
| 时间格式 | ISO 8601 字符串，如 `2026-06-24T12:00:00+08:00` |
| 金额格式 | 整数分，字段名以 `Amount` 结尾 |
| 登录头 | `Authorization: Bearer <accessToken>` |

### 1.2 成功响应

```json
{
  "success": true,
  "data": {},
  "traceId": "req_001"
}
```

### 1.3 错误响应

```json
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "参数校验失败",
    "details": {}
  },
  "traceId": "req_001"
}
```

### 1.4 ID 序列化约定

> **背景**：后端主键使用 MyBatis-Plus 雪花 ID（`ASSIGN_ID`），值超过 JavaScript `Number.MAX_SAFE_INTEGER`（9,007,199,254,740,991）。

所有 API 响应中的 `id` / `*Id` 字段统一以 **JSON string** 序列化，防止前端 JS 精度丢失。前端在 URL 路径、请求体、查询参数中使用 ID 时，直接传递字符串即可。

```json
{
  "id": "2071642363297644546",
  "storeId": "2071642363297644547",
  "leaderId": "2071642363297644548"
}
```

请求体中的 ID 字段同时支持字符串和数值输入（后端反序列化兼容两种格式）。

### 1.5 常用错误码

| 错误码 | 场景 |
|---|---|
| `UNAUTHORIZED` | 未登录或令牌无效 |
| `VALIDATION_ERROR` | 参数校验失败 |
| `LEADER_REQUIRED` | 当前操作需要团长身份 |
| `STORE_ALREADY_EXISTS` | 当前用户已创建店铺 |
| `STORE_FORBIDDEN` | 不能操作他人店铺 |
| `GROUP_BUY_NOT_PURCHASABLE` | 团购不可购买 |
| `GROUP_BUY_ENDED` | 团购已结束 |
| `ITEM_NOT_IN_GROUP_BUY` | 下单商品不属于该团购 |
| `ADDRESS_FORBIDDEN` | 地址不属于当前用户 |
| `INSUFFICIENT_STOCK` | 库存不足 |
| `ORDER_NOT_PAYABLE` | 订单不可支付 |
| `ORDER_ALREADY_PAID` | 订单已支付 |
| `ORDER_NOT_SHIPPABLE` | 订单不可发货 |
| `ORDER_ALREADY_SHIPPED` | 订单已发货 |
| `SUBSCRIPTION_EXISTS` | 已订阅该团长 |
| `COUPON_NOT_AVAILABLE` | 优惠券不可用（已停用/未到有效期/未达门槛） |
| `COUPON_ALREADY_CLAIMED` | 已领取过该优惠券，达到每人限领 |
| `COUPON_OUT_OF_STOCK` | 优惠券已被领完 |
| `AFTER_SALE_NOT_APPLICABLE` | 当前订单不可申请售后 |
| `AFTER_SALE_IN_PROGRESS` | 当前订单已有进行中的售后单 |
| `AFTER_SALE_NOT_APPROVABLE` | 当前售后单不可审核通过 |
| `AFTER_SALE_NOT_REJECTABLE` | 当前售后单不可拒绝 |
| `AFTER_SALE_NOT_REFUNDABLE` | 当前售后单不可退款 |

### 1.5 空库初始化说明

首次启动后端项目时，数据库迁移会自动执行：

| 步骤 | 说明 |
|---|---|
| 创建数据库 | 在 MySQL 中创建 `groupshop` 数据库（如 `CREATE DATABASE groupshop DEFAULT CHARSET utf8mb4`） |
| 配置连接 | 在 `application.yml` 中配置 `spring.datasource` 连接信息 |
| 启动应用 | 运行 `GroupshopApplication`，Flyway 自动执行 `classpath:db/migration/` 下的迁移脚本 |
| 迁移结果 | 空库执行 Flyway 后，一次性获得全部 12 张 MVP 核心表 |
| 验证迁移 | 启动日志中应看到 Flyway 迁移成功记录，或调用 `GET /api/v1/health` 确认应用可用 |

本地开发推荐配置：
- 数据库名称：`groupshop_dev`（通过 `application-dev.yml` profile 切换）
- Flyway 迁移位置：`src/main/resources/db/migration/`
- 迁移版本管理：后续 batch 只在确有必要时追加迁移文件，不反复修改 V1

#### H5 通用约定

1. **Safe Area 适配**：页面已包含 `viewport-fit=cover` meta 和 CSS `env(safe-area-inset-*)` 变量，底部 TabBar 和内容面已自动适配刘海屏和小白条。
2. **金额展示**：后端金额以"整数分"传输。前端统一使用 `formatAmount(amount: number | null | undefined)` 转换为 `¥xx.xx` 格式展示。**严禁将金额作为浮点数直接使用或传递回后端。**
3. **非 MVP 灰态入口**：真实微信支付、购物车、优惠券/红包、售后退款、帮卖分销、积分商城、公众号推送、平台后台等非 MVP 功能在前端均保持禁用/灰态，不可跳转。可使用 `isFeatureDisabled(feature)` 工具函数判断。
4. **状态文案**：店铺、团购、订单、支付、配送等状态码统一通过 `src/utils/status.ts` 中的映射函数转换为中文展示，未知状态回退为"未知状态"或原值。
5. **API 前缀**：所有接口请求以 `/api/v1` 开头，开发时由 Vite proxy 转发到后端，无需单独配置跨域。
6. **Design Foundation（Frontend Batch 05.8）**：移动端设计基线统一为"温暖奶油背景 + 绿色零售 CTA + 移动 marketplace 信息流"。CSS 设计变量集中在 `variables.css`、`global.css`、`vant-theme.css` 中。
   - 主 CTA 使用 50px 胶囊按钮（`--button-capsule-height: 50px`，`--button-capsule-radius: 50px`），触控区域不小于 44px。
   - 首页团购卡片采用 demo 信息流商品卡设计，突出团长信任信息、标题、价格、三图区域和主 CTA。
   - 底部 Tab、FixedActionBar、浮动入口均已处理 safe area（`env(safe-area-inset-bottom)`）。
   - 非 MVP 浮动入口（购物车）仅展示灰态图标 + toast，不调用真实接口。
   - 金额展示保持整数分传输入口，不引入浮点业务值。

## 2. 测试数据总口径

MVP 联调建议至少准备以下数据：

| 数据 | 准备方式 |
|---|---|
| 买家用户 | 调用 `POST /api/v1/auth/mock-login` 创建或获取 |
| 团长用户 | 使用另一个手机号调用 `POST /api/v1/auth/mock-login` |
| 店铺 | 团长用户调用 `POST /api/v1/stores` 创建 |
| 团购 | 团长用户调用 `POST /api/v1/my/store/group-buys` 发布普通团购 |
| 地址 | 买家用户调用 `POST /api/v1/my/addresses` 创建 |
| 订单 | 买家用户调用 `POST /api/v1/orders` 创建 |
| 已支付订单 | 买家用户调用 `POST /api/v1/orders/{orderId}/simulate-pay` |
| 已发货订单 | 团长用户调用 `POST /api/v1/my/store/orders/{orderId}/ship` |
| 会员关系 | 买家对某店铺订单模拟支付成功后自动创建或更新 |

说明：同一条联调链路中的 ID，如 `leaderId`、`storeId`、`groupBuyId`、`groupBuyItemId`、`addressId`、`orderId`，均以前一步接口响应为准，不写死。

## 3. 模拟登录与获取当前用户

| 项目 | 内容 |
|---|---|
| 页面入口 | 登录页 `/login`；应用启动时自动恢复登录态 |
| 前置条件 | 无 |
| 调用顺序 | 登录流程：页面入口 -> 填充表单 -> `POST /api/v1/auth/mock-login` -> 保存 `accessToken` -> `GET /api/v1/me` 刷新完整身份 |
| 启动恢复 | 应用加载时从 localStorage 读取 `accessToken` -> 若存在则调用 `GET /api/v1/me` -> 恢复 `user/leader/store`；token 无效或过期则清除本地 token 并回到未登录状态 |
| 路由拦截 | 未登录访问受保护页面时重定向到 `/login?redirect=<fullPath>`，登录成功后回到原目标页 |
| 团长身份拦截 | 已登录但非团长访问需团长身份页面（如一键开团）时重定向到 `/store/create?redirect=<fullPath>` |

### 3.1 请求样例

```http
POST /api/v1/auth/mock-login
Content-Type: application/json
```

```json
{
  "nickname": "买家用户",
  "avatarUrl": "https://example.com/avatar.png",
  "phone": "13800000000"
}
```

```http
GET /api/v1/me
Authorization: Bearer mock_access_token
```

### 3.3 成功判定

| 项目 | 判定 |
|---|---|
| 登录成功 | 返回 `accessToken` 和 `user` |
| 恢复登录态 | `GET /api/v1/me` 返回当前用户、团长摘要和店铺摘要 |
| 未创建店铺用户 | `hasLeader=false`，`leaderId=null`，`storeId=null` |

### 3.4 常见错误码

| 错误码 | 场景 |
|---|---|
| `VALIDATION_ERROR` | 昵称、手机号等参数不合法 |
| `UNAUTHORIZED` | `GET /api/v1/me` 未携带或携带无效 token |

### 3.5 测试数据准备方式

直接调用模拟登录接口。需要买家和团长两个身份时，使用不同手机号分别登录。

### 3.6 前端页面与交互说明

#### 登录页 (`/login`)

- 使用 Vant Form + Field 组件展示手机号、昵称、头像 URL 三个输入字段。
- 提供两个快捷填充按钮，方便测试：
  - **买家测试用户**：填充手机号 `13800000000`、昵称 `买家用户`
  - **团长测试用户**：填充手机号 `13700000000`、昵称 `团长用户`
- 表单提交前进行手机号格式校验（`/^1\d{10}$/`）和必填校验。
- 加载状态：登录按钮显示 `loading` 并禁用重复点击。
- 错误展示：后端返回的错误码和错误文案展示在表单下方红色提示区域，使用 `getErrorMessage()` 补充中文文案。
- 登录成功后跳转：优先回到 `redirect` 查询参数指定的路径，未指定则跳转到 `/profile`。

#### 启动恢复

- 应用初始化时（`main.ts` 中 `app.mount()` 之前）执行 `authStore.restoreSession()`：
  - 从 `localStorage` 读取 `accessToken`，若不存在则标记启动完成。
  - 若存在则调用 `GET /api/v1/me` 恢复 `user`、`leader`、`store`。
  - 若 `GET /api/v1/me` 返回 `UNAUTHORIZED`，清除本地 token 并回到未登录状态。
- 启动恢复完成后 `authStore.isBootstrapped` 置为 `true`。

#### 路由守卫

- `router.beforeEach` 中按顺序执行：
  1. 设置 `document.title`。
  2. `requiresAuth`：未登录 -> 重定向到 `/login?redirect=<fullPath>`。
  3. `requiresLeader`：已登录但非团长 -> 重定向到 `/store/create?redirect=<fullPath>`。
  4. 已登录用户访问 `/login`：重定向到 `redirect` 查询参数或 `/profile`。
- 受保护页面列表：`/orders`、`/checkout`、`/addresses`、`/store/create`、`/leader`。
- 公开页面（无需登录）：`/profile`、`/open-group`、`/messages`。
- 需要团长身份的页面：`/leader`。
- 注意：`/profile`（个人中心）和 `/open-group`（一键开团）已在 Batch 04+05 改为公开页面。未登录用户可访问，页面内部根据登录状态展示不同的内容或跳转 CTA。

#### 错误展示方式

- 后端错误码通过 `src/api/errors.ts` 中的 `getErrorMessage()` 映射为中文文案。
- 登录页等表单页将错误展示在表单下方红色背景提示区域。
- 非表单页可通过 Vant Toast 或 Dialog 展示错误。

## 4. 创建店铺

| 项目 | 内容 |
|---|---|
| 页面入口 | 个人中心页 -> 创建店铺页 |
| 前置条件 | 用户已登录，且当前用户未创建店铺 |
| 调用顺序 | `GET /api/v1/me` 判断是否已有店铺 -> `POST /api/v1/stores` -> `GET /api/v1/me` 刷新身份 |

### 4.1 请求样例

```http
POST /api/v1/stores
Authorization: Bearer <accessToken>
Content-Type: application/json
```

```json
{
  "name": "某某的小店",
  "logoUrl": "https://example.com/logo.png",
  "description": "店铺简介",
  "defaultDeliveryType": "express"
}
```

### 4.2 响应样例

```json
{
  "success": true,
  "data": {
    "leader": {
      "id": 10,
      "displayName": "某某的小店",
      "avatarUrl": "https://example.com/logo.png"
    },
    "store": {
      "id": 20,
      "leaderId": 10,
      "name": "某某的小店",
      "logoUrl": "https://example.com/logo.png",
      "description": "店铺简介",
      "defaultDeliveryType": "express",
      "distributionEnabled": false,
      "status": "active"
    }
  },
  "traceId": "req_001"
}
```

### 4.3 成功判定

创建成功后生成 `leader` 和 `store`；再次调用 `GET /api/v1/me` 时 `user.hasLeader=true`，并返回 `leaderId`、`storeId`。

### 4.4 获取当前用户店铺

创建店铺后，可通过 `GET /api/v1/my/store` 获取当前用户自己的店铺信息。

#### 请求样例

```http
GET /api/v1/my/store
Authorization: Bearer <accessToken>
```

#### 响应样例（已有店铺）

```json
{
  "success": true,
  "data": {
    "leader": {
      "id": 10,
      "displayName": "某某的小店",
      "avatarUrl": "https://example.com/logo.png"
    },
    "store": {
      "id": 20,
      "leaderId": 10,
      "name": "某某的小店",
      "logoUrl": "https://example.com/logo.png",
      "description": "店铺简介",
      "defaultDeliveryType": "express",
      "distributionEnabled": false,
      "status": "active"
    }
  },
  "traceId": "req_001"
}
```

#### 响应样例（未创建店铺）

```json
{
  "success": true,
  "traceId": "req_001"
}
```

说明：未创建店铺时，`GET /api/v1/my/store` 返回 200 成功，但省略 `data` 字段。前端应通过 `response.data` 是否存在判断。已有店铺时同时返回团长身份(`leader`)和店铺(`store`)信息。

|---|---|

| `RESOURCE_NOT_FOUND` | 订单不存在 |

### 9.5 测试数据准备方式

### 9.6 Batch 04 支付增强说明

使用优惠券的订单支付成功后：
1. 用户券从 `locked` 变为 `used`，不可重复使用；重复支付不会导致券被核销两次（幂等）
2. 会员成长值按 `payAmount`（抵扣后）累计，而非 `totalAmount`
3. 如果该店铺设定了会员等级规则，等级名称会根据累计成长值自动更新

## 16B. Idempotency-Key 使用约定

P1 Batch 03 为关键交易动作引入可选的 `Idempotency-Key` 请求头。

### 16B.1 适用接口

| 接口 | 请求体 |
|---|---|
| `POST /api/v1/orders` | 有 |
| `POST /api/v1/orders/{orderId}/simulate-pay` | 无 |
| `POST /api/v1/orders/{orderId}/cancel` | 无 |
| `POST /api/v1/orders/{orderId}/complete` | 无 |
| `POST /api/v1/my/store/orders/{orderId}/ship` | 有 |

### 16B.2 使用方式

在 HTTP 请求头中传入可选 `Idempotency-Key`：

```http
POST /api/v1/orders
Authorization: Bearer <accessToken>
Idempotency-Key: unique-key-from-client-001
Content-Type: application/json

{
  "addressId": "2071642363297644501",
  "cartItemIds": ["2071642363297644550"]
}
```

### 16B.3 重复提交行为

| 场景 | 行为 |
|---|---|
| 首次请求 | 正常执行业务逻辑，返回结果 |
| 同 key 重复（请求体一致） | 返回首次响应数据或首次错误 |
| 同 key 重复（请求体不一致） | 返回 `IDEMPOTENCY_KEY_MISMATCH` |
| 同 key 并发 | 返回 `IDEMPOTENCY_IN_PROCESSING`，客户端应等 1-2 秒后重试 |
| 不同用户同 key | 独立计算，互不影响 |
| 不同路径同 key | 独立计算，互不影响 |
| 未传 key | 保持旧行为，不做幂等保护 |

### 16B.4 响应样例

首次成功后的重复请求返回相同 `data`（外层 `traceId` 重新生成）：

```json
{
  "success": true,
  "data": { /* 与首次响应相同的结构 */ },
  "traceId": "req_002"
}
```

首次失败后的重复请求返回相同错误：

```json
{
  "success": false,
  "error": {
    "code": "INSUFFICIENT_STOCK",
    "message": "库存不足"
  },
  "traceId": "req_002"
}
```

请求体冲突：

```json
{
  "success": false,
  "error": {
    "code": "IDEMPOTENCY_KEY_MISMATCH",
    "message": "同一 Idempotency-Key 请求体不一致"
  },
  "traceId": "req_002"
}
```

返回结果懒更新过期券（查询时自动将已过期的 `unused` 券标记为 `expired`）。

### 16D.4 常见错误码

| 错误码 | 场景 |
|---|---|
| `COUPON_NOT_AVAILABLE` | 优惠券已停用、不在有效期内或未达到使用门槛 |
| `COUPON_ALREADY_CLAIMED` | 已达到每人限领数量 |
| `COUPON_OUT_OF_STOCK` | 优惠券已被领完 |
| `UNAUTHORIZED` | 未登录 |

### 16D.5 下单用券流程

1. 预览时省略 `userCouponId`，响应中带 `availableCoupons` / `unavailableCoupons` 供前端展示
2. 预览时指定 `userCouponId`，响应带 `selectedCoupon` 和抵扣后金额
3. 创建订单时传 `userCouponId`，券状态变为 `locked`
4. 模拟支付成功后券状态变为 `used`
5. 取消待支付订单后券状态恢复为 `unused`

## 17. 售后退款

P1 Batch 05 新增。买家对已支付订单申请售后，团长审核/拒绝/完成模拟退款。

| 项目 | 内容 |
|---|---|
| 页面入口 | 订单详情页 -> 申请售后；团长后台 -> 售后管理 |
| 前置条件 | 订单已支付（payStatus=paid，orderStatus 为 paid/shipped/completed 之一） |
| 调用顺序 | 买家申请 -> 团长审核通过 -> 团长完成模拟退款；或团长拒绝 |

### 17.1 申请售后

```http
POST /api/v1/orders/{orderId}/after-sales
Authorization: Bearer <buyerAccessToken>
Content-Type: application/json

{
  "type": "refund",
  "reason": "商品质量问题"
}
```

**业务规则：**
- 仅订单本人可申请。
- 仅 `payStatus=paid` 且 `orderStatus in paid/shipped/completed` 的订单可申请。
- 同一订单不能存在 `pending` 或 `approved` 的售后单。
- 创建后订单状态变为 `afterSale`（DB 为 `after_sale`）。

**响应样例：**

```json
{
  "success": true,
  "data": {
    "id": 10001,
    "orderId": 9001,
    "orderNo": "202606240001",
    "type": "refund",
    "reason": "商品质量问题",
    "status": "pending",
    "amount": 2990,
    "originalOrderStatus": "paid",
    "orderStatus": "afterSale",
    "payStatus": "paid",
    "createdAt": "2026-07-02T12:00:00+08:00"
  },
  "traceId": "req_001"
}
```

### 17.2 买家查看售后

**我的售后列表：**

```http
GET /api/v1/my/after-sales?page=1&pageSize=20
Authorization: Bearer <buyerAccessToken>
```

**我的售后详情：**

```http
GET /api/v1/my/after-sales/{afterSaleId}
Authorization: Bearer <buyerAccessToken>
```

### 17.3 团长处理售后

**店铺售后列表：**

```http
GET /api/v1/my/store/after-sales?page=1&pageSize=20
Authorization: Bearer <leaderAccessToken>
```

**店铺售后详情：**

```http
GET /api/v1/my/store/after-sales/{afterSaleId}
Authorization: Bearer <leaderAccessToken>
```

**审核通过（pending → approved）：**

```http
POST /api/v1/my/store/after-sales/{afterSaleId}/approve
Authorization: Bearer <leaderAccessToken>
```

成功后 `status=approved`，订单保持 `afterSale`。

**拒绝售后（pending → rejected）：**

```http
POST /api/v1/my/store/after-sales/{afterSaleId}/reject
Authorization: Bearer <leaderAccessToken>
Content-Type: application/json

{
  "rejectReason": "商品完好，不予退款"
}
```

成功后 `status=rejected`，订单恢复为申请时的状态。

**完成退款（approved → completed）：**

```http
POST /api/v1/my/store/after-sales/{afterSaleId}/complete-refund
Authorization: Bearer <leaderAccessToken>
```

**退款完成后：**
- 售后 `status=completed`
- 订单 `orderStatus=refunded`，`payStatus=refunded`
- 库存返还：`group_stock += quantity`，`sold_count -= quantity`（最低 0）
- 会员冲正：`total_order_amount -= payAmount`，`growth_value -= payAmount`，`total_orders -= 1`（最低 0），重算等级
- 优惠券不退回
- 重复调用幂等，不重复执行副作用

### 17.4 常见错误码

| 错误码 | HTTP | 场景 |
|---|---|---|
| `AFTER_SALE_NOT_APPLICABLE` | 422 | 订单未支付/状态不可售后 |
| `AFTER_SALE_IN_PROGRESS` | 409 | 已有进行中的售后单 |
| `AFTER_SALE_NOT_APPROVABLE` | 422 | 售后单不可审核通过 |
| `AFTER_SALE_NOT_REJECTABLE` | 422 | 售后单不可拒绝 |
| `AFTER_SALE_NOT_REFUNDABLE` | 422 | 售后单不可退款 |
| `UNAUTHORIZED` | 401 | 未登录 |
| `RESOURCE_NOT_FOUND` | 404 | 售后单/订单不存在 |
| `STORE_FORBIDDEN` | 403 | 售后单不属于当前店铺 |
| `LEADER_REQUIRED` | 403 | 非团长用户 |

### 17. Batch 06 店铺定位与距离展示（后端 + 联调）

> 本批为后端 batch，前端 Geolocation 接入另开前端 batch。

#### 17.1 背景

- 为店铺保存 GPS 经纬度。
- 公共团购列表/详情在收到用户坐标时返回距离。
- 距离仅用于展示、可选筛选和排序，不影响下单、支付、发货、售后、优惠券或会员成长。

#### 17.2 涉及后端接口

| 接口 | 变更 |
|---|---|
| `POST /api/v1/stores` | 请求/响应中增加可选 `latitude`、`longitude` |
| `GET /api/v1/my/store` | 响应中增加 `latitude`、`longitude` |
| `PATCH /api/v1/my/store` | 请求/响应中增加可选 `latitude`、`longitude` |
| `GET /api/v1/group-buys` | 增加 `latitude`、`longitude`、`maxDistanceMeters`、`sort` 查询参数；响应中 StoreLite 增加 `latitude`、`longitude`、`distanceMeters`、`distanceText` |
| `GET /api/v1/group-buys/{groupBuyId}` | 增加 `latitude`、`longitude` 查询参数；响应中 StoreDetail 增加 `latitude`、`longitude`、`distanceMeters`、`distanceText` |

#### 17.3 联调场景

##### 场景 A：店铺创建/更新时设置坐标

1. 团长在创建店铺时请求体传入 `latitude: 31.2304, longitude: 121.4737`。
2. 验证响应中 `store.latitude` 和 `store.longitude` 返回正确坐标。
3. 团长通过 `PATCH /api/v1/my/store` 修改坐标，仅传入 `latitude` 或仅传 `longitude` 应返回 `VALIDATION_ERROR`。

##### 场景 B：公共列表展示距离

1. 用户（买家）请求 `GET /api/v1/group-buys?latitude=30.2741&longitude=120.1551`。
2. 验证响应中每个团购的 `store` 对象包含：
   - `store.latitude` / `store.longitude`（店铺坐标）
   - `store.distanceMeters`（整数米数，如 `164874`）
   - `store.distanceText`（人类可读文本，如 `"164.9km"`）
3. 不传坐标时，上述距离字段为 `null`。
4. 传入非法坐标（如 `latitude=100`）返回 `VALIDATION_ERROR`。

##### 场景 C：距离筛选

1. 请求 `GET /api/v1/group-buys?latitude=30.2741&longitude=120.1551&maxDistanceMeters=5000`。
2. 验证只返回距离 ≤ 5km 的团购。

##### 场景 D：距离排序

1. 请求 `GET /api/v1/group-buys?latitude=30.2741&longitude=120.1551&sort=distance`。
2. 验证结果按距离升序排列。

##### 场景 E：团购详情展示距离

1. 请求 `GET /api/v1/group-buys/{id}?latitude=30.2741&longitude=120.1551`。
2. 验证 `data.store` 返回店铺坐标和距离。
3. 不传坐标时距离字段为 `null`。

#### 17.4 距离计算规则

- 使用 Haversine 公式，基于 WGS84 坐标系。
- `distanceMeters`：四舍五入后的整数米数。
- `distanceText`：
  - 小于 1000m 时显示如 `"800m"`
  - 大于等于 1000m 时显示一位小数并去掉 `.0`，如 `"1km"`、`"1.2km"`

#### 17.5 限制说明

- 当前不接地图 SDK，不做逆地理编码，不存用户实时 GPS。
- 店铺坐标不参与配送范围、下单资格或履约承诺。
- 前端定位弹窗和 H5 展示控件在独立前端 batch 中实现。

### 18. P1 Batch 07 团购活动内容模型和商品图文说明（后端 + API 响应）

#### 18.1 概览

本批为团购活动补齐自有的内容表达能力。用户进入团购详情是进入一次团购活动，团购活动和商品是独立的内容来源：

| 对象 | 内容来源 | 用途 |
|---|---|---|
| 团购活动 | `groupBuy.galleryImageUrls` + `groupBuy.contentBlocks` + `groupBuy.introduction` | 团购首屏、活动正文、团长推荐 |
| 热销商品 | `featuredItem` | 默认展示、快捷购买、首屏热销卡片 |
| 售卖商品 | `items[].product.description` + `items[].product.detailImageUrls` | 商品自己的图文说明、购买弹层 |

#### 18.2 新增 / 增强 API

| 接口 | 变更 |
|---|---|
| `POST /my/store/group-buys` | 请求新增 `galleryImageUrls`、`contentBlocks` |
| `POST /my/store/group-buys/drafts` | 请求新增 `galleryImageUrls`、`contentBlocks` |
| `PATCH /my/store/group-buys/{id}` | 请求新增 `galleryImageUrls`、`contentBlocks`（传 `[]` 清空，不传不改） |
| `GET /my/store/group-buys/{id}` | 响应 `groupBuy` 新增 `galleryImageUrls`、`contentBlocks` |
| `GET /group-buys/{id}` | 响应 `groupBuy` 新增 `galleryImageUrls`、`contentBlocks`；响应新增 `featuredItem`；`items[]` 新增 `product` 嵌套对象 |
| `POST/GET /my/store/products` | 请求/响应新增 `detailImageUrls` |
| `PATCH /my/store/products/{id}` | 请求新增 `detailImageUrls`（传 `[]` 清空，不传不改） |

#### 18.3 响应字段说明

**团购活动内容**
- `galleryImageUrls: string[]` — 活动多图，最多 9 张
- `contentBlocks: ContentBlock[]` — 结构化正文，最多 20 块，支持类型：
  - `paragraph`: `text`
  - `section`: `title` + `text`
  - `image`: `url` + `caption`
  - `list`: `items: string[]`
  - `deliveryNote`: `text`
- 旧数据（无内容）返回 `[]`

**featuredItem**
- 热销商品选择规则：`soldCount` 降序 → `sortOrder` 升序 → `id` 升序
- 无商品时返回 `null`

**items[].product**
```json
{
  "id": 501,
  "name": "白玉蜜桃",
  "description": "商品自己的描述",
  "coverImageUrl": "https://...",
  "detailImageUrls": ["https://..."],
  "basePriceAmount": 3990,
  "status": "active"
}
```

#### 18.4 联调关键点

1. 进入 `/group-buys/:id` 是进入团购活动，不是进入商品。
2. 团购首屏主图使用 `groupBuy.coverImageUrl` / `groupBuy.galleryImageUrls`。
3. 热销商品使用 `featuredItem`。
4. 商品购买弹层和商品说明使用 `items[].product.description` / `items[].product.detailImageUrls`。
5. 团购列表 `GET /group-buys` 不返回完整 contentBlocks。
6. 旧数据没有 contentBlocks 时返回 `[]`，前端可用 `groupBuy.introduction` 降级渲染。

#### 18.5 安全与校验

- `galleryImageUrls` / `detailImageUrls` 最多 9 张，仅 http/https，拒绝 HTML/XSS。
- `contentBlocks` 最多 20 块，类型白名单校验，拒绝 HTML/script/iframe/style/javascript:。
- 非法内容返回 `VALIDATION_ERROR`。
