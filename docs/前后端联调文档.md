# 前后端联调文档

> 来源：基于 `docs/功能需求定义.md`、`docs/API风格规范.md`、`docs/API设计.md`、`docs/页面与交互文档.md` 和 `docs/MVP开发指南.md` 整理。
>
> 定位：本文用于记录 MVP 阶段前后端按功能链路联调的入口、前置条件、调用顺序、样例、成功判定、错误码和测试数据准备方式。
>
> 维护规则：本文按功能链路组织，不按 Controller 文件组织。每个后端 batch 完成后，必须同步更新对应链路，不等待所有后端接口完成后再统一联调。

## 1. 联调通用约定

### 1.1 基础信息

| 项目 | 约定 |
|---|---|
| API 前缀 | `/api/v1` |
| 数据格式 | JSON |
| 字段命名 | `camelCase` |
| 时间格式 | ISO 8601 字符串，如 `2026-06-24T12:00:00+08:00` |
| 金额格式 | 整数分，字段名以 `Amount` 结尾 |
| 登录头 | `Authorization: Bearer <accessToken>` |

### 1.2 成功响应

```json
{
  "success": true,
  "data": {},
  "traceId": "req_001"
}
```

### 1.3 错误响应

```json
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "参数校验失败",
    "details": {}
  },
  "traceId": "req_001"
}
```

### 1.4 常用错误码

| 错误码 | 场景 |
|---|---|
| `UNAUTHORIZED` | 未登录或令牌无效 |
| `VALIDATION_ERROR` | 参数校验失败 |
| `LEADER_REQUIRED` | 当前操作需要团长身份 |
| `STORE_ALREADY_EXISTS` | 当前用户已创建店铺 |
| `STORE_FORBIDDEN` | 不能操作他人店铺 |
| `GROUP_BUY_NOT_PURCHASABLE` | 团购不可购买 |
| `GROUP_BUY_ENDED` | 团购已结束 |
| `ITEM_NOT_IN_GROUP_BUY` | 下单商品不属于该团购 |
| `ADDRESS_FORBIDDEN` | 地址不属于当前用户 |
| `INSUFFICIENT_STOCK` | 库存不足 |
| `ORDER_NOT_PAYABLE` | 订单不可支付 |
| `ORDER_ALREADY_PAID` | 订单已支付 |
| `ORDER_NOT_SHIPPABLE` | 订单不可发货 |
| `ORDER_ALREADY_SHIPPED` | 订单已发货 |
| `SUBSCRIPTION_EXISTS` | 已订阅该团长 |

### 1.5 空库初始化说明

首次启动后端项目时，数据库迁移会自动执行：

| 步骤 | 说明 |
|---|---|
| 创建数据库 | 在 MySQL 中创建 `groupshop` 数据库（如 `CREATE DATABASE groupshop DEFAULT CHARSET utf8mb4`） |
| 配置连接 | 在 `application.yml` 中配置 `spring.datasource` 连接信息 |
| 启动应用 | 运行 `GroupshopApplication`，Flyway 自动执行 `classpath:db/migration/` 下的迁移脚本 |
| 迁移结果 | 空库执行 Flyway 后，一次性获得全部 12 张 MVP 核心表 |
| 验证迁移 | 启动日志中应看到 Flyway 迁移成功记录，或调用 `GET /api/v1/health` 确认应用可用 |

本地开发推荐配置：
- 数据库名称：`groupshop_dev`（通过 `application-dev.yml` profile 切换）
- Flyway 迁移位置：`src/main/resources/db/migration/`
- 迁移版本管理：后续 batch 只在确有必要时追加迁移文件，不反复修改 V1

### 1.6 前端本地启动与通用 H5 约定

#### 启动方式

```bash
cd frontend
npm install
# 配置环境变量（可选，默认转发到 localhost:8080）
cp .env.example .env.local
# VITE_API_BASE_URL=http://localhost:8080
npm run dev          # 启动开发服务器，默认 5173 端口
```

#### 可用脚本

| 命令 | 说明 |
|---|---|
| `npm run dev` | 启动 Vite 开发服务器 |
| `npm run build` | 生产构建 |
| `npm run preview` | 预览生产构建 |
| `npm run typecheck` | TypeScript 类型检查（vue-tsc） |
| `npm run lint` | ESLint 检查 |
| `npm run format` | Prettier 格式化 |
| `npm run test:unit` | Vitest 单元测试 |
| `npm run test:e2e` | Playwright E2E 测试（需先运行 `npx playwright install` 安装浏览器） |

#### H5 通用约定

1. **Safe Area 适配**：页面已包含 `viewport-fit=cover` meta 和 CSS `env(safe-area-inset-*)` 变量，底部 TabBar 和内容面已自动适配刘海屏和小白条。
2. **金额展示**：后端金额以"整数分"传输。前端统一使用 `formatAmount(amount: number | null | undefined)` 转换为 `¥xx.xx` 格式展示。**严禁将金额作为浮点数直接使用或传递回后端。**
3. **非 MVP 灰态入口**：真实微信支付、购物车、优惠券/红包、售后退款、帮卖分销、积分商城、公众号推送、平台后台等非 MVP 功能在前端均保持禁用/灰态，不可跳转。可使用 `isFeatureDisabled(feature)` 工具函数判断。
4. **状态文案**：店铺、团购、订单、支付、配送等状态码统一通过 `src/utils/status.ts` 中的映射函数转换为中文展示，未知状态回退为"未知状态"或原值。
5. **API 前缀**：所有接口请求以 `/api/v1` 开头，开发时由 Vite proxy 转发到后端，无需单独配置跨域。

## 2. 测试数据总口径

MVP 联调建议至少准备以下数据：

| 数据 | 准备方式 |
|---|---|
| 买家用户 | 调用 `POST /api/v1/auth/mock-login` 创建或获取 |
| 团长用户 | 使用另一个手机号调用 `POST /api/v1/auth/mock-login` |
| 店铺 | 团长用户调用 `POST /api/v1/stores` 创建 |
| 团购 | 团长用户调用 `POST /api/v1/my/store/group-buys` 发布普通团购 |
| 地址 | 买家用户调用 `POST /api/v1/my/addresses` 创建 |
| 订单 | 买家用户调用 `POST /api/v1/orders` 创建 |
| 已支付订单 | 买家用户调用 `POST /api/v1/orders/{orderId}/simulate-pay` |
| 已发货订单 | 团长用户调用 `POST /api/v1/my/store/orders/{orderId}/ship` |
| 会员关系 | 买家对某店铺订单模拟支付成功后自动创建或更新 |

说明：同一条联调链路中的 ID，如 `leaderId`、`storeId`、`groupBuyId`、`groupBuyItemId`、`addressId`、`orderId`，均以前一步接口响应为准，不写死。

| 项目 | 判定 |
|---|---|
| 登录成功 | 返回 `accessToken` 和 `user` |

| 未创建店铺用户 | `hasLeader=false`，`leaderId=null`，`storeId=null` |

### 3.4 常见错误码

| 错误码 | 场景 |
|---|---|
| `VALIDATION_ERROR` | 昵称、手机号等参数不合法 |
| `UNAUTHORIZED` | `GET /api/v1/me` 未携带或携带无效 token |

## 4. 创建店铺

| 项目 | 内容 |
|---|---|
| 页面入口 | 个人中心页 -> 创建店铺页 |
| 前置条件 | 用户已登录，且当前用户未创建店铺 |
| 调用顺序 | `GET /api/v1/me` 判断是否已有店铺 -> `POST /api/v1/stores` -> `GET /api/v1/me` 刷新身份 |

### 4.1 请求样例

```http
POST /api/v1/stores
Authorization: Bearer <accessToken>
Content-Type: application/json
```

```json
{
  "name": "某某的小店",
  "logoUrl": "https://example.com/logo.png",
  "description": "店铺简介",
  "defaultDeliveryType": "express"
}
```

### 4.2 响应样例

```json
{
  "success": true,
  "data": {
    "leader": {
      "id": 10,
      "displayName": "某某的小店",
      "avatarUrl": "https://example.com/logo.png"
    },
    "store": {
      "id": 20,
      "leaderId": 10,
      "name": "某某的小店",
      "logoUrl": "https://example.com/logo.png",
      "description": "店铺简介",
      "defaultDeliveryType": "express",
      "distributionEnabled": false,
      "status": "active"
    }
  },
  "traceId": "req_001"
}
```

### 4.3 成功判定

创建成功后生成 `leader` 和 `store`；再次调用 `GET /api/v1/me` 时 `user.hasLeader=true`，并返回 `leaderId`、`storeId`。

### 4.4 获取当前用户店铺

创建店铺后，可通过 `GET /api/v1/my/store` 获取当前用户自己的店铺信息。

#### 请求样例

```http
GET /api/v1/my/store
Authorization: Bearer <accessToken>
```

#### 响应样例（已有店铺）

```json
{
  "success": true,
  "data": {
    "leader": {
      "id": 10,
      "displayName": "某某的小店",
      "avatarUrl": "https://example.com/logo.png"
    },
    "store": {
      "id": 20,
      "leaderId": 10,
      "name": "某某的小店",
      "logoUrl": "https://example.com/logo.png",
      "description": "店铺简介",
      "defaultDeliveryType": "express",
      "distributionEnabled": false,
      "status": "active"
    }
  },
  "traceId": "req_001"
}
```

#### 响应样例（未创建店铺）

```json
{
  "success": true,
  "traceId": "req_001"
}
```

说明：未创建店铺时，`GET /api/v1/my/store` 返回 200 成功，但省略 `data` 字段。前端应通过 `response.data` 是否存在判断。已有店铺时同时返回团长身份(`leader`)和店铺(`store`)信息。

### 4.6 更新当前用户店铺

创建店铺后，可通过 `PATCH /api/v1/my/store` 局部更新店铺信息。未创建店铺时返回 `LEADER_REQUIRED`。

#### 请求样例

```http
PATCH /api/v1/my/store
Authorization: Bearer <accessToken>
Content-Type: application/json
```

```json
{
  "name": "新店铺名称",
  "description": "新简介"
}
```

#### 响应样例

```json
{
  "success": true,
  "data": {
    "leader": {
      "id": 10,
      "displayName": "新店铺名称",
      "avatarUrl": "https://example.com/logo.png"
    },
    "store": {
      "id": 20,
      "leaderId": 10,
      "name": "新店铺名称",
      "logoUrl": "https://example.com/logo.png",
      "description": "新简介",
      "defaultDeliveryType": "express",
      "distributionEnabled": false,
      "status": "active"
    }
  },
  "traceId": "req_001"
}
```

### 4.7 常见错误码

| 错误码 | 场景 |
|---|---|
| `UNAUTHORIZED` | 未登录 |
| `STORE_ALREADY_EXISTS` | 当前用户已创建店铺（重复创建） |
| `LEADER_REQUIRED` | 未创建店铺时更新店铺 |
| `VALIDATION_ERROR` | 店铺名称或配送方式不合法 |

### 4.8 测试数据准备方式

使用团长测试手机号完成模拟登录，然后调用创建店铺接口。重复创建场景使用同一 token 再次提交。更新场景先创建店铺再局部更新。

## 4A. 商品管理

| 项目 | 内容 |
|---|---|
| 页面入口 | 团长后台 -> 商品管理页 |
| 前置条件 | 用户已登录，且已经创建店铺成为团长 |
| 调用顺序 | `POST /api/v1/my/store/products` 创建商品 -> `GET /api/v1/my/store/products` 查看列表 -> `GET /api/v1/my/store/products/{productId}` 查看详情 -> `PATCH /api/v1/my/store/products/{productId}` 更新 -> `DELETE /api/v1/my/store/products/{productId}` 删除 |

### 4A.1 创建商品

```http
POST /api/v1/my/store/products
Authorization: Bearer <leaderAccessToken>
Content-Type: application/json
```

```json
{
  "name": "白玉蜜桃",
  "description": "山东蒙阴产地直发",
  "coverImageUrl": "https://example.com/product.png",
  "basePriceAmount": 2990,
  "stock": 100
}
```

响应样例：

```json
{
  "success": true,
  "data": {
    "id": 501,
    "storeId": 20,
    "name": "白玉蜜桃",
    "description": "山东蒙阴产地直发",
    "coverImageUrl": "https://example.com/product.png",
    "basePriceAmount": 2990,
    "stock": 100,
    "status": "active"
  },
  "traceId": "req_001"
}
```

### 4A.2 商品列表

```http
GET /api/v1/my/store/products?page=1&pageSize=20
Authorization: Bearer <leaderAccessToken>
```

响应样例：

```json
{
  "success": true,
  "data": {
    "items": [
      {
        "id": 501,
        "storeId": 20,
        "name": "白玉蜜桃",
        "basePriceAmount": 2990,
        "stock": 100,
        "status": "active"
      }
    ],
    "page": 1,
    "pageSize": 20,
    "total": 1,
    "hasMore": false
  },
  "traceId": "req_001"
}
```

### 4A.3 商品详情

```http
GET /api/v1/my/store/products/{productId}
Authorization: Bearer <leaderAccessToken>
```

响应同创建响应，返回完整的商品对象。

### 4A.4 更新商品

```http
PATCH /api/v1/my/store/products/{productId}
Authorization: Bearer <leaderAccessToken>
Content-Type: application/json
```

```json
{
  "name": "新名称",
  "basePriceAmount": 2000
}
```

说明：PATCH 仅更新传入字段，不传字段保持原值。

### 4A.5 删除商品

```http
DELETE /api/v1/my/store/products/{productId}
Authorization: Bearer <leaderAccessToken>
```

说明：使用软删除，删除后商品状态变为 `deleted`，不再出现在商品列表中。
已删除商品被团购引用时不影响历史订单快照。

### 4A.6 常见错误码

| 错误码 | 场景 |
|---|---|
| `UNAUTHORIZED` | 未登录 |
| `LEADER_REQUIRED` | 当前用户不是团长 |
| `STORE_FORBIDDEN` | 商品不属于当前店铺（跨店铺操作） |
| `RESOURCE_NOT_FOUND` | 商品不存在 |
| `VALIDATION_ERROR` | 商品名称、金额、库存参数不合法 |

### 4A.7 测试数据准备方式

使用团长账号登录后调用创建商品接口。跨店铺场景使用另一个团长账号的商品 ID。

| 前置条件 | 买家已登录；有无会员关系均需支持 |
| 调用顺序 | `GET /api/v1/my/member-cards` |

### 13.1 请求样例

```http
GET /api/v1/my/member-cards
Authorization: Bearer <buyerAccessToken>
```

### 13.3 成功判定

无会员关系时返回空列表；支付成功后返回对应店铺会员关系；`totalOrderAmount` 和 `totalOrders` 与已支付订单累计一致。

### 13.4 常见错误码

| 错误码 | 场景 |
|---|---|
| `UNAUTHORIZED` | 未登录 |

### 13.5 测试数据准备方式

空状态可使用新买家账号直接访问。已有会员关系需要买家完成至少一笔订单模拟支付。
