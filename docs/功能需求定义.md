## 1. 产品目标

本项目要实现一个以微信私域传播为主要场景的社群团购系统。系统围绕“用户可成为团长、团长创建店铺、店铺发布团购、用户跟团购买、团长履约、用户复购订阅”展开。

核心目标：

| 目标 | 说明 |
|---|---|
| 买家可浏览和购买团购商品 | 用户可以从首页、团长主页、商品详情进入购买流程 |
| 普通用户可低门槛成为团长 | 用户可以创建店铺，获得团长主页，并发布团购 |
| 团长可发布普通团购 | 团长可以填写团购介绍、添加商品、设置基础规则并发布 |
| 系统可完成交易闭环 | 支持下单、支付状态记录、订单查看、团长发货管理 |
| 为后续运营能力预留扩展 | 会员、订阅、优惠、帮卖、消息、后台管理可逐步增强 |

---

## 2. 功能优先级定义

| 优先级 | 含义 | 处理方式 |
|---|---|---|
| MVP | 最小可用闭环 | 第一阶段必须实现，能跑通核心业务 |
| P1 | 增强体验和运营效率 | MVP 后优先补充，不影响主闭环成立 |
| P2 | 完整平台能力 | 后续扩展，主要包含复杂运营、风控、分销和后台 |

本项目不是缩小产品形态，而是分阶段实现完整产品能力。MVP 保留身份、店铺、团购、商品、订单的核心骨架；P1/P2 补齐营销、分销、触达、风控和后台。

## 3. 术语表

| 术语 | 定义 | 关键说明 |
|---|---|---|
| 用户 | 系统中的基础账号主体，可以浏览、购买，也可以申请经营能力 | 用户不是天然团长，但可以通过创建店铺激活团长身份 |
| 团员 | 用户与某个团长 / 店铺产生订阅、购买或会员关系后的业务身份 | 团员是相对于某个团长存在的关系身份 |
| 团长 | 用户激活经营能力后的身份，是买家看到的信任主体 | 团长代表人设、信用、服务和私域关系，不等同于店铺 |
| 店铺 | 团长的经营容器，承载店铺资料、默认物流、团购列表、商品资产 | 店铺是长期经营资产 |
| 团购 | 店铺发布的一次销售活动，包含介绍、商品和活动规则 | 团购有发布、结束、下架等状态 |
| 商品 | 店铺下长期可复用的售卖资产 | 商品可以被多个团购引用 |
| SKU | 商品的具体规格，如重量、数量、颜色、套餐 | MVP 可先支持单规格，复杂多规格可后续增强 |
| 团购商品 | 商品在某次团购中的售卖配置 | 下单价格、团购库存、展示名称以团购商品为准 |
| 订单 | 用户购买某次团购商品后产生的交易记录 | 订单必须关联用户、团长、店铺、团购和商品快照 |
| 订阅关系 | 用户关注某个团长 / 店铺的关系 | MVP 只表示关注关系，不做真实消息推送 |
| 会员关系 | 用户与某个团长 / 店铺之间的私域会员关系 | MVP 只做基础展示，不做等级升级、折扣计算和积分商城 |

## 4. 核心业务闭环

### 4.1 MVP 实现闭环

```text
用户进入系统
    ↓
浏览首页商品流
    ↓
进入商品详情
    ↓
跟团购买
    ↓
填写地址、数量、备注
    ↓
生成订单
    ↓
团长查看订单并发货
    ↓
用户查看订单状态
```

```text
用户进入个人中心 / 一键开团
    ↓
创建店铺
    ↓
生成团长主页
    ↓
发布普通团购
    ↓
添加商品
    ↓
设置团购时间、物流方式、发货时间
    ↓
发布团购
    ↓
商品出现在团长主页和首页商品流
```

### 4.2 MVP 阶段实现范围

| 环节 | MVP 是否实现 | 说明 |
|---|---:|---|
| 用户浏览商品 | 是 | 首页商品流、商品卡片、商品详情 |
| 用户创建店铺 | 是 | 店铺名称、头像、简介、物流方式 |
| 用户成为团长 | 是 | 创建店铺后生成团长身份和默认主页 |
| 团长主页 | 是 | 展示团长信息、店铺信息、团购商品列表、空状态 |
| 发布普通团购 | 是 | 只支持普通团购 |
| 团购介绍 | 是 | 支持标题、介绍、图片等基础内容 |
| 添加商品 | 是 | 发布团购时创建或选择商品 |
| 商品规格价格库存 | 是 | 支持基础规格、价格、库存 |
| 团购时间设置 | 是 | 开始时间、结束时间 |
| 物流和发货时间 | 是 | 基础物流方式、承诺发货时间 |
| 下单确认 | 是 | 地址、数量、备注、协议确认 |
| 订单生成 | 是 | 记录订单、商品、金额、状态 |
| 团长订单管理 | 是 | 查看订单、标记发货 |
| 用户订单中心 | 是 | 全部、待支付、待发货、已发货；售后只预留状态 |
| 基础订阅关系 | 是 | 支持关注 / 订阅团长或店铺的关系记录 |
| 会员卡基础展示 | 是 | 展示用户与团长 / 店铺的会员关系，不做复杂成长规则 |
| 模拟支付 | 是 | MVP 不接真实支付，使用模拟支付完成状态流转 |

### 4.3 MVP 暂不实现范围

| 功能 | MVP 状态 | 后续优先级 | 说明 |
|---|---|---:|---|
| 预售团购 | 不实现 | P1 | MVP 只做普通团购 |
| 卡券团购 | 不实现 | P2 | 涉及发码和核销 |
| 报名团购 | 不实现 | P2 | 涉及报名表单和统计 |
| 商品库复用 | 简化 | P1 | MVP 可在发团时直接创建商品，后续支持商品库 |
| 优惠券 / 红包 | 不实现 | P1 | 可先预留字段和入口 |
| 会员成长值规则 | 简化 | P1 | MVP 只做会员关系展示 |
| 积分商城 | 不实现 | P2 | 后续运营能力 |
| 帮卖分销 | 不实现 | P2 | MVP 只预留店铺帮卖开关字段 |
| 佣金结算 | 不实现 | P2 | 依赖帮卖体系 |
| 隐私权限细分 | 简化 | P1 | MVP 默认公开团购 |
| 复制团购权限 | 不实现 | P1 | 后续增强发布效率和权限控制 |
| 分享海报 | 简化 | P1 | MVP 可仅支持分享链接 |
| 站内通知 | 不实现 | P1 | 可先用轮询实现活动、订单和物流通知，不依赖外部平台 |
| 公众号 / 服务通知 | 不实现 | P2 | 需要外部平台能力 |
| 平台管理后台 | 不实现 | P2 | 放入后续扩展 |
| 风控审核 | 不实现 | P2 | 后台扩展能力 |
| 数据统计看板 | 不实现 | P2 | 后续经营分析能力 |

## 5. 角色定义

| 角色 | 定义 | MVP 能力 | 后续能力 |
|---|---|---|---|
| 基础用户 | 所有进入系统的用户 | 浏览商品、下单、查看订单、创建店铺 | 收藏、浏览历史、优惠券、会员成长 |
| 团员 | 订阅或购买某团长商品的用户 | 订阅团长、跟团购买、查看会员卡基础信息 | 积分、成长值、专属折扣 |
| 团长 | 创建店铺后的经营身份 | 编辑店铺、发布普通团购、查看订单、发货 | 会员运营、优惠、帮卖、数据统计 |
| 店铺 | 团长的经营载体 | 承载店铺资料、团购列表、物流配置 | 帮卖分销、积分商城、素材库 |
| 平台管理员 | 平台运营和审核角色 | MVP 不实现 | 用户管理、团长管理、商品审核、订单监管 |

### 5.1 角色权限规则

| 能力 | 未登录用户 | 普通用户 | 团长 |
|---|---|---|---|
| 浏览首页商品 | 可以 | 可以 | 可以 |
| 查看商品详情 | 可以 | 可以 | 可以 |
| 查看团长主页 | 可以 | 可以 | 可以 |
| 跟团下单 | 不可以，需要登录 | 可以 | 可以 |
| 查看自己的订单 | 不可以，需要登录 | 可以 | 可以 |
| 管理收货地址 | 不可以，需要登录 | 可以 | 可以 |
| 订阅团长 / 店铺 | 不可以，需要登录 | 可以 | 可以 |
| 查看会员卡 | 不可以，需要登录 | 可以 | 可以 |
| 创建店铺 | 不可以，需要登录 | 可以 | 已创建后不可重复创建 MVP 店铺 |
| 发布团购 | 不可以 | 不可以，需要先创建店铺成为团长 | 可以 |
| 管理商品 | 不可以 | 不可以 | 只能管理自己店铺下的商品 |
| 管理团购 | 不可以 | 不可以 | 只能管理自己店铺发布的团购 |
| 查看团长订单 | 不可以 | 不可以 | 只能查看自己店铺 / 团购产生的订单 |
| 发货 | 不可以 | 不可以 | 只能处理自己店铺 / 团购产生的订单 |
| 修改他人店铺、团购、商品、订单 | 不可以 | 不可以 | 不可以 |

权限规则补充：

- 未登录用户只能浏览公开内容，不能产生订单、订阅、会员关系或经营数据。
- 普通用户登录后可以购买、订阅、查看自己的订单和会员关系。
- 普通用户创建店铺后，系统生成团长身份。
- 团长仍然保留普通用户能力，可以作为买家购买其他团长的商品。
- 团长只能管理自己的店铺、团购、商品和订单，不能跨店铺操作。

## 6. 核心业务规则

| 规则 | MVP 要求 |
|---|---|
| 团长激活 | 用户创建店铺后，系统生成团长身份 |
| 用户与团长 | 一个用户 MVP 阶段最多拥有一个团长身份 |
| 团长与店铺 | 一个团长 MVP 阶段最多拥有一个店铺 |
| 店铺与团购 | 一个店铺可以发布多个团购 |
| 团购与商品 | 一个团购可以包含一个或多个商品 |
| 商品资产 | 商品是店铺下长期可复用资产，MVP 可先在发布团购时创建 |
| 团购商品 | 团购商品是商品在某次团购中的售卖配置 |
| 下单价格 | 用户下单价格以团购商品价格为准，不以商品基础价格为准 |
| 库存口径 | MVP 以团购商品库存为主要售卖库存；复杂库存同步后续处理 |
| 订单快照 | 订单明细必须保存下单时的商品名称、规格、单价、数量等快照 |
| 支付方式 | MVP 支付为模拟支付，不接入真实微信支付 |
| 售后退款 | MVP 只预留售后、退款状态，不实现完整申请、审核、退款流程 |
| 优惠券 | MVP 不实现优惠券领取、核销和抵扣流程 |
| 帮卖 | MVP 只预留帮卖相关字段，不实现帮卖、佣金和结算 |
| 积分 | MVP 不实现积分账户、积分流水和积分商城 |
| 会员 | MVP 只展示基础会员关系，不实现等级升级规则和折扣计算 |
| 订阅 | MVP 只记录关注关系，不实现公众号推送、订阅邀请卡和订阅红包 |

## 10. 状态定义与触发条件

### 10.1 店铺状态

| 状态 | 触发条件 | MVP 使用 |
|---|---|---:|
| 未创建 | 用户尚未完成创建店铺流程 | 是 |
| 已创建 | 用户提交店铺名称、头像、简介、物流等基础资料后生成店铺 | 是 |
| 已停用 | 平台停用或用户关闭店铺 | P2 |

### 10.2 团购状态

| 状态 | 触发条件 | MVP 使用 |
|---|---|---:|
| 草稿 | 团长保存但未发布团购 | P1 |
| 已发布 | 团长提交发布，且团购达到可售条件 | 是 |
| 已结束 | 当前时间超过团购结束时间，或团长手动结束 | 是 |
| 已下架 | 平台因审核、违规或风控下架 | P2 |

MVP 可不做草稿保存，团长发布后直接进入已发布状态。

### 10.3 订单状态

| 状态 | 触发条件 | MVP 使用 |
|---|---|---:|
| 待支付 | 用户提交订单但尚未完成模拟支付 | 是 |
| 已支付 | 用户完成模拟支付，订单进入待发货阶段 | 是 |
| 已发货 | 团长填写发货信息或标记发货 | 是 |
| 已完成 | 用户确认收货，或系统按规则自动完成 | 是 |
| 已取消 | 用户未支付前取消，或订单超时关闭 | 是 |
| 售后中 | 用户提交退款 / 退货申请 | P1 |
| 已退款 | 售后通过并完成退款 | P1 |

### 10.4 支付状态

| 状态 | 触发条件 | MVP 使用 |
|---|---|---:|
| 未支付 | 订单创建后默认状态 | 是 |
| 已支付 | 用户点击模拟支付成功 | 是 |
| 已退款 | 售后退款完成 | P1 |

MVP 支付说明：

- MVP 不接真实微信支付。
- 模拟支付成功后，订单支付状态变为已支付，订单状态变为已支付 / 待发货。
- 模拟支付不处理支付渠道、支付回调、退款回调、对账和资金结算。

## 13. 文档冲突处理说明

调研文档中的优先级表示产品重要性，本文中的 MVP/P1/P2 表示本项目实现优先级。以下能力在调研中很重要，但本项目 MVP 不完整实现：

| 能力 | 调研文档判断 | 本文处理 | 说明 |
|---|---|---|---|
| 微信支付 | 交易闭环关键能力 | MVP 使用模拟支付 | 产品重要，但本阶段不接真实支付 |
| 订阅邀请卡 / 分享至微信 | 私域沉淀重要能力 | MVP 不实现，仅保留订阅关系 | 先记录关注关系，后续扩展邀请和触达 |
| 站内通知 | 复购触达基础能力 | P1 | 不依赖外部平台，可通过轮询实现订单、物流和开团通知 |
| 公众号 / 服务通知 | 复购触达重要能力 | P2 | 依赖外部平台能力 |
| 帮卖分销 | 目标产品特色能力 | P2 | MVP 只预留字段，不做佣金和结算 |
| 会员成长和折扣 | 复购运营重要能力 | MVP 只做基础展示 | 不做升级规则、折扣计算和积分商城 |
| 后台管理 | 完整平台必要能力 | P2 | MVP 不做平台后台 |
| 售后退款 | 交易完整性能力 | P1 | MVP 只预留状态，不做完整流程 |
| 商品库 | 团长效率能力 | P1 | MVP 可在发团时创建商品 |
| 权限细分 | 私域传播控制能力 | P1 | MVP 默认公开团购 |

当出现理解冲突时，以本文的实现边界为准。

## 15. 阶段结论

第一阶段按 MVP 完成核心交易和开团闭环：

```text
用户浏览商品
→ 用户创建店铺成为团长
→ 团长发布普通团购
→ 用户下单
→ 团长发货
→ 用户查看订单
```

P1 重点补齐体验和运营效率：商品库、优惠券、搜索、购物车、售后、权限设置、分享海报、保存预览、站内通知和上传资产治理。

P2 重点扩展完整平台能力：帮卖分销、佣金结算、积分商城、客服聊天、公众号 / 服务通知、平台管理后台、风控审核、数据统计。其中订单关系下的轻量客服聊天已先行落地，完整客服中心仍不在当前范围。
