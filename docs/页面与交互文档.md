## 2. 全局登录拦截流程

登录拦截只在本节统一描述，其他业务流程只保留“是否已登录”的判断节点。

```mermaid
flowchart TD
    A[用户触发需要登录的操作] --> B{是否已登录}
    B -- 是 --> C[继续原操作]
    B -- 否 --> D[打开登录页或登录弹窗]
    D --> E[手机号获取验证码]
    E --> F[提交验证码登录/注册]
    F --> G{认证是否成功}
    G -- 是 --> H[保存 accessToken]
    H --> I[调用 GET /api/v1/me]
    I --> J[恢复并继续原操作]
    G -- 否 --> K[展示认证失败提示]
    K --> D
```

需要登录的典型操作：

| 操作 | 登录后继续 |
|---|---|
| 跟团购买 | 进入下单确认页 |
| 创建订单 | 停留在下单确认页并重试提交 |
| 模拟支付 | 回到订单详情页并继续支付 |
| 创建店铺 | 进入创建店铺页 |
| 发布团购 | 进入发布团购页 |
| 订阅团长 | 回到当前团长主页或团购详情 |
| 查看我的订单 / 会员卡 | 进入对应页面 |

## 3. 买家浏览与下单流程

```mermaid
flowchart TD
    A[首页] --> B[查看团购列表]
    B --> C[点击团购卡片]
    C --> D[团购详情页]
    D --> H[点击跟团购买]
    H --> E{是否已登录}
    E -- 否 --> F[触发全局登录拦截]
    F --> I[下单确认页]
    E -- 是 --> I[下单确认页]
    I --> J[调用 POST /api/v1/orders/preview]
    J --> K{预览是否成功}
    K -- 否 --> L[展示不可购买/库存不足/地址错误]
    K -- 是 --> M[展示地址 商品明细 totalAmount discountAmount payAmount]
    M --> N[用户确认提交订单]
    N --> O[前端禁用提交按钮防重复点击]
    O --> P[调用 POST /api/v1/orders]
    P --> Q{创建订单是否成功}
    Q -- 否 --> R[展示错误并恢复提交按钮]
    Q -- 是 --> S[订单详情页]
    S --> T[点击模拟支付]
    T --> U["调用 POST /api/v1/orders/{orderId}/simulate-pay"]
    U --> V{支付是否成功}
    V -- 否 --> W[展示库存不足或订单不可支付]
    V -- 是 --> X[订单状态变为 paid]
    X --> Y[等待团长发货]
```

关键交互约束：

| 节点 | 规则 |
|---|---|
| 下单确认页 | 必须先调用订单预览，不创建订单，不扣库存 |
| 提交订单 | MVP 不做强幂等，前端需要禁用按钮防重复点击 |
| 创建订单成功 | 订单为 `orderStatus=pendingPay`、`payStatus=unpaid` |
| 模拟支付成功 | 订单为 `orderStatus=paid`、`payStatus=paid` |
| 库存扣减 | 只扣减 `groupBuyItems.groupStock`，不扣减 `products.stock` |

## 4. 用户创建店铺流程

```mermaid
flowchart TD
    A[个人中心页] --> B[点击创建店铺]
    B --> C{是否已登录}
    C -- 否 --> D[触发全局登录拦截]
    D --> E[登录成功后进入创建店铺页]
    C -- 是 --> E
    E --> F[填写店铺名称 头像 简介 默认物流]
    F --> G[点击一键开店]
    G --> H[调用 POST /api/v1/stores]
    H --> I{创建是否成功}
    I -- 否 --> J{是否已创建店铺}
    J -- 是 --> K[提示 STORE_ALREADY_EXISTS 并进入我的店铺]
    J -- 否 --> L[展示参数错误]
    I -- 是 --> M[生成 leader 和 store]
    M --> N[进入团长主页或发布团购入口]
```

关键交互约束：

| 节点 | 规则 |
|---|---|
| 创建店铺 | 创建成功后同时激活团长身份 |
| 重复创建 | MVP 通过一个用户一个店铺的业务约束拦截 |
| `/my/store` | 后续店铺操作从 token 获取当前团长自己的店铺 |

## 5. 团长发布团购流程

```mermaid
flowchart TD
    A[个人中心页或团长主页管理入口] --> B[点击发布团购]
    B --> C{是否已登录}
    C -- 否 --> D[触发全局登录拦截]
    D --> E[登录成功后继续]
    C -- 是 --> E
    E --> F{是否已有团长身份和店铺}
    F -- 否 --> G[跳转创建店铺页]
    F -- 是 --> H[发布团购页]
    H --> I[填写标题 介绍 封面 时间 发货时间]
    I --> J[添加团购商品]
    J --> K{商品来源}
    K -- 内联创建 --> L[填写 product 信息和 groupStock]
    K -- 复用已有商品 --> M[选择当前店铺 productId]
    L --> N[校验 groupStock >= 0]
    M --> N
    N --> O[点击发布]
    O --> P[调用 POST /api/v1/my/store/group-buys]
    P --> Q{发布是否成功}
    Q -- 否 --> R[展示参数错误或 STORE_FORBIDDEN]
    Q -- 是 --> S[团购状态 published]
    S --> T[进入我的团购详情或团长主页]
```

关键交互约束：

| 节点 | 规则 |
|---|---|
| 团购类型 | MVP 只发布普通团购 |
| 商品创建 | 推荐内联创建商品，后端事务创建 product 和 groupBuyItem |
| 库存 | `groupStock` 必须大于等于 0，不允许 -1 |
| 团购修改 | 有订单后不可修改已下单团购商品价格，不可删除已下单团购商品 |

## 6. 用户订单状态流程

```mermaid
flowchart TD
    A[订单列表页] --> B[点击订单]
    B --> C[订单详情页]
    C --> D{订单状态}
    D -- pendingPay + unpaid --> E[显示模拟支付和取消订单]
    E --> F[点击模拟支付]
    F --> G["调用 POST /api/v1/orders/{orderId}/simulate-pay"]
    G --> H{支付成功}
    H -- 是 --> I[状态变为 paid]
    H -- 否 --> J[展示 ORDER_NOT_PAYABLE 或 INSUFFICIENT_STOCK]
    E --> K[点击取消订单]
    K --> L["调用 POST /api/v1/orders/{orderId}/cancel"]
    L --> M{取消成功}
    M -- 是 --> N[状态变为 canceled]
    M -- 否 --> O[展示 ORDER_NOT_CANCELABLE]
    D -- paid --> P[显示待发货]
    D -- shipped --> Q[显示确认收货]
    Q --> R["调用 POST /api/v1/orders/{orderId}/complete"]
    R --> S[状态变为 completed]
    D -- completed --> T[显示已完成]
    D -- canceled --> U[显示已取消]
```

状态展示规则：

| 状态 | 页面操作 |
|---|---|
| `pendingPay` + `unpaid` | 可模拟支付、可取消 |
| `paid` + `paid` | 等待团长发货 |
| `shipped` | 可确认收货 |
| `completed` | 只展示结果 |
| `canceled` | 只展示结果 |

## 7. 团长订单发货流程

```mermaid
flowchart TD
    A[团长订单列表页] --> B{是否已登录}
    B -- 否 --> C[触发全局登录拦截]
    C --> D[登录成功后返回团长订单列表]
    B -- 是 --> D
    D --> E{是否为团长且有自己的店铺}
    E -- 否 --> F[跳转创建店铺页]
    E -- 是 --> G["调用 GET /api/v1/my/store/orders"]
    G --> H[展示当前店铺订单]
    H --> I[点击订单详情]
    I --> J[团长订单详情页]
    J --> K{订单是否 paid}
    K -- 否 --> L[不可发货 展示当前状态]
    K -- 是 --> M[填写物流公司和物流单号]
    M --> N[点击发货]
    N --> O["调用 POST /api/v1/my/store/orders/{orderId}/ship"]
    O --> P{发货是否成功}
    P -- 否 --> Q[展示 STORE_FORBIDDEN 或 ORDER_NOT_SHIPPABLE]
    P -- 是 --> R[创建 shipment]
    R --> S[订单状态变为 shipped]
```

关键交互约束：

| 节点 | 规则 |
|---|---|
| 团长订单列表 | 只展示当前团长自己店铺产生的订单 |
| 发货 | 只有 `orderStatus=paid` 的订单可发货 |
| 发货成功 | 创建发货记录，订单状态变为 `shipped` |
| 重复发货 | MVP 通过订单状态校验和事务防重复 |

## 8. 订阅团长流程

```mermaid
flowchart TD
    A[团长主页或团购详情页] --> B[点击订阅]
    B --> C{是否已登录}
    C -- 否 --> D[触发全局登录拦截]
    D --> E[登录成功后继续订阅]
    C -- 是 --> E
    E --> F["调用 POST /api/v1/leaders/{leaderId}/subscription"]
    F --> G{订阅成功}
    G -- 是 --> H[viewer.subscribed=true]
    G -- 否 --> I[展示错误提示]
    H --> J[点击取消订阅]
    J --> K["调用 DELETE /api/v1/leaders/{leaderId}/subscription"]
    K --> L[viewer.subscribed=false]
```

说明：订阅表示关注团长 / 店铺，当前可触发站内通知；仍不做公众号推送、订阅邀请卡、订阅红包。

## 8.1 订单上下文轻量聊天流程（P2 部分先行）

```mermaid
flowchart TD
    A[买家订单列表/详情] --> B[点击联系团长]
    B --> C[POST /my/chat-conversations/orders/{orderId}]
    C --> D[聊天详情页]
    E[团长订单详情] --> F[点击联系买家或备货完成]
    F --> C
    D --> G[发送文字/图片/订单卡片]
    D --> H[每 5 秒轮询新消息]
    I[消息页] --> J[聊天/通知双 Tab]
```

规则：

- 只有下单后才开放聊天，不做团购详情或团长主页售前咨询。
- 会话按买家和店铺聚合，同一买家在同一店铺的多笔订单进入同一会话。
- “备货完成”只是聊天卡片，不改变订单状态。
- 消息页保留站内通知能力，同时新增聊天列表；底部消息角标显示聊天未读 + 通知未读。

## 10. 订单列表筛选流程

```mermaid
flowchart TD
    A[订单列表页] --> B[点击状态 tab]
    B --> C{是否已登录}
    C -- 否 --> D[触发全局登录拦截]
```

状态范围：

|---|---|
| 待支付 | `pendingPay` |

| 已发货 | `shipped` |
| 已完成 | `completed` |
| 已取消 | `canceled` |
