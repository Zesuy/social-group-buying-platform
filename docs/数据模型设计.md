# 数据模型设计

> 来源：基于 `docs/程序功能调研.md` 和 `docs/功能需求定义.md` 整理。
>
> 定位：本文先定义业务概念模型，再给出核心表结构草案。MVP 以核心交易和开团闭环为准，P1/P2 为后续扩展。
>
> 口径：业务范围和实现优先级以 `docs/功能需求定义.md` 为准；本文只描述数据如何承载这些需求，不扩大 MVP 范围。

## 1. 建模原则

本系统不能把“用户、团长、店铺、团购、商品”合并成一个对象。它们的职责不同：

```text
用户账号
  ↓ 激活经营能力
团长身份
  ↓ 创建经营载体
店铺
  ↓ 发布销售活动
团购
  ↓ 承载售卖对象
商品 / SKU
```

核心判断：

| 对象 | 本质 |
|---|---|
| 用户 | 登录和购买主体 |
| 团长 | 用户激活后的经营身份，也是买家看到的信任主体 |
| 店铺 | 团长的长期经营容器 |
| 团购 | 店铺发布的一次销售活动 |
| 商品 | 被售卖的具体货品，可被团购引用 |
| 订单 | 用户针对团购商品产生的交易记录 |

## 2. 业务概念模型

### 2.1 核心关系

```text
User 1 ── 0..1 Leader
Leader 1 ── 1 Store
Store 1 ── N GroupBuy
Store 1 ── N Product
GroupBuy 1 ── N GroupBuyItem
GroupBuyItem N ── 1 Product
User 1 ── N Order
Order N ── 1 GroupBuy
Order N ── 1 Store
Order N ── 1 Leader
Order 1 ── N OrderItem
OrderItem N ── 1 Product
```

### 2.2 扩展关系

```text
User N ── N Leader       通过 Subscription 订阅关系
User N ── N Store        通过 MemberCard / MemberRelation 会员关系
GroupBuy 1 ── N ContentBlock
User 1 ── N Address
Store 1 ── N Coupon      P1
Product / GroupBuy ── N DistributionRule   P2
Leader N ── N Leader     通过 DistributionRelation 帮卖关系 P2
```

## 3. MVP 业务约束

| 约束 | MVP 规则 |
|---|---|
| 用户与团长 | 一个用户最多拥有一个团长身份 |
| 团长与店铺 | 一个团长最多拥有一个店铺 |
| 店铺与团购 | 一个店铺可以发布多个团购 |
| 店铺与商品 | 一个店铺可以沉淀多个商品 |
| 团购与商品 | 一个团购通过团购商品关系包含一个或多个商品 |
| 商品与团购商品 | 商品是长期资产，团购商品是某次团购里的售卖配置 |
| 下单价格 | 订单价格以 `group_buy_items.group_price` 为准 |
| 商品快照 | 订单明细必须保存下单时的商品名称、规格、单价、数量等快照 |
| 支付 | MVP 使用模拟支付，只记录状态变化，不接真实支付渠道 |
| 售后退款 | MVP 不建完整售后退款流程，只在订单状态中预留 |
| 优惠券 | MVP 不参与订单金额计算 |
| 帮卖 | MVP 只预留店铺帮卖开关，不产生帮卖关系和佣金流水 |
| 积分 | MVP 不建立积分账户和积分流水 |
| 会员 | MVP 只记录会员关系展示字段，不实现升级和折扣计算 |
| 订阅 | MVP 只记录关注关系，不产生真实消息推送 |

建议数据库约束：

| 对象 | 建议约束 |
|---|---|
| leaders | `user_id` 唯一，保证一个用户一个团长身份 |
| stores | `leader_id` 唯一，保证一个团长一个店铺 |
| subscriptions | `user_id + leader_id` 唯一，避免重复订阅 |
| member_relations | `user_id + store_id` 唯一，避免重复会员关系 |
| orders | `order_no` 唯一 |

## 4. MVP 必建对象

| 对象 | 是否 MVP 必建 | 说明 |
|---|---:|---|
| 用户 | 是 | 登录、购买、开店的基础账号 |
| 团长身份 | 是 | 用户创建店铺后生成 |
| 店铺 | 是 | 团长经营容器 |
| 团长主页 | 是 | 可并入店铺或单独建表，建议 MVP 并入店铺展示字段 |
| 团购 | 是 | 普通团购活动 |
| 团购内容模块 | 简化 | MVP 可先用团购介绍字段，P1 再拆模块表 |
| 商品 | 是 | 店铺商品或团购商品 |
| 团购商品关系 | 是 | 支持一个团购包含多个商品 |
| 地址 | 是 | 下单履约需要 |
| 订单 | 是 | 交易主表 |
| 订单明细 | 是 | 记录购买商品 |
| 订阅关系 | 是 | 基础关注团长 |
| 会员关系 | 是 | 基础展示，成长规则后置 |

## 5. MVP 暂不建或仅预留对象

| 对象 | MVP 状态 | 后续优先级 | 说明 |
|---|---|---:|---|
| 商品库分类 | 不建 | P1 | MVP 商品直接挂店铺 |
| 优惠券 / 红包 | 不建 | P1 | 后续营销 |
| 购物车 | 不建 | P1 | MVP 直接购买 |
| 售后单 | 不建 | P1 | MVP 订单只预留售后状态 |
| 消息通知 | 不建 | P2 | 后续接公众号 / 服务通知 |
| 帮卖规则 | 预留字段 | P2 | MVP 店铺保留帮卖开关 |
| 佣金流水 | 不建 | P2 | 依赖帮卖成交 |
| 积分账户 | 不建 | P2 | 会员成长后续扩展 |
| 平台后台账号 | 不建 | P2 | 后续管理后台 |
| 审核 / 风控记录 | 不建 | P2 | 后台能力 |
| 数据统计快照 | 不建 | P2 | 经营看板 |

## 6. 核心表结构草案

字段类型以关系型数据库为参考，实际开发可按所选技术栈调整。

### 6.1 users 用户表

| 字段 | 类型 | 说明 |
|---|---|---|
| id | bigint | 用户 ID |
| nickname | varchar | 昵称 |
| avatar_url | varchar | 头像 |
| phone | varchar | 手机号 |
| wechat_openid | varchar | 微信 OpenID，后续接微信时使用 |
| status | varchar | normal / disabled |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

MVP 实现：基础用户信息、状态。  
MVP 不实现：复杂账号安全、黑名单、第三方授权完整流程。

### 6.2 leaders 团长身份表

| 字段 | 类型 | 说明 |
|---|---|---|
| id | bigint | 团长 ID |
| user_id | bigint | 关联用户 ID |
| display_name | varchar | 团长展示名 |
| avatar_url | varchar | 团长头像 |
| bio | text | 团长介绍 |
| service_status | varchar | normal / disabled |
| member_count | int | 成员数，可缓存 |
| follower_count | int | 关注数，可缓存 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

MVP 实现：用户创建店铺后生成团长身份，`user_id` 建议唯一。  
MVP 不实现：金牌团长、冠军团长、回复超快等认证标签体系。

### 6.3 stores 店铺表

| 字段 | 类型 | 说明 |
|---|---|---|
| id | bigint | 店铺 ID |
| leader_id | bigint | 团长 ID |
| name | varchar | 店铺名称 |
| logo_url | varchar | 店铺头像 / Logo |
| description | text | 店铺简介 |
| default_delivery_type | varchar | express / pickup / local_delivery |
| distribution_enabled | boolean | 是否开启帮卖分销，MVP 仅预留 |
| status | varchar | active / disabled |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

MVP 实现：店铺名称、头像、简介、物流、状态，`leader_id` 建议唯一。  
MVP 不实现：店铺认证、复杂物流模板、帮卖分销实际规则。

### 6.4 store_profiles 店铺 / 团长主页配置表

MVP 可将这些字段合并在 `stores` 中；如果希望主页配置独立，使用本表。

| 字段 | 类型 | 说明 |
|---|---|---|
| id | bigint | 主页配置 ID |
| store_id | bigint | 店铺 ID |
| leader_id | bigint | 团长 ID |
| background_url | varchar | 主页背景图 |
| location_text | varchar | 位置说明 |
| service_time | varchar | 客服服务时间 |
| help_sale_intro | text | 帮卖介绍，P2 使用 |
| album_ref | varchar | 相册素材号，P1/P2 使用 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

MVP 实现：可选。建议第一阶段先不单独建表，避免过早复杂化。  
MVP 不实现：素材号、帮卖介绍、复杂主页装修。

### 6.5 products 商品表

| 字段 | 类型 | 说明 |
|---|---|---|
| id | bigint | 商品 ID |
| store_id | bigint | 所属店铺 |
| name | varchar | 商品名称 |
| description | text | 商品描述 |
| cover_image_url | varchar | 商品主图 |
| category_id | bigint | 类目 ID，P1 可用 |
| base_price | decimal | 商品基础价格，不作为最终下单价格 |
| stock | int | 默认库存，-1 可表示不限库存 |
| status | varchar | active / inactive / deleted |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

MVP 实现：名称、描述、图片、基础价格、库存、状态。用户下单价格以团购商品价格为准。  
MVP 不实现：复杂 SKU、多图视频、商品审核、商品库分类。

### 6.6 product_skus 商品规格表

如果 MVP 只支持单规格，可以暂时不建本表，把规格名称放在 `group_buy_items` 中。若要支持多规格，使用本表。

| 字段 | 类型 | 说明 |
|---|---|---|
| id | bigint | SKU ID |
| product_id | bigint | 商品 ID |
| sku_name | varchar | 规格名称，如 3 瓶装 |
| price | decimal | SKU 价格 |
| stock | int | SKU 库存 |
| status | varchar | active / inactive |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

MVP 实现：可选。  
MVP 建议：若开发周期紧，先单规格；若页面已有规格选择，直接建本表。

| 字段 | 类型 | 说明 |
|---|---|---|
| id | bigint | 团购 ID |
| store_id | bigint | 店铺 ID |
| leader_id | bigint | 团长 ID |

| status | varchar | draft / published / ended / removed |

MVP 实现：普通团购、标题、介绍、封面、物流、发货时间、团购时间、公开状态。

### 6.8 group_buy_items 团购商品关系表

|---|---|---|

## 10. MVP 数据流

### 10.1 创建店铺

```text
users
  ↓ 创建
leaders
  ↓ 创建
stores
```

说明：

| 步骤 | 写入对象 |
|---|---|
| 用户点击创建店铺 | users 已存在 |
| 创建团长身份 | leaders |
| 创建店铺资料 | stores |
| 初始化主页 | MVP 可使用 stores 字段，P1 再拆 store_profiles |

### 10.2 发布团购

```text
stores
  ↓ 发布
group_buys
  ↓ 添加商品
products
  ↓ 绑定
group_buy_items
```

说明：

| 步骤 | 写入对象 |
|---|---|
| 填写团购信息 | group_buys |
| 新增商品 | products |
| 设置团购价和库存 | group_buy_items |
| 发布 | group_buys.status = published |

### 10.3 下单购买

```text
users
  ↓ 选择地址
addresses
  ↓ 下单
orders
  ↓ 商品快照
order_items
  ↓ 更新会员关系
member_relations
```

说明：

| 步骤 | 写入对象 |
|---|---|
| 选择或新增地址 | addresses |
| 提交订单 | orders |
| 写入商品快照 | order_items |

| 更新会员关系 | member_relations |

### 10.4 团长发货

```text
orders
  ↓ 发货

  ↓ 更新状态
orders.status = shipped
```

## 11. 设计结论

MVP 必须保留以下主线对象：

```text
User
Leader
Store
GroupBuy
Product
GroupBuyItem
Address
Order
OrderItem
Shipment
Subscription
MemberRelation
```

第一阶段不建议把优惠券、积分、帮卖、后台审核、消息通知一起纳入实现。它们应该在模型上预留扩展点，但不阻塞核心交易闭环。

最关键的边界是：

```text
店铺是长期经营资产；
团购是一次销售活动；
商品是可复用的售卖对象；
订单必须同时关联用户、团长、店铺、团购和商品快照。
```
