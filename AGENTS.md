# AGENTS.md

> 定位：本文是 AI 开发本项目时必须遵守的总规则。`docs/dev/batches/` 主要作为历史批次归档和口径参考；当前新功能开发以本文件、核心产品 / API 文档、实际代码契约和用户明确需求为准。

## 1. 项目目标

本项目实现一个以微信私域传播为主要场景的社群团购产品。核心闭环是：用户浏览团购、下单、模拟支付、团长发货；普通用户创建店铺后成为团长，并发布团购活动。

MVP 核心交易闭环已经冻结；后端 P1 到 `batch-08-notification-upload-asset-governance.md` 的能力已完成，并在此后继续按用户提出的新功能整体演进。当前项目不再严格按 batch 或 P1/P2 顺序推进，开发时必须先核对实际代码、API 文档和联调文档，不能仅凭旧 batch 文件判断能力是否可用。

当前主要工作重点是：保持已实现能力与文档一致，继续按明确页面、链路或功能需求做小步闭环开发。涉及进度、阶段、能力状态的文档表达必须及时校准，避免把已完成接口误判为占位或越界能力。

## 3. 禁止扩展项

不得实现以下仍禁止或未落地的能力：

| 能力 | 当前处理方式 |
|---|---|
| 真实微信支付 | 仅使用模拟支付 |
| 帮卖分销 | 不实现帮卖关系、佣金和结算 |
| 积分商城 | 不建立积分账户、积分流水和兑换 |
| 公众号 / 微信服务通知 | 不实现真实触达；站内通知只用 REST 轮询 |
| 平台后台 | 不建设后台页面和后台 API |
| 完整客服中心 | 已有订单上下文轻量聊天；不实现客服分配、撤回、评价、敏感词审核、实时推送 |
| 真实地图 SDK / 逆地理编码 | 不接入外部地图服务，不做实时 GPS 轨迹 |
| 复杂海报后端生成 | 后端不强制生成 bitmap 海报 |
| 任意 HTML 富文本 | 不允许任意 HTML、iframe、脚本、样式入库 |

已经完成的能力不得回退成占位，也不得跳过真实接口改用假数据通过验收。未实现能力只能隐藏、灰态或 toast，不得抢占当前页面主流程。

## 4. 技术栈

后端：Java 17 + Spring Boot + Maven + Spring Web + Validation + MyBatis-Plus + MySQL + Flyway + JUnit 5 + MockMvc。

前端：H5 移动端网页，Vue 3 + Vite + TypeScript + Vue Router + Pinia + Vant + Axios + Vitest + Playwright。

### 4.1 前端产品与设计约束

前端页面必须同时遵守 `docs/前端产品与页面设计准则.md` 和 `DESIGN.md`。

前端实现不得按传统低价商城或货架电商理解本项目。页面应优先表达“团购活动、团长信任、店铺经营、订单履约、订阅复购”，而不是搜索比价、购物车、平台补贴或 SKU 丰富度。

所有 UI 改动必须满足以下要求：

| 规则 | 要求 |
|---|---|
| 业务定位 | 页面必须表达私域社群团购活动，不得退化成普通商品商城 |
| 视觉系统 | 颜色、字号、间距、圆角、阴影、按钮、卡片和状态标签遵守 `DESIGN.md` |
| 组件必要性 | 不保留无业务意义的占位组件 |
| 主操作 | 每个页面只突出一个最重要的主操作 |
| 团长信任 | 首页、团购详情、团长主页等关键页面必须明显展示团长 / 店铺信息 |
| 阶段边界 | 已实现能力必须接真实接口；未实现能力只能隐藏、灰态或 toast，不得抢占主流程 |
| 移动端验收 | 固定底栏、Tabbar、购买栏必须处理 safe-area，不得遮挡内容 |

页面级、链路级、核心组件级前端任务必须先遵守 `docs/dev/frontend-ai-workflow.md`：完成需求澄清和前端技术方案后，再进入代码实现。若需求、接口、权限、状态或阶段边界不清，必须先查阅实际代码、API 文档、联调文档和必要的历史批次文档，再决定实现口径。

团购详情和发布团购相关页面必须参考 `docs/团购活动内容块设计.md`。当前采用结构化内容块，不实现完整富文本编辑器；页面结构不得把团购内容表达做死，也不得退回为单一商品详情描述。

消息页已经从 MVP 占位升级为真实消息中心。涉及通知、聊天、Tabbar 未读角标、通知详情、已读 / 全部已读、轮询时，必须使用 `frontend/src/api/notifications.ts`、`frontend/src/api/chats.ts`、`frontend/src/composables/useNotificationPolling.ts`、`frontend/src/composables/useChatPolling.ts` 和后端接口口径，不得恢复本地静态消息数据。

## 5. 开发策略

| 规则 | 要求 |
|---|---|
| 当前阶段 | 后端 P1 已完成，项目已进入按明确需求整体演进和文档校准阶段 |
| 当前范围 | 每次只执行用户明确指定的页面、链路、功能或修复范围 |
| 执行顺序 | 不得把已完成能力当作未实现；新增能力必须先核对 API、数据模型、页面和测试边界 |
| 代码组织 | 后端代码按模块组织，如 auth、user、store、product、groupbuy、address、order、subscription、member、cart、coupon、aftersale、notification、chat、upload |
| 批次组织 | 历史 batch 只作参考；当前开发按功能链路或页面闭环推进 |
| 联调节奏 | 每完成一个功能链路，必须同步更新 `docs/前后端联调文档.md` |
| 文档约束 | 不得绕过既有 API 文档、数据模型、页面与交互文档 |
| 问题处理 | 不得把已知问题留到下一批 |

前端改造时，必须优先复用已有 API 封装、类型、组合式函数和通用组件；若页面仍使用旧 MVP 占位、mock 数据、死链接、无业务意义入口或不符合当前真实状态流转，应在当前页面任务内一并修正。

## 6. 测试硬性要求

| 范围 | 要求 |
|---|---|
| 所有接口 | 每个接口至少有 MockMvc 测试 |
| 关键 Service | 必须有单元测试 |
| 订单 / 支付 / 库存 / 发货 | 必须有事务和异常路径测试 |
| 权限 | 覆盖未登录、非资源所有者、非团长身份、跨店铺访问 |
| 状态 | 覆盖订单、支付、发货、团购等状态流转 |
| 库存 | 覆盖库存不足和支付扣减 |
| 快照 | 覆盖订单商品快照和地址快照 |
| 重复动作 | 覆盖重复支付、重复发货 |
| 前端功能闭环 | 完成页面 / 链路级改动时必须运行 `npm run typecheck`、`npm run lint`、`npm run test:unit`、`npm run build` 和相关 `npm run test:e2e`；完整前端验收仍需运行全量 E2E |
| 前端 E2E | 页面 / 链路级改动必须补充或维护匹配的 Playwright E2E；首次运行前执行 `npx playwright install`；不得为了通过 E2E 绕过真实接口、登录拦截、状态流转或灰态约束 |

测试不得留到下一批。

如果只修改单页或局部组件，也必须至少运行与该页面相关的单测 / E2E；完成一个页面链路或功能闭环时再运行完整前端验收命令。

## 7. 文档维护要求

每个页面、链路或功能闭环完成后必须同步维护 `docs/前后端联调文档.md`，至少包含页面入口、前置条件、调用顺序、请求样例、响应样例、成功判定、常见错误码和测试数据准备方式。

如果实现中需要修改上游文档，必须明确说明原因，并优先保持 `docs/功能需求定义.md`、`docs/数据模型设计.md`、`docs/API风格规范.md`、`docs/API设计.md`、`docs/页面与交互文档.md` 的口径一致。

## 8. 每次功能闭环完成后的输出

每次页面、链路或功能闭环完成后必须输出：

| 项目 | 要求 |
|---|---|
| 已实现接口清单 | 列出本次完成或接入的接口 |
| 已新增 / 修改文件清单 | 列出代码和配置文件 |
| 已新增 / 修改 Flyway 迁移清单 | 如无新增迁移，写“无” |
| 已新增 / 修改测试清单 | 列出 MockMvc 和 Service 测试 |
| 已更新联调文档的位置 | 指明章节或标题 |
| 测试运行命令和结果 | 写明命令和通过 / 失败；前端链路级改动必须包含相关 `npm run test:e2e` 结果 |
| 本次未解决问题 | 如无则写“无” |
